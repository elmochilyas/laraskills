# LaraSkills 1.0.0-beta.22

**Release date:** 2026-06-22
**Type:** Feature release — Senior Laravel Architecture Judgment Pack

## Summary

LaraSkills now goes beyond "use the right Laravel thing" and improves toward "use the right Laravel thing, explain when it fits, when it fails, what the escape hatch is, and how to operate it safely in production."

This release adds 31 new knowledge units covering calibrated package recommendations, SaaS billing architecture, transaction boundaries, queue deployment safety, billing observability, team-scoped authorization, and calibrated architectural language — all driven by a real benchmark comparing raw OpenCode against OpenCode with LaraSkills on a Laravel SaaS architecture decision task.

## Why This Release Matters

The benchmark exposed gaps in LaraSkills' architecture guidance:

1. Package recommendations were too absolute ("Use Cashier" without conditions)
2. SaaS billing stopped at Plan → Features without entitlement modeling
3. Stripe/Cashier guidance lacked escape hatches
4. Transaction boundaries and afterCommit guidance were weak
5. Laravel events were sometimes described too close to event sourcing
6. Team-scoped Spatie Permission guidance needed more depth
7. Queue architecture needed more advanced Laravel primitives
8. Production observability and deployment guidance were too light
9. Architecture language used too many absolute words like "always/every/never"

Beta.22 closes these gaps while preserving LaraSkills' strengths: Laravel-native, ecosystem-aligned, maintainable, package-aware, and team-friendly.

## What Changed

### New Senior Architecture Concepts

- **Calibrated package recommendations** — Every major package recommendation now includes 8 dimensions: default recommendation, when it fits, when it does not fit, alternative, escape hatch, tradeoffs accepted, testing impact, and operational impact.

- **Package decision matrices** — Decision matrices for Laravel Cashier, Spatie Permission, Laravel Pennant, Horizon, and Telescope/Pulse with fit conditions, non-fit conditions, and escape hatches.

- **SaaS billing architecture** — Plan / Subscription / Feature / Entitlement / UsageLimit / UsageRecord separation. Stripe as billing source of truth, local DB as application cache / entitlement projection. StripeEvent idempotency table, webhook audit log, webhook replay flow, reconciliation job, subscription drift detection.

- **Transaction boundaries and afterCommit** — `dispatchAfterCommit()`, `event(...)->afterCommit()`, `DB::afterCommit()`. Compensating actions for external API failures. Race condition mitigation for after-commit dispatched jobs.

- **Laravel events vs event sourcing** — Explicit rule: Laravel events are not event sourcing. Use "domain events" for side-effect decoupling, "audit log" for persisted diagnostic records, and "event sourcing" only when the full pattern (event store, aggregates, projections, snapshots, replay) is implemented.

- **Advanced queue architecture** — Dedicated webhook queue with idempotency (ShouldBeUnique, WithoutOverlapping). Billing queue topology (webhooks, billing, notifications, default). Queue deployment safety (backward-compatible deploys, class rename aliases, feature flags, queue:restart, Horizon verification).

- **Billing observability** — Production metrics (webhook received/processed/failed/duplicate counts, processing latency, subscription drift, queue depth, feature gate denials). Alert runbooks (6 alert types). Support repair flows (webhook replay, subscription sync, entitlement recalculation with --dry-run, audit logging, Stripe rate-limit awareness).

- **Team-scoped authorization** — Separation of authentication, team membership, team role authorization, subscription entitlements, and platform admin privileges. Spatie Permission team support depth (guard consistency, permission cache invalidation, global vs team-scoped roles). SaaS authorization test matrix.

- **Calibrated architecture language** — Architecture guidance uses calibrated language ("default to", "prefer", "usually", "when appropriate") rather than absolute language ("always", "never", "every") except for security-critical and data-integrity rules.

### Benchmark Motivation

A benchmark was run comparing raw OpenCode against OpenCode with LaraSkills on a Laravel SaaS architecture decision task (teams, Stripe billing, webhooks, role permissions, subscription feature gates, queues, long-term maintainability).

LaraSkills produced a more Laravel-native, maintainable, ecosystem-aligned proposal. But the benchmark exposed the gaps listed above. Beta.22 closes those gaps so future architecture proposals become more senior, calibrated, production-aware, and Laravel-native.

### Benchmark Regression

New benchmark fixture `bench-073` verifies that retrieval for a SaaS architecture query surfaces senior concepts (Cashier decision matrix, plan/feature/entitlement, webhook idempotency, subscription drift, billing failure states) rather than only generic CRUD guidance.

## Upgrade Instructions

### Global users

```powershell
npm update -g laraskills
laraskills -v
cd my-laravel-project
laraskills update --assistants all --yes
laraskills doctor
```

### Local/project users

```powershell
npm install --save-dev laraskills@latest
npx laraskills -v
npx laraskills update --assistants all --yes
npx laraskills doctor
```

### New users

```powershell
npm install -g laraskills
cd my-laravel-project
laraskills init --assistants all --integration full --profile core --yes
```
