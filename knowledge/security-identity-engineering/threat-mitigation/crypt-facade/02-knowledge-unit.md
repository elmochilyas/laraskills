# Executive Summary

Laravel's `Crypt` facade provides AES-256 encryption/decryption using the application's `APP_KEY`. The default cipher is `AES-256-CBC` (Laravel ≤10) or `AES-256-GCM` (Laravel 11+). GCM provides authenticated encryption (AEAD) — it verifies both confidentiality and integrity, preventing ciphertext tampering. CBC requires a separate HMAC for integrity (encrypt-then-MAC). The encryption key (`APP_KEY`) is derived from the base64-encoded key in `.env`. Rotating `APP_KEY` invalidates all encrypted data — plan accordingly.

---

# Core Concepts

- **APP_KEY**: Base64-encoded random string in `.env`. Generated by `php artisan key:generate`. Must be 32 bytes (AES-256) — `base64:...` prefix encodes a 32-byte random value.
- **AES-256-CBC**: Cipher Block Chaining mode. Encrypts each block XOR'd with the previous ciphertext. Requires a random Initialization Vector (IV) per operation. Does NOT provide integrity — ciphertext can be modified in transit without detection if used alone.
- **AES-256-GCM**: Galois/Counter mode. Authenticated encryption — provides confidentiality + integrity + authenticity. Includes an authentication tag. Preferred cipher. Introduced as default in Laravel 11.
- **Encryption vs Hashing**: Encryption is reversible (with key). Hashing is one-way. `Crypt` encrypts data that must be readable later. `Hash` stores passwords — never use `Crypt` for passwords.
- **Serialization**: `Crypt::encrypt()` serializes the value (JSON) before encryption. `Crypt::decrypt()` deserializes after decryption. Works with arrays, objects, and strings.

---

# Mental Models

- **Lockbox and Tamper Seal**: CBC is a lockbox — it keeps contents secret but someone could tamper with the box. GCM is a lockbox with a tamper-evident seal — if anyone modifies the ciphertext, decryption fails.
- **APP_KEY as Master Key**: The `APP_KEY` is the master key for all encrypted data. If it leaks, all encrypted data is compromised. If it changes, all encrypted data is lost.

---

# Internal Mechanics

- `Illuminate\Encryption\Encrypter` handles encryption/decryption. Constructed with the decoded key and cipher.
- `encrypt($value, $serialize = true)`: serializes → generates random IV (16 bytes for CBC, 12 bytes for GCM) → encrypts with OpenSSL → for CBC: HMAC-SHA256 the ciphertext → base64-encodes (HMAC + IV + ciphertext). For GCM: the authentication tag is appended by OpenSSL.
- `decrypt($payload, $unserialize = true)`: base64-decodes → for CBC: verify HMAC → decrypt. For GCM: OpenSSL verifies the authentication tag automatically.
- Key derivation: `base64_decode(substr($key, 7))` (strips `base64:` prefix). If the key is not base64-encoded, it's used as-is.
- The cipher is configured in `config/app.php` `'cipher' => 'AES-256-GCM'`. Cannot be changed per environment without migrating all encrypted data.

# Patterns

## Encrypted Model Attributes Pattern
- **Purpose**: Store sensitive data (SSN, API keys) encrypted in the database.
- **Implementation**: Cast the attribute as `'encrypted'` in the Model: `protected $casts = ['api_key' => 'encrypted']`. Laravel handles encryption/decryption transparently.
- **Benefits**: Transparent encryption/decryption — application code never deals with raw `Crypt` calls.
- **Tradeoffs**: Cannot query on encrypted fields (no WHERE on encrypted column). Index encrypted columns only for equality, not range.

## Encryption for Queue Payloads
- **Purpose**: Protect sensitive job data in the queue.
- **Implementation**: Laravel's queue system encrypts payloads by default when `config('queue.default')` with encryption settings. Or manually: `Crypt::encrypt($jobData)` before dispatching, decrypt in job.
- **Benefits**: Queue data at rest (Redis, database) is protected.

## Cross-Service Encryption
- **Purpose**: Two services share the same APP_KEY (or a derived key) to exchange encrypted data.
- **Implementation**: Both services use the same cipher (`AES-256-GCM`). Service A encrypts, Service B decrypts with the same key.
- **Benefits**: Secure inter-service communication without TLS overhead for async messages.
- **Tradeoffs**: Key distribution — both services must have the same APP_KEY. Key rotation requires coordinated updates.

# Architectural Decisions

| Decision | Context | Recommendation |
|---|---|---|
| AES-256-CBC vs AES-256-GCM | Cipher selection | GCM (default in L11+) — authenticated encryption prevents tampering. Only use CBC for legacy compatibility |
| APP_KEY per environment | Environment isolation | Unique APP_KEY per environment (dev, staging, prod). Dev and prod keys must differ — encrypted data is not portable |
| Encrypted casts vs manual Crypt calls | Frequent encryption vs one-off | Encrypted casts for model attributes; manual Crypt only for non-model data or API calls |

---

# Tradeoffs

| Benefit | Cost | Consequence |
|---|---|---|
| GCM authenticated encryption prevents tampering | GCM ciphertext is 12-28 bytes larger than CBC (nonce + tag) | Negligible storage increase |
| Encrypted casts are transparent | Cannot query encrypted columns | WHERE clauses on encrypted fields return no results — must decrypt in application |
| APP_KEY-based encryption is simple | Rotating APP_KEY invalidates ALL encrypted data | Key rotation requires decrypting with old key, re-encrypting with new key |
| Serialization handles complex data | Deserialization failures on format changes | Changing encrypted attribute type or structure breaks existing encrypted records |

# Performance Considerations

- AES-256-GCM encryption/decryption: ~0.5-2ms for typical payload sizes (up to 10KB). OpenSSL is hardware-accelerated on modern CPUs.
- Larger payloads (images, documents) should use dedicated file encryption, not `Crypt`.
- Encrypted casts add overhead on every model load/save. Consider lazy decryption for frequently accessed but rarely needed encrypted fields.

---

# Production Considerations

- **APP_KEY Length**: Must be 32 bytes for AES-256. `php artisan key:generate` creates a 32-byte key encoded as base64. Manually written keys must also be 32 bytes.
- **Key Rotation**: There is no built-in key rotation. Custom script: load all records, decrypt with old key, re-encrypt with new key, save. For large tables, use chunked processing.
- **Cipher Migration**: If switching from CBC to GCM, all existing encrypted payloads (CBC-encrypted) must be re-encrypted. Old payloads cannot be decrypted with GCM.
- **Storage of Encrypted Data**: Encrypted data is binary-safe (base64-encoded). Store in TEXT or VARCHAR column with sufficient length. Encrypted payloads are ~40% larger than plaintext due to base64 encoding + IV/tag overhead.

---

# Common Mistakes

- **Encrypting passwords**: Passwords must be hashed (one-way), not encrypted. Never use `Crypt::encrypt()` for passwords — user `Hash::make()`.
- **Not using encrypted cast for model attributes**: Manually calling `Crypt::encrypt()`/`decrypt()` on every attribute access is error-prone and easy to forget.
- **Assuming encryption = security everywhere**: Encryption protects data at rest. It does not protect against SQL injection (attacker reads decrypted data from the app) or server-side file inclusion.
- **Changing APP_KEY without decrypt/re-encrypt**: Existing encrypted data becomes permanently undecryptable. No recovery without the old key.
- **Sharing APP_KEY between environments**: Dev encrypted data accidentally pushed to production cannot be read by production's APP_KEY. Use unique keys per environment.

---

# Failure Modes

- **APP_KEY Rotation**: All encrypted data in the database (2FA secrets, encrypted model attributes, session data) becomes undecryptable. Users cannot log in (2FA), jobs fail, API calls break. Mitigation: rotate keys with a script that decrypts/re-encrypts.
- **Ciphertext Tampering (CBC only)**: An attacker modifies the encrypted cookie or hidden form field. CBC decrypts to garbage data without raising an exception — the application receives corrupted data. Mitigation: use GCM (authenticated encryption) or implement custom HMAC verification.
- **Payload Too Large**: `Crypt::encrypt()` with payload > 100MB exhausts memory. Use file encryption for large data.
- **Mcrypt Deprecation**: Older Laravel versions used `Mcrypt` (deprecated in PHP 7.1, removed in PHP 7.2). OpenSSL is the only supported engine since Laravel 5.x.

## Research Notes
- Laravel rate limiting was significantly enhanced in Laravel 12 with the introduction of named rate limiters that can reference other limiters for inheritance — RateLimiter::for('api', fn() => RateLimiter::for('global')->by('ip')).
- The 	hrottle middleware uses dynamic rate limit resolution when a Closure is passed — the limit is re-evaluated on every request, allowing per-user rate limit overrides based on subscription tier or trust level.
- Signed URLs in Laravel use HMAC-SHA256 with the application key — the signature includes all query parameters and the expires timestamp, providing tamper-proof URL validation without server-side state.
- File upload validation in Laravel 12+ includes built-in SVG upload protection (svg validation rule) that checks for embedded scripts and event handlers in SVG files.
- The Crypt facade uses serialization for encrypting objects and arrays — this introduces a potential unserialization vulnerability if an attacker can control the encrypted data; use Crypt::encryptString() for simple values.
- Form Request validation executes in the middleware pipeline before the controller — the prepareForValidation() hook allows preprocessing input before validation, useful for normalizing data format.
- Plan-aware throttling patterns use RateLimiter::for() with dynamic limit resolution based on the authenticated user's plan — the 	hrottle middleware accepts a RateLimiter::limiter() callback for complex limit definitions.
- Community rate limiting packages (spatie/laravel-rate-limited-job-middleware) extend rate limiting to queued jobs, not just HTTP requests — this prevents downstream API rate limit violations during batch job processing.

## Internal Mechanics
- **RateLimiter Resolution**: RateLimiter::for('login', fn(, ) => Limit::perMinute(5)) registers a named limiter. The 	hrottle middleware resolves the limiter by name at runtime, applies the limit, and returns a 429 Too Many Requests response with Retry-After header when exceeded.
- **Signed URL Generation**: URL::signedRoute('verify', ['id' => ->id], expires: 3600) → collects route name, parameters, and expiration → builds URL → computes HMAC-SHA256 signature over the URL string using APP_KEY → appends ?signature=<hash> to the URL. The ValidateSignature middleware re-computes the hash and compares using hash_equals().
- **Crypt Facade Encryption Flow**: Crypt::encrypt('value') → generates random IV (16 bytes for AES-256-CBC) → serializes the value → encrypts with AES-256-CBC using APP_KEY as encryption key → computes HMAC-SHA256 for integrity → JSON-encodes the payload ({iv, value, mac, tag}). Decryption reverses the process and verifies the MAC.
- **Form Request Validation Flow**: Custom form request class extends Illuminate\Foundation\Http\FormRequest → middleware pipeline calls FormRequest->authorize() → if false, returns 403 Forbidden → if true, calls FormRequest->rules() → FormRequest->validator() validates the request data against rules → if validation fails, throws ValidationException with error bag → if passes, the validated data is available via $request->validated().
- **File Upload Processing**: Uploaded file arrives as Symfony\Component\HttpFoundation\File\UploadedFile → $request->file('document') returns UploadedFile instance → $file->store('uploads') moves file to configured filesystem disk → MIME type is detected by Symfony's MimeTypeGuesser (not by client-provided content-type).
- **Dependency Audit Flow**: composer audit reads composer.lock → matches each package/version against the Security Advisories Database → returns list of known vulnerabilities with CVE IDs, severity, and advisory URLs. The command fails with exit code 1 when vulnerabilities match.
