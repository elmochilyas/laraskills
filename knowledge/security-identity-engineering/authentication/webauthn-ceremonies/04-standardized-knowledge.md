## Overview

WebAuthn (Web Authentication) is a W3C standard for passwordless authentication using public key cryptography. Two ceremonies define the protocol: **Attestation** (registration — create a credential) and **Assertion** (authentication — prove possession of the credential). During attestation, the user's device generates a new public/private key pair, sends the public key to the server, and retains the private key. During assertion, the server sends a challenge, the device signs it with the private key, and the server verifies with the stored public key. Both ceremonies rely on the browser's `navigator.credentials` API.

---

## Core Concepts

- **Attestation**: Registration ceremony. Authenticator creates key pair → returns public key, credential ID, and attestation statement to server.
- **Assertion**: Authentication ceremony. Server sends challenge → authenticator signs with private key → server verifies with stored public key.
- **Challenge**: Random byte string generated by the server. Must be unique per ceremony and time-limited.
- **Credential ID**: Unique identifier for the public key. Stored server-side alongside the public key.
- **Relying Party (RP)**: The server (your Laravel app) that registers and verifies credentials. Identified by domain.
- **User Verification**: `required` (biometric/PIN), `preferred`, or `discouraged`. Controls whether the user must actively verify.
- **Attestation Statement**: Metadata about the authenticator (optional). For enterprise, may verify the authenticator model.

## When To Use

- Implementing passkey authentication (backends for `laravel/passkeys`, `spatie/laravel-passkeys`, `laragear/webauthn`)
- Understanding the underlying protocol for debugging passkey issues
- Custom WebAuthn implementations not covered by existing packages
- Enterprise WebAuthn requiring attestation verification (trusted platform modules)

## When NOT To Use

- Most applications should use `laravel/passkeys` or `spatie/laravel-passkeys` — implementing ceremonies manually is error-prone
- Simple passwordless auth (use a package — the protocol is complex)
- When browser compatibility is a concern (WebAuthn requires modern browsers + HTTPS)

---

## Best Practices

- **Use Existing Packages**: `laravel/passkeys`, `spatie/laravel-passkeys`, `laragear/webauthn` handle ceremony logic. Only implement manually if absolutely necessary.
- **Challenge Uniqueness**: Each ceremony must have a unique, cryptographically random challenge. Store in session/temporary store. Expire after 5 minutes.
- **Store Credential ID + Public Key**: Both are needed for assertion. Index by credential ID for efficient lookup.
- **Counter Tracking**: Each authenticator maintains a signature counter. Track it server-side to detect cloned authenticators.
- **RP ID Validation**: Always validate the RP ID against the request origin. The RP ID must match the effective domain.

---

## Architecture Guidelines

- Attestation flow: Server generates challenge → browser calls `navigator.credentials.create()` → returns `PublicKeyCredential` → server validates and stores credential
- Assertion flow: Server generates challenge → browser calls `navigator.credentials.get()` → returns `AuthenticationResponse` → server verifies with stored public key
- Credential storage: `credential_id`, `public_key`, `counter`, `user_id`, `transports`
- Challenge storage: challenge → user association, expiry (5 minutes), single-use

---

## Performance Considerations

- Key generation: <1 second on modern devices (TPM/secure enclave)
- Signature verification: ~1-5ms server-side using stored public key
- Challenge storage: minimal overhead (session or temporary cache entry)

---

## Security Considerations

- **Challenge Must Be Random**: Use `random_bytes(32)` — never reuse or predict challenges.
- **Challenge Expiry**: Challenges should expire (5 minutes recommended). Prevents replay of captured challenges.
- **Counter Monitoring** : If `signCount` decreases, the authenticator may be cloned — force re-registration.
- **User Verification**: `required` is recommended for production. `discouraged` reduces security to device-presence only.
- **Attestation Verification** (enterprise): Verify the `aaguid` against a list of trusted authenticator models.

---

## Common Mistakes

| Mistake | Cause | Consequence | Better Approach |
|---------|-------|-------------|-----------------|
| Implementing ceremonies manually | Unaware of existing packages | Protocol implementation bugs; security vulnerabilities | Use `laravel/passkeys` or `spatie/laravel-passkeys` |
| Reusing challenges | Performance optimization | Replay attacks possible | Generate unique random challenge per ceremony |
| Not setting challenge expiry | Storing indefinitely | Old challenges can be used for replay | Expire challenges after 5 minutes |
| Ignoring counter value | Not tracking signature count | Cloned authenticator goes undetected | Verify counter increases on each assertion |
| Wrong RP ID configuration | Using full URL instead of domain | Ceremonies fail | RP ID must be the effective domain (no port, no path) |

---

## Anti-Patterns

- **Implementing custom WebAuthn JavaScript**: Use `@laravel/passkeys` npm package or `@simplewebauthn/browser`
- **Storing private keys on server**: Private key never leaves the device — not even transmitted
- **Allowing ceremonies without HTTPS**: WebAuthn requires secure context

## Examples

**Attestation (server-side challenge):**
```php
// Generate registration challenge
$challenge = random_bytes(32);
session()->put('webauthn_registration_challenge', [
    'challenge' => base64_encode($challenge),
    'user_id' => $user->id,
    'expires_at' => now()->addMinutes(5),
]);
```

**Assertion verification:**
```php
// Verify authentication assertion
$publicKey = $credential->public_key; // stored during attestation
$challenge = base64_decode(session('webauthn_authentication_challenge.challenge'));

// Use a WebAuthn library to verify the assertion
$valid = $webauthnLib->verifyAssertion(
    $publicKey,
    $challenge,
    $clientDataJSON,
    $authenticatorData,
    $signature
);

if ($valid) {
    $credential->increment('counter');
}
```
