# ku-03: Deferred Providers

## Metadata
- **Domain:** Laravel Execution Lifecycle & Framework Internals
- **Subdomain:** Boot Order & Timing
- **KU:** ku-03-deferred-providers
- **Version:** 1.0
- **Last Updated:** 2026-06-02

## Overview
Deferred service providers postpone both `register()` and `boot()` until the first time a service they provide is actually resolved from the container. They are identified during provider registration, stored in a manifest, and loaded lazily when `$app->make(Service::class)` is called. This optimization reduces bootstrap overhead by skipping providers whose services are not used on a given request.

## Core Concepts
- **Deferral mechanism**: A provider is deferred by implementing `DeferrableProvider` (or setting `$defer = true`) and implementing `provides()` to list the service abstractions it registers.
- **Manifest**: `bootstrap/cache/services.php` maps service abstracts to provider classes. Generated by `php artisan optimize` or `ProviderRepository::createManifest()`.
- **Lazy loading trigger**: When `$app->make(UnboundService::class)` is called and no binding exists, the container checks `$this->deferredServices` and loads the matching provider.
- **Auto-boot**: A deferred provider loaded post-boot phase immediately runs `register()` then `boot()` because `$this->isBooted()` is true.
- **`when()` method**: Allows event-triggered deferred loading — if a deferred provider needs to respond to an event, `when()` lists the events that trigger preemptive loading.
- **Deferred vs eager**: Eager providers run on every request. Deferred providers run only when their services are first resolved.

## When To Use
- Providers that only bind services (no boot logic) — the primary use case for deferral.
- Providers for optional features (admin panels, analytics, reporting) that aren't used on every request.
- Providers that register heavy services (mail, notifications, PDF generation) when these services are infrequently used.
- Package providers that offer optional functionality gated behind feature checks.

## When NOT To Use
- Providers that register routes, event listeners, view composers, or gates in `boot()` — these would not be registered until the provider is triggered.
- Providers that must run on every request (e.g., logging setup, error handling, middleware registration).
- Providers with `when()` dependencies that fire on every event dispatch — the check overhead may exceed the deferral benefit.
- When the deferred service is type-hinted in every controller/service constructor — this defeats deferral by triggering resolution immediately.

## Best Practices (WHY)
- **Use `DeferrableProvider` interface** (not `$defer` property): The interface enables IDE autocompletion and static analysis.
- **Keep `provides()` complete**: Every binding and alias registered in `register()` must be in `provides()`. Missing entries cause `BindingResolutionException`.
- **Use `when()` sparingly**: Each entry adds overhead to every event dispatch. Only use when the provider must load before a listener executes.
- **Audit after Octane deployment**: Deferred providers are less impactful under Octane (boot cost amortized), but still reduce per-worker memory.

## Architecture Guidelines
- Place deferred providers in `config/app.php` `providers` array — they are recognized during registration and deferred automatically.
- The services manifest should be regenerated after any `provides()` change — stale manifests cause resolution failures.
- For providers with both bindings and boot logic, consider splitting into one deferred binding provider and one eager boot provider.
- Use `when()` for deferred providers that need to register event listeners before specific events fire.

## Performance
- Without services cache: must reflect on every provider to detect deferral (5-20ms per request).
- With services cache: deferred providers are skipped entirely until their service is resolved.
- First `make()` penalty: loading a deferred provider adds `register()` + `boot()` time synchronously (10-50ms for heavy providers).
- Memory savings: deferred provider class files are still autoloaded if type-hinted, but the provider instance (~2-5KB) is not created until resolution.

## Security
- A deferred provider that registers auth-related services loads at first auth call — ensure the provider's `provides()` includes all auth-related bindings.
- Deferred providers loaded after request processing starts have access to the current request context — ensure closures don't capture sensitive data.

## Common Mistakes

| Mistake | Description | Cause | Consequence | Better Approach |
|---------|-------------|-------|-------------|-----------------|
| Forgetting `provides()` | Provider is deferred but `provides()` is empty | Developer adds `$defer = true` but doesn't list services | `BindingResolutionException` on resolution — provider never loads | List every binding in `provides()` |
| Early type-hint defeat | Controller constructor type-hints deferred service | Developer injects the deferred service in a widely-used class | Provider loads on every request — no optimization gained | Use lazy resolution or `ShouldBeDeferred` proxy |
| Missing aliases in provides() | Registers alias in register() but not in provides() | Alias resolution bypasses deferred detection | Facade/alias calls don't trigger provider | Include all aliases in `provides()` |
| Boot() logic that must run early | Deferred provider registers event listeners in boot() | Provider loads too late to catch early events | Listener never fires for first event | Use `when()` to trigger provider on event |
| Stale manifest after changes | Modified `provides()` but didn't clear cache | Not running `optimize:clear` | Old manifest prevents new services from resolving | Clear and regenerate services cache |

## Anti-Patterns
- **Fat deferred provider**: A deferred provider with heavy `register()` and `boot()` logic defeats the purpose — the first use pays the full cost.
- **Deferred facade accessor**: Registering a facade alias in `provides()` but not including it — facade calls return `null` silently.
- **Over-deferral**: Making every provider deferred creates unpredictable latency spikes on first service usage.

## Examples
```php
class AnalyticsServiceProvider extends ServiceProvider implements DeferrableProvider
{
    public function register()
    {
        $this->app->singleton(Analytics::class, function ($app) {
            return new AnalyticsService($app['config']['analytics.key']);
        });
        $this->app->alias(Analytics::class, 'analytics');
    }

    public function provides()
    {
        return [Analytics::class, 'analytics'];
    }
}
```
