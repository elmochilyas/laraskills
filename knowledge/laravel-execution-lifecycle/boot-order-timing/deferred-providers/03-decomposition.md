# Decomposition: Deferred Providers

## Boundary Analysis
**Scope:** The deferred service provider mechanism — the `DeferrableProvider` interface, the `provides()` method, the deferred manifest, lazy registration on first resolution, and the tradeoffs between eager and deferred provider loading.

**Excluded:**
- The main bootstrap pipeline (covered in Complete Boot Sequence)
- Provider registration order mechanics (covered in Provider Registration Order)
- Services cache generation and invalidation (covered in Services Cache)
- Octane-specific deferred loading behavior (covered in Octane Boot Timing)
- Eager provider detailed behavior (covered in Eager Providers)

## Atomicity Assessment
**Status:** ✅ Atomic (no split needed)

**Rationale:** Deferred providers are a single, self-contained optimization mechanism. The `DeferrableProvider` interface, `provides()` method, manifest-based resolution, and lazy boot behavior are all components of one feature. They cannot be understood in isolation — the provides() method without the manifest, or the manifest without the lazy resolution, are meaningless.

## Dependency Graph
```
Deferred Providers
  ├─ DeferrableProvider interface
  ├─ provides() method
  ├─ Deferred manifest (bootstrap/cache/services.php)
  │   ├─ Generated by ProviderRepository::createManifest()
  │   └─ Maps services → provider classes
  ├─ Lazy registration on resolve
  │   ├─ make() → check deferredServices → register() → boot()
  │   └─ Fallback: scan all providers without manifest
  └─ when() conditional deferral
```
