# Rules

## Rule 1: Always Run `php artisan optimize` During Deployment
---
## Category
Reliability
---
## Rule
Include `php artisan optimize` in your deployment script to regenerate the package discovery cache.
---
## Reason
`bootstrap/cache/packages.php` is a static snapshot of discovered providers, facades, and aliases. Without regeneration, code changes (package additions, removals, provider modifications) are not reflected, causing fatal errors or missing services.
---
## Bad Example
```bash
# Deploy with no cache regeneration
git pull origin main
composer install --no-dev
# packages.php still references old providers
```
---
## Good Example
```bash
composer install --no-dev
php artisan optimize
```
---
## Exceptions
If `php artisan optimize` is already part of a broader deployment tool (Laravel Forge, Envoyer), ensure it specifically triggers the optimize command, not just config/route caching.
---
## Consequences Of Violation
Fatal errors from stale provider references after package removal; new providers not loaded; facade aliases missing; silent resolution failures.

## Rule 2: Use `dont-discover` for Development-Only Packages
---
## Category
Security
---
## Rule
List development-only packages (Debugbar, Telescope, IDE helpers) in `extra.laravel.dont-discover` in your root `composer.json`.
---
## Reason
Auto-discovered packages register eagerly in all environments. Excluding development packages via `dont-discover` prevents them from registering in production, avoiding both performance overhead and security risks.
---
## Bad Example
```json
{
    "extra": {
        "laravel": {
            // Everything auto-discovers — including dev tools in production
        }
    }
}
```
---
## Good Example
```json
{
    "extra": {
        "laravel": {
            "dont-discover": [
                "barryvdh/laravel-debugbar",
                "laravel/dusk",
                "nunomaduro/collision"
            ]
        }
    }
}
```
---
## Exceptions
Packages required in all environments (auth, payments, logging) should remain auto-discovered.
---
## Consequences Of Violation
Development tools running in production; sensitive information leaked; wasted bootstrap time; increased attack surface.

## Rule 3: Never Manually Edit `bootstrap/cache/packages.php`
---
## Category
Maintainability
---
## Rule
Do not modify `bootstrap/cache/packages.php` by hand — it is regenerated by Composer and Artisan commands.
---
## Reason
Manual edits are overwritten on the next `composer dump-autoload`, `php artisan optimize`, or package update. Any changes are temporary and create a false sense of configuration.
---
## Bad Example
```php
// Manually added a provider to bootstrap/cache/packages.php
// Next composer update overwrites the entire file
```
---
## Good Example
```php
// Use dont-discover or manual registration in bootstrap/providers.php
// These are the correct configuration points
```
---
## Exceptions
Debugging only — temporarily editing to test a hypothesis, with the understanding it will be overwritten.
---
## Consequences Of Violation
Lost configuration on next deploy; confusion when manual changes disappear; invalid state where cached providers don't match installed packages.

## Rule 4: Never Add an Auto-Discovered Provider to `bootstrap/providers.php`
---
## Category
Reliability
---
## Rule
Do not manually register a provider in `bootstrap/providers.php` if it is already auto-discovered by the package discovery system.
---
## Reason
Auto-discovered providers are appended to the provider list automatically. Manually adding them causes duplicate registration, which can result in duplicate binding exceptions or unexpected behavior.
---
## Bad Example
```php
// bootstrap/providers.php
return [
    // ...
    Spatie\Permission\PermissionServiceProvider::class, // Also auto-discovered!
];
```
---
## Good Example
```php
// bootstrap/providers.php
return [
    // Only non-discovered or manually excluded providers
    App\Providers\AppServiceProvider::class,
    App\Providers\DomainServiceProvider::class,
    // Spatie provider auto-discovers — not listed here
];
```
---
## Exceptions
When you have excluded a package via `dont-discover` and want to manually register it (e.g., for conditional/guarded registration). In that case, do NOT list it in `bootstrap/providers.php` — use `$app->register()` in a proxy provider.
---
## Consequences Of Violation
Duplicate provider registration; "Service provider already registered" errors; potential duplicate binding exceptions; registration order confusion.

## Rule 5: Audit Discovered Providers After Every Package Installation
---
## Category
Security
---
## Rule
Inspect `bootstrap/cache/packages.php` or run `php artisan about` to review which providers were discovered after each new package installation.
---
## Reason
A package's `extra.laravel.providers` may register providers unexpectedly — including ones that alter application behavior in security-relevant ways. Audit ensures you know exactly what runs during bootstrap.
---
## Bad Example
```bash
composer require vendor/package
# No verification of what was auto-discovered
```
---
## Good Example
```bash
composer require vendor/package
php artisan about --json | findstr provider
# Review the list to confirm only expected providers were added
```
---
## Exceptions
Well-known, trusted packages with stable provider lists may not require audit on every update — but still verify on initial installation.
---
## Consequences Of Violation
Unexpected providers registering in production; security-relevant bindings overridden; performance regressions from heavy packages; difficult-to-trace behavior changes.

## Rule 6: Use Manual Registration When Provider Ordering Is Critical
---
## Category
Reliability
---
## Rule
Register providers with specific ordering requirements in `bootstrap/providers.php` using explicit manual registration, rather than relying on auto-discovery ordering.
---
## Reason
Discovered providers always append after manual providers. If a discovered provider must register before a manual one (or after specific other providers), the ordering guarantee is lost. Manual registration gives explicit control over registration order.
---
## Bad Example
```php
// bootstrap/providers.php
return [
    App\Providers\ThirdPartyDependentProvider::class, // Needs ThirdParty to register first
    // ThirdParty is auto-discovered — registers AFTER this one
];
```
---
## Good Example
```php
// bootstrap/providers.php
return [
    App\Providers\ThirdPartyServiceProvider::class, // Manually registered for ordering
    App\Providers\ThirdPartyDependentProvider::class,
];
```
Also add to `composer.json`:
```json
{
    "extra": {
        "laravel": {
            "dont-discover": ["vendor/third-party"]
        }
    }
}
```
---
## Exceptions
When ordering does not matter — most providers operate independently and do not depend on each other's registration order.
---
## Consequences Of Violation
Provider A tries to modify a binding from Provider B, but Provider B hasn't registered yet; intermittent "Target class does not exist" errors depending on registration order.
