# Compilation Optimization

## Metadata
- **Domain:** Laravel Execution Lifecycle & Framework Internals
- **Subdomain:** Caching & Optimization
- **Last Updated:** 2026-06-02

## Executive Summary
Compilation optimization refers to the bundle of techniques Laravel uses to reduce bootstrap overhead by caching compiled representations of configuration, routes, events, views, and services. The `php artisan optimize` command orchestrates most of these, while others (OpCache, Composer autoloader optimization) are complementary infrastructure-level optimizations. Together, these techniques can reduce Laravel bootstrap time by 50-150ms per request.

## Core Concepts

### Cache Types
Config cache, route cache, event cache, view cache, services cache — each caches a different aspect of bootstrap.

### Compile Strategy
Move expensive operations (file parsing, Reflection, serialization) from request time to build/deploy time.

### Pre-computation
The optimized form is pre-computed and stored as plain PHP files that can be `require`d in microseconds.

### Composite Command
`php artisan optimize` runs multiple sub-commands in sequence to generate all caches.

### Symmetry
Every cache operation has a corresponding clear operation — `optimize` and `optimize:clear` are symmetric inverses.

## Mental Models

### The Meal Prep
Imagine cooking 50 meals on Sunday for the week ahead. Each meal is prepped, portioned, and stored. During the week, you just reheat — no chopping, no measuring, no cooking. Compilation optimization is the Sunday meal prep for Laravel requests.

## Internal Mechanics

### Cache Order Dependency
```
config:cache → route:cache → event:cache
    ↓             ↓
config.php    routes.php    events.php
    ↓
services.php (generated by optimize)
```

Each cache has specific dependencies:
- `config:cache` — standalone (no dependency)
- `route:cache` — depends on config being cached
- `event:cache` — depends on config being cached
- Services manifest — generated during `optimize`

### What optimize Includes
```
php artisan optimize:
  1. config:cache — merge all config files
  2. route:cache — compile routes (if no closures)
  3. Services manifest regeneration (side effect of bootstrap)
  NOTE: event:cache is NOT included in most Laravel versions
```

### File Locations
```
bootstrap/cache/
├── config.php     # Config array
├── routes-v3.php  # Serialized compiled routes
├── events.php     # Event listener mappings
└── services.php   # Deferred provider manifest

storage/framework/views/
└── a1b2c3d4.php   # Compiled Blade templates
```

## Patterns

### Clear-Before-Warm Pattern
Always clear old caches before building new ones: `optimize:clear` then `optimize`. This prevents stale + new hybrid state if the build fails partway.

### Cache-As-Deployment-Artifact Pattern
Cache files are treated as deployment artifacts — built during deployment, committed to the release, and consumed by requests. They are never built at request time.

### Layered Caching Pattern
Different cache types operate at different layers of the stack: bootstrap caches (config, routes, events, services) in `bootstrap/cache/`, compiled views in `storage/framework/views/`, and OpCache at the PHP engine level.

## Architectural Decisions

### Why separate cache files instead of one?
Separate files allow independent cache invalidation. A route change only requires `route:cache`, not a full rebuild of all caches.

### Why bootstrap/cache/ instead of storage/?
Bootstrap caches must be available before the storage system is initialized. The `bootstrap/` directory is outside the `storage/` directory tree and is available at the earliest bootstrap phase.

### Why event:cache is separate from optimize?
Historically, event caching was added later and wasn't integrated into the `optimize` command. This is a version-dependent behavior — check your Laravel version's documentation.

## Tradeoffs

| Benefit | Cost | Consequence |
|---|---|---|
| 50-150ms bootstrap reduction per request | Cache maintenance overhead | Must manage cache lifecycle in deployment |
| Deterministic application state | Changes invisible until cache rebuild | Development must run uncached |
| OpCache caches compiled files | Cache files contain serialized data | Version-dependent cache format |
| Separate caches for independent invalidation | Must know which cache to clear for each change | Developer education needed |

## Performance Considerations

- **Total bootstrap improvement:** 50-150ms reduction per request with all caches enabled.
- **Cache file sizes:** config 100-500KB, routes 200KB-1MB, events 10-50KB, services 5-15KB.
- **OpCache interaction:** all cache files benefit from OpCache — zero PHP parsing cost after first request.
- **Cache generation time:** 2-5 seconds for full `optimize` in typical applications.

## Production Considerations

- **Run optimize last in deployment:** After `composer install`, migrations, and other setup — caches should reflect the final state.
- **Run optimize:clear before renovating:** Always clear old caches before building new ones.
- **Use individual cache commands for targeted changes:** If only routes changed, `route:cache` is faster than full `optimize`.
- **Verify cache integrity:** After optimization, run health checks that exercise cached paths.
- **Document cache strategy:** What caches are used, when they're rebuilt, and how to clear them.

## Common Mistakes

- **Running optimize in development:** Changes don't take effect — confusing debugging sessions.
- **Not running event:cache separately:** Events not cached — auto-discovery runs on every request.
- **Optimize before migrations:** Cached routes reference non-existent columns.
- **Partial cache failure recovery:** optimize fails partway — config cached but routes not.

## Failure Modes

| Failure | Symptom | Root Cause | Mitigation |
|---|---|---|---|
| Partial cache state | Config cached, routes not | optimize fails on route:cache | Fix the issue and re-run full optimize |
| Stale across deployments | Mixed old/new behavior | optimize:clear not run before optimize | Always clear before build |
| Wrong env cached | Wrong config values | Cache built with wrong APP_ENV | Verify environment before caching |
| Cache permission errors | optimize command fails | bootstrap/cache/ not writable | Set correct permissions |

## Ecosystem Usage

- **Laravel Forge:** Deployment scripts include `php artisan optimize` as a standard step.
- **Laravel Vapor:** Caches are built during `vapor build` and included in the Lambda artifact.
- **Envoyer:** The zero-downtime deployment tool runs optimize after code sync.
- **Laravel Shift:** Automated upgrade tooling checks for cache-related configuration changes.

## Research Notes
- `php artisan optimize` runs `config:cache`, `route:cache`, and triggers services manifest generation.
- `event:cache` is NOT always included in `optimize` — check your Laravel version.
- When debugging post-deploy issues, start with `php artisan optimize:clear` to eliminate cache-related causes.
- The `optimize:clear` command is idempotent — running it multiple times is safe.
