# Services Cache

## Metadata
- **Domain:** Laravel Execution Lifecycle & Framework Internals
- **Subdomain:** Caching & Optimization
- **Last Updated:** 2026-06-02

## Executive Summary
The services cache (`bootstrap/cache/services.php`) is a deferred service provider manifest generated by `php artisan optimize`. It tracks which service providers are deferred, which services they provide, and the order in which providers should be registered. This file is consumed during application bootstrap to avoid scanning and resolving all provider classes on every request. It is the most misunderstood caching mechanism in Laravel because it is generated automatically but must be manually invalidated.

## Core Concepts
- **Deferred Providers:** Service providers that implement `DeferrableProvider` interface. Their services are not registered until explicitly requested via the container.
- **Provider Manifest:** The cached file maps each service provider to:
  - Its class name and path
  - The services it provides (for deferred resolution)
  - Whether it is deferred or eager
  - The provider registration order (priority)
- **Eager Providers:** Non-deferred providers whose `register()` and `boot()` methods run on every request. The manifest tracks these but they still execute.
- **Cache File Location:** `bootstrap/cache/services.php` — returns a serialized array of provider metadata.
- **Generation Trigger:** Running `php artisan optimize` or any command that calls `\Illuminate\Foundation\Application::servicesAreCached()`.

## Mental Models
- **Deferred Loading:** Like lazy loading in ORMs — services are not instantiated until first use. The manifest is the "map" that enables lazy resolution.
- **Registry Model:** The services cache is a registry of all providers and their services. Think of it like the Windows Registry: a centralized database that avoids scanning individual provider files.
- **Dependency Map:** The manifest is a dependency graph flattened into an ordered list. It tells Laravel: "these are the providers, this is what they provide, register them in this order."

## Internal Mechanics
1. **`\Illuminate\Foundation\Application::servicesAreCached()`** checks for `bootstrap/cache/services.php`. If present, it requires the file and populates the `$this->loadedProviders` and deferred service mappings.
2. **Manifest Structure:** The cached file contains:
   ```php
   <?php return array (
     'providers' => array (
       0 => 'App\\Providers\\AppServiceProvider',
       // ... all provider classes
     ),
     'eager' => array (
       'App\\Providers\\AppServiceProvider' => true,
       // ... eager providers
     ),
     'deferred' => array (
       'Illuminate\\Bus\\BusServiceProvider' => array('Illuminate\\Contracts\\Bus\\Dispatcher', 'Illuminate\\Contracts\\Bus\\Queue'),
       // ... deferred providers mapped to their services
     ),
     'when' => array (
       // deferred providers with `when()` dependencies
     ),
   );
   ```
3. **Deferred Resolution:** When the container resolves a service that matches a deferred provider, it registers that provider on the spot and resolves the service.
4. **Provider Ordering:** The manifest preserves the order of providers as they appear in `config/app.php`. Eager providers run in order; deferred providers are registered as needed.
5. **Regeneration:** The manifest is rebuilt by calling `$app->getProviderRepository()->createManifest()` or by clearing and running `optimize`.

## Patterns
- **Manifest-Based Lazy Loading:** The manifest enables deferred resolution without service provider autoloading. The container knows which provider handles which service without loading the provider class.
- **Eager vs. Deferred Separation:** Providers are categorized into two groups: always-run (eager) and on-demand (deferred). The manifest tracks both.
- **Automatic Generation, Manual Invalidation:** The manifest is generated automatically but is never invalidated automatically. Providers added or changed require explicit cache clearing.

## Architectural Decisions
- **Decision:** Generate services cache automatically as a side effect of various commands.
  - **Rationale:** Simplifies the developer experience — most developers don't think about deferred providers. But this leads to confusion when `services.php` is stale.
- **Decision:** Store provider paths as class names (strings), not file paths.
  - **Rationale:** Relies on the autoloader to resolve classes. File paths would require filesystem operations and break with class map changes.
- **Decision:** Separate deferred service mapping from provider registration.
  - **Rationale:** Allows the container to resolve services from deferred providers without registering the provider immediately. Critical for performance — many providers are never used on a given request.

## Tradeoffs

| Benefit | Cost | Consequence |
|---|---|---|
| Deferred providers only load when needed | Manifest becomes stale when providers change | Manual cache clear required after provider changes |
| ~10-30ms bootstrap improvement via eliminated provider scanning | Eager providers still run; only deferred providers benefit | Performance gain depends on provider structure |
| Automatic manifest generation reduces developer friction | Manifest regeneration is a side effect of other commands | Developers are unaware when the manifest was rebuilt |
| Enables lazy service resolution without autoloader coupling | Complex provider `when()` dependencies may not cache correctly | Rare edge cases with conditionally deferred providers |

## Performance Considerations
- **Bootstrap Overhead:** Without the manifest, Laravel must scan `config/app.php`, instantiate all providers, and check for deferred interfaces. This adds 15-40ms.
- **Deferred Provider Savings:** The real gain is avoiding `register()`/`boot()` calls for providers whose services are unused on the current request. An admin panel provider (e.g., Nova) that loads on every request despite only being used on `/admin/*` routes is wasted overhead.
- **Manifest File Size:** Typically 5-15KB for a medium application. Loading it is a single `require` — negligible cost.
- **Memory:** The manifest array is small (~100KB) and held in memory for the request lifetime. Each deferred provider is loaded only when its service is first resolved.

## Production Considerations
- **Clear services cache after adding/removing service providers.** Run `php artisan optimize:clear` or delete `bootstrap/cache/services.php` manually.
- **Stale manifest is the most common cause of "service provider not found" errors.** Always include `optimize:clear` in deployment scripts before the warmup phase.
- **`composer install`/`update` can invalidate provider paths.** Running `composer dump-autoload` followed by cache clear resolves path mismatches.
- **If you add a new deferred provider**, the manifest must be regenerated. The provider does not take effect until the manifest includes it.
- **Monitoring:** Use Telescope or Clockwork to measure the number of providers loaded per request. A high count of eager providers indicates opportunity for deferral.

## Common Mistakes
- **Adding a provider to `config/app.php` but not clearing the services cache.** The new provider does not register until the cache is regenerated.
- **Removing a provider from `config/app.php` but not clearing the services cache.** The old provider is still loaded from the manifest, potentially causing errors if the class no longer exists.
- **Assuming `php artisan optimize` always regenerates the services cache.** It does, but only if the destination file is writable. Watch for permission errors.
- **Not understanding the difference between deferred and eager providers.** The services cache only optimizes deferred providers. Eager providers execute on every request regardless.
- **Committing `bootstrap/cache/services.php` to version control.** This is environment-specific; it's already in `.gitignore`.

## Failure Modes
- **Stale Manifest After Provider Removal:** If a provider class is deleted from the filesystem but the manifest still references it, the application crashes when the container tries to register it.
- **Stale Manifest After Provider Addition:** The new provider's services are not available. Symptoms: binding resolution failures (`Target class does not exist`), missing services.
- **Partial Manifest Write:** Corrupted manifest causes a PHP parse error on bootstrap, taking down the entire application. Mitigation: atomic write.
- **Version Control Conflict:** If the manifest is accidentally committed, every environment with a different provider list breaks. The `.gitignore` prevents this by default.

## Ecosystem Usage
- **Laravel Horizon:** Horizon registers its own service provider which appears in the manifest. If cached before Horizon is installed, Horizon does not function.
- **Laravel Telescope:** Same pattern as Horizon — must be cached after installation.
- **Laravel Nova:** Nova is a deferred provider in many setups. The services cache enables Nova to load only on admin routes (if properly configured with middleware-based loading).
- **Package Development:** Package providers must be deferred if they register services conditionally. The services cache respects the `DeferrableProvider` interface.

## Research Notes
- The services cache was introduced in Laravel 5.0 and has remained largely unchanged. It was originally part of the `optimize` command's output.
- `DeferrableProvider` interface extends `Illuminate\Contracts\Support\DeferrableProvider` (Laravel 8+). Earlier versions used `$defer` property on the provider class.
- The `when()` method on providers allows conditional deferral based on other container bindings. This adds complexity to the manifest — the `when` array in the manifest maps provider to the bindings that trigger it.
- Laravel 11's service provider consolidation reduces the number of providers in `config/app.php`, simplifying the manifest.
