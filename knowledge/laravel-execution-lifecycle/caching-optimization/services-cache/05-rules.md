## Clear Cache After Every Service Provider Change
---
## Maintainability
---
## Always run `php artisan optimize:clear` after adding, removing, or reordering service providers in `config/app.php`.
---
## The services manifest (`bootstrap/cache/services.php`) is a snapshot of the provider list and their deferred status. Adding a provider without clearing the cache means the new provider is never registered. Removing a provider without clearing causes a fatal error when the container tries to resolve the old provider.
---
Adding a new service provider to `config/app.php` without clearing the cache.
---
```bash
php artisan optimize:clear
# Verify
php artisan optimize
```
---
## No common exceptions.
---
## Provider services unavailable after adding new packages; runtime fatal errors after removing providers; deployment rollbacks required.
---
## Never Edit bootstrap/cache/services.php Manually
---
## Maintainability
---
## Never modify `bootstrap/cache/services.php` by hand; always regenerate it via `php artisan optimize:clear` followed by `php artisan optimize`.
---
## The services manifest is machine-generated by `Illuminate\Foundation\ProviderRepository::createManifest()`. Manual edits are overwritten on the next `optimize` run and may introduce syntax errors that break the entire bootstrap.
---
```bash
# Editing the cached manifest by hand
nano bootstrap/cache/services.php
```
---
```bash
php artisan optimize:clear
php artisan optimize
```
---
## No common exceptions.
---
## Syntax errors break application bootstrap; manual edits lost on next deploy; debugging time wasted on phantom provider issues.
---
## Use Deferred Providers for Infrequently Used Services
---
## Performance
---
## Implement `DeferrableProvider` and the `provides()` method for service providers whose services are not needed on every request.
---
## Deferred providers are never loaded during the register or boot phase. Their `register()` and `boot()` methods execute only when a service they provide is first resolved via the container — eliminating bootstrap cost for unused services.
---
```php
// Eager provider that runs on every request
class AnalyticsServiceProvider extends ServiceProvider
{
    public function register(): void
    {
        $this->app->singleton(Analytics::class);
    }
}
```
---
```php
// Deferred provider — only loads when Analytics is first resolved
class AnalyticsServiceProvider extends ServiceProvider implements DeferrableProvider
{
    public function register(): void
    {
        $this->app->singleton(Analytics::class);
    }
    public function provides(): array
    {
        return [Analytics::class];
    }
}
```
---
## Providers that register event listeners, middleware, or routes in their `boot()` method — deferred providers skip `boot()` until first resolution.
---
## 15-40ms bootstrap overhead per request from eager providers whose services are never used; unnecessary memory allocation.
---
## Run optimize After Every Composer Package Change
---
## Maintainability
---
## Regenerate the services manifest after every `composer install`, `composer update`, or manual addition/removal of packages that register service providers.
---
## Composer operations may add or remove package service providers. Without regenerating the manifest, the cached provider list becomes stale — new providers are missing, removed providers cause fatal errors.
---
```bash
composer require spatie/laravel-medialibrary
# Missing: php artisan optimize:clear + optimize
```
---
```bash
composer require spatie/laravel-medialibrary
php artisan optimize:clear
php artisan optimize
```
---
## No common exceptions.
---
## Package features unavailable after install; "class not found" errors for removed package providers; inconsistent application behavior across environments.
---
## Monitor Eager Provider Count for Optimization Opportunities
---
## Scalability
---
## Use `app()->getLoadedProviders()` to track the number of eager providers loading on every request and identify deferral opportunities.
---
## Each eager provider consumes bootstrap time for `register()` and `boot()`. A growing provider count indicates the application is accumulating bootstrap overhead. Monitoring enables data-driven decisions to mark providers as deferred.
---
```php
// Never checking how many providers load per request
```
---
```php
// In a service provider or Tinker
$providers = app()->getLoadedProviders();
$eagerCount = count(array_filter($providers));
// Target: keep under 30 eager providers
```
---
## No common exceptions.
---
## Unnoticed bootstrap bloat as the application grows; avoidable performance degradation at scale.
---
## Understand That optimize Does Not Always Regenerate the Manifest
---
## Framework Usage
---
## Verify the output of `php artisan optimize` confirms the services manifest was regenerated; do not assume it succeeds silently.
---
## The services manifest generation is a side effect of a fresh application bootstrap within the `optimize` command. Permissions errors, missing directories, or PHP errors can cause the manifest to remain stale without a clear error message.
---
```bash
php artisan optimize
# Assuming success without checking output
```
---
```bash
php artisan optimize
if ($?) {
    Write-Host "Manifest regenerated successfully"
} else {
    Write-Host "Optimize failed — check bootstrap/cache permissions"
}
```
---
## No common exceptions.
---
## Stale manifest serves old provider list; new providers silently missing; debugging time wasted when providers "work in dev but not production."
