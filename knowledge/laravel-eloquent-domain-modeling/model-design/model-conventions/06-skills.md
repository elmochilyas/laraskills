# Skill: Verify and Align Model-Table Mapping Conventions

## Purpose

Ensure each Eloquent model correctly maps to its database table by validating that naming conventions are followed or explicit overrides are correctly configured, preventing silent data corruption.

## When To Use

- Setting up new models to ensure correct table mapping
- Auditing existing models for convention compliance
- After schema migrations that rename or restructure tables
- Onboarding to a project with existing models

## When NOT To Use

- The schema is auto-generated by Laravel conventions (no overrides needed)
- Models fully rely on conventions with no custom table or FK names

## Prerequisites

- List of all model classes in the application
- Database schema documentation or migration files
- Access to database to verify table and column names

## Inputs

- Model class names
- Expected table names (by convention: snake_case plural of class)
- Actual table names (from schema)
- Foreign key column names

## Workflow

1. For each model, determine the convention-based table name:
   - `User` → `users`, `FlightTicket` → `flight_tickets`, `Person` → `people`
   - Verify using `Str::pluralStudly(class_basename($model))`
2. Compare the conventional name against the actual table name:
   - If they match: omit `$table` property
   - If they differ: set `protected $table = 'actual_table_name'`
3. Add a comment explaining why the override exists:
   ```
   // Legacy schema from migrated system — table was named before Laravel convention
   protected $table = 'catalog_products'
   ```
4. Verify foreign key conventions in relationship methods:
   ```
   public function author(): BelongsTo
   {
       return $this->belongsTo(User::class, 'author_id')  // Explicit FK when non-standard
   }
   ```
5. Write a schema assertion test for convention overrides:
   ```
   public function test_table_exists(): void
   {
       $this->assertTrue(Schema::hasTable('customer_orders'))
   }
   ```

## Validation Checklist

- [ ] Every model has the correct `$table` (explicitly or by convention)
- [ ] Convention overrides include a comment explaining why
- [ ] Foreign keys are explicitly specified when they diverge from `{model_name}_id`
- [ ] Custom pivot tables follow alphabetical singular name convention
- [ ] Schema assertion tests exist for models with overrides
- [ ] No unnecessary configuration properties are declared (matching conventions are omitted)

## Common Failures

- **Silent wrong-table mapping**: `Person` model maps to `people` by convention but the actual table is `persons`. Data writes go to the wrong table.
- **Missing FK override**: `Post::author()` relationship expects `user_id` but the column is `author_id`. Queries silently return empty results.
- **Missing schema test**: A migration renames `customer_orders` to `orders`, but `Order` model still references `customer_orders`. Without a test, all queries silently fail.

## Decision Points

- **$table vs convention**: Set `$table` only when the actual name differs. Don't set it when convention is correct.
- **Explicit FK vs convention**: Specify the foreign key explicitly in relationship methods when the column name doesn't match `{model_name}_id`. Omit it only when the convention is correct.

## Performance Considerations

- Convention verification is a compile-time concern — no runtime impact
- Schema assertion tests run during test suite — negligible overhead

## Security Considerations

- Incorrect table mapping could expose or modify unintended data — verify mappings carefully

## Related Rules

- Rely on Conventions for New Projects
- Override `$table` Explicitly When Convention Fails
- Specify Foreign Keys Explicitly for Non-Standard Relationships
- Test Model-Table Mapping with Schema Assertions
- Document Every Convention Override with a Reason
