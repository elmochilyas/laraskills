# Overview

Token usage monitoring tracks the consumption of LLM tokens (input + output) across models, features, users, and time periods. It is the foundation for cost management, capacity planning, and usage-based billing in LLM-powered applications.

Token monitoring is unique to LLM observability because tokens are both the unit of cost ($ per 1M tokens) and the unit of performance (latency scales with token count). Unlike traditional metrics (request count, error rate), token usage is a direct cost driver — a single user running a large prompt can cost more than 1,000 typical API calls.

Engineers should care because LLM costs can easily run away without monitoring. A prompt injection attack can generate millions of output tokens and cost $10,000+ in hours. A misconfigured agent can loop indefinitely, consuming tokens in an infinite request/response cycle.

# Core Concepts

**Input Tokens:** Tokens consumed by the prompt sent to the LLM. Includes the system message, conversation history, context (RAG), and user message. Input tokens are typically cheaper per-token than output tokens.

**Output Tokens:** Tokens generated by the LLM in response. Output tokens are more expensive and harder to predict. A model might generate 100 or 10,000 tokens for similar prompts depending on configuration.

**Per-Model Cost:** Each LLM model has different per-token pricing. GPT-4 costs ~10x GPT-3.5-turbo. Tracking per-model cost enables optimization decisions.

**Per-User Cost:** Total token usage attributed to a single user. Essential for usage-based billing, abuse detection, and understanding high-value users.

**Per-Feature Cost:** Total token usage attributed to a specific feature (chat, summarization, code generation). Enables ROI analysis — is the feature generating enough value to justify its LLM cost.

**Cost Anomaly:** An unexpected spike in token usage or cost. Detection requires baseline comparison — a user who normally uses 1,000 tokens/day suddenly using 100,000 tokens.

**Token Budget:** A spending limit for a user, feature, or time period. When the budget is exceeded, the feature degrades (switch to cheaper model) or blocks (return error). Prevents cost overruns.

# When To Use

- **Any production LLM feature** — without token monitoring, costs are invisible
- **Multi-model applications** — track which model is used for each request
- **User-facing AI features** — enable usage-based billing or fair usage limits

# When NOT To Use

- **Non-production development** where API costs are negligible
- **Internal-only tools** where cost tracking is unnecessary

# Best Practices

**Track token usage per request, aggregated per minute.** Use OTel metrics for minute-level aggregation. Log per-request counts for detailed analysis. Metrics provide cost monitoring; logs provide debugging.

**Set per-user token budgets.** Define daily or monthly token limits per user. When approaching the limit, warn the user. When the limit is exceeded, switch to a cheaper model or block requests.

**Monitor cost per feature.** Track which features consume the most tokens. If the "summarization" feature costs $500/month but only generates $50 in value, consider whether to keep it.

**Alert on cost anomalies.** Baseline normal token usage per user per day. Alert when any user exceeds 5x their normal usage. This catches prompt injection attacks and runaway agents.

**Log token usage breakdown by model.** If you upgrade from GPT-3.5 to GPT-4, you need per-model tracking to understand cost impact. Model upgrades can 10x costs unexpectedly.

# Architecture Guidelines

Token monitoring architecture:
1. **SDK instrumentation:** Track tokens at the LLM API call site. Record prompt_tokens and completion_tokens returned by the API response
2. **OTel metrics:** Create Counter instruments for tokens by model, feature, user
3. **Cost calculation:** Apply per-model pricing in the observability backend or application. Calculate cost = input_tokens × input_price + output_tokens × output_price
4. **Alerting:** Set cost anomaly detection based on baseline comparison
5. **Dashboards:** Grafana dashboard showing token usage and cost trends per model, per feature, per user

# Performance Considerations

- **Token counting overhead:** The API response already includes token counts. Tracking them adds <1μs. No additional tokenization cost
- **Metric cardinality:** Tracking per-user token usage creates high-cardinality metrics. Use log-based storage for per-user detail; use aggregated metrics for per-feature and per-model
- **Cost calculation frequency:** Calculate costs in batch (every minute) rather than per-request to avoid floating-point overhead on every request

# Security Considerations

- **Per-user cost tracking reveals user activity patterns.** Token usage patterns can reveal which features a user is interacting with. Ensure cost dashboards have access control
- **Budget bypass:** Users should not be able to see or manipulate their own token budgets. Budget enforcement must be server-side, not client-side
- **API key abuse:** Monitor token usage per API key. A compromised key will show abnormal token consumption patterns

# Common Mistakes

**No per-user tracking.** Tracking total token usage only, without per-user breakdown. When costs spike, cannot determine which user or feature is responsible.

**No cost anomaly alerts.** Only monitoring total cost, not cost anomalies. A cost spike from $10/day to $100/day goes unnoticed until the monthly bill arrives.

**Mixing model costs.** Tracking all models under a single "LLM cost" metric. Cannot determine whether the cost increase is from model upgrades or increased usage.

**Ignoring output token costs.** Tracking only input tokens because they are easier to count. Output tokens are typically 2-3x more expensive per token and should be tracked separately.

# Anti-Patterns

**Token tracking after the fact.** Calculating token usage from LLM API billing data at the end of the month. By the time costs are visible, the budget is already exceeded.

**No user attribution.** API calls tracked without user context. Cannot identify which user is responsible for high token usage. Abuse detection is impossible.

**No budget enforcement.** Setting budgets but not enforcing them. Budget notification is sent but the feature continues to operate. Costs continue to accumulate.

**Prometheus-only token tracking.** Storing per-user token counts as a Prometheus metric with user_id label. High cardinality destroys Prometheus performance. Use logs for per-user, metrics for aggregated.

# Examples

**Cost tracking OTel metric:**
```php
$costCounter = $meter->createCounter('llm.cost', ['unit' => 'USD']);
$costCounter->add($cost, [
    'model' => $response->model,
    'feature' => 'chat',
    'user_tier' => $user->tier,
]);
```
