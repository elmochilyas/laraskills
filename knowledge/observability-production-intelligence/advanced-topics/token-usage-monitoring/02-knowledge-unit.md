## Executive Summary

Token usage monitoring tracks LLM token consumption across models, features, users, and time periods — the foundation for cost management, capacity planning, and usage-based billing in LLM-powered applications. Unlike traditional metrics (request count, error rate), token usage is a direct cost driver — a prompt injection attack can generate millions of output tokens and cost $10,000+ in hours.

---

## Core Concepts

- **Input Tokens:** Tokens consumed by the prompt — system message, conversation history, context (RAG), user message, typically cheaper per-token
- **Output Tokens:** Tokens generated by the LLM — more expensive, harder to predict, 2-3x input token cost typically
- **Per-Model Cost:** Each model has different per-token pricing — GPT-4 costs ~10x GPT-3.5-turbo
- **Per-User Cost:** Total token usage attributed to a single user — essential for usage-based billing and abuse detection
- **Per-Feature Cost:** Token usage for a specific feature — enables ROI analysis
- **Cost Anomaly:** Unexpected spike in token usage — detection requires baseline comparison
- **Token Budget:** Spending limit for user, feature, or time period — degrades or blocks when exceeded

---

## Mental Models

- **Fuel Gauge Model:** Token monitoring is the fuel gauge on your LLM car — you need to know how much fuel (tokens) you're burning, what grade (model) you're using, and whether anyone is siphoning the tank (abuse)
- **Electric Meter Model:** Tokens are kilowatt-hours of LLM usage — the meter runs continuously, billing accumulates per-unit, and a faulty appliance (bug, runaway agent) can spike your bill to astronomical levels
- **Checkbook Register Model:** Each token usage is a check written against your budget. Without registering every check, you don't know how much is left in the account

---

## Internal Mechanics

Token monitoring flows through a pipeline: LLM API call returns token counts (prompt + completion) in the response → application extracts counts and records them via OTel metrics (Counter instruments) for aggregation → cost calculation applies per-model pricing to token counts → data stored both as metrics (aggregated per feature, model) and logs (per-user detail) → dashboards visualize trends → anomaly detection compares current usage to baseline → budget enforcement checks limits before processing requests.

---

## Patterns

- **Metrics for Aggregates, Logs for Per-User:** Use OTel Counter metrics for per-feature and per-model token counts (low cardinality). Use logs for per-user token detail (high cardinality). Benefit: Prometheus-compatible metrics + detailed user attribution. Tradeoff: two data stores to manage.
- **Cost Anomaly Detection:** Baseline normal token usage per user per day — alert when any user exceeds 5x their normal usage. Benefit: catches prompt injection and runaway agents. Tradeoff: requires historical usage data.
- **Per-User Token Budgets:** Define daily/monthly token limits per user. Warn when approaching limit; switch to cheaper model or block when exceeded. Benefit: prevents cost overruns. Tradeoff: legitimate high-usage users may be affected.

---

## Architectural Decisions

**Track input AND output tokens separately.** They have different costs (output 2-3x more expensive). Track them separately for accurate cost attribution.

**Set per-user token budgets.** Define daily limits per user. When exceeding, switch to cheaper model or block requests. Budget enforcement must be server-side.

**Alert on cost anomalies.** Baseline normal usage per user — alert when exceeding 5x baseline. Catches prompt injection attacks and runaway agents.

---

## Tradeoffs

| Benefit | Cost | Consequence |
|---|---|---|
| Cost tracking prevents runaway spending | Requires instrumentation at each LLM call site | Essential — AI costs can spiral without monitoring |
| Per-user tracking enables abuse detection | High cardinality incompatible with metrics | Use logs for per-user, metrics for aggregates |
| Per-model cost comparison enables optimization | Model pricing changes frequently | Update cost calculation when models change |

---

## Performance Considerations

API response already includes token counts — tracking them adds <1μs. Per-user token tracking creates high-cardinality data — use logs for per-user detail, metrics for per-feature and per-model. Calculate costs in batch (every minute) rather than per-request to avoid floating-point overhead.

---

## Production Considerations

Per-user cost tracking reveals user activity patterns — ensure cost dashboards have access control. Users should not be able to see or manipulate their own token budgets — enforcement must be server-side. Monitor token usage per API key — a compromised key shows abnormal consumption patterns.

---

## Common Mistakes

**No per-user tracking** — tracking total token usage only. When costs spike, cannot determine which user or feature is responsible.

**No cost anomaly alerts** — only monitoring total cost. A cost spike from $10/day to $100/day goes unnoticed until the monthly bill arrives.

**Mixing model costs** — tracking all models under single "LLM cost" metric. Cannot determine cost source from model upgrades or increased usage.

**Ignoring output token costs** — tracking only input tokens. Output tokens are 2-3x more expensive and should be tracked separately.

---

## Failure Modes

**Cost explosion from prompt injection:** Attacker crafts prompt that causes model to generate excessive output tokens. Detection: cost anomaly alert for specific user. Mitigation: input guardrails; per-user budgets; cost anomaly detection.

**Budget bypass:** Client-side budget enforcement disabled by user. Detection: server-side costs exceed expected. Mitigation: always enforce budgets server-side; never trust client-side limits.

**Cardinality explosion from per-user metrics:** Using `user_id` as a Prometheus label creates millions of time series. Detection: Prometheus storage growth; query slowdown. Mitigation: use logs for per-user tracking; metrics for aggregated views only.

---

## Ecosystem Usage

Token monitoring uses OTel Counter instruments for aggregated token counts. Laravel applications integrate with `laravel/ai` SDK or direct API clients. Per-user tracking uses application database for detailed logs. Cost anomaly detection uses Prometheus alerting or custom alerting logic.
