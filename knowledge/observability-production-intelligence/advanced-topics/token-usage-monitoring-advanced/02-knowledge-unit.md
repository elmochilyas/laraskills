## Executive Summary

Token usage is the primary cost driver for LLM-powered Laravel applications. Monitoring token consumption per model, user, feature, and time period enables cost attribution, budget control, and anomaly detection. OTel metrics (Histogram, Counter) and span attributes (`gen_ai.response.usage.*`) provide the instrumentation foundation. Combining token counts with API pricing data yields real-time cost dashboards.

---

## Core Concepts

- **Prompt tokens:** Input tokens sent to the model (user message + system prompt + conversation history)
- **Completion tokens:** Output tokens generated by the model — typically 2-4x more expensive per token
- **Total tokens:** Prompt + completion; basis for API billing
- **Cost per 1K tokens:** Provider-specific pricing (e.g., GPT-4o: $2.50/1M input tokens, $10/1M output tokens)
- **Token counting methods:** Provider API response includes usage, or use `tiktoken`-compatible tokenizers offline

---

## Mental Models

- **Utility Meter Model:** Token monitoring is like tracking electricity usage per appliance — you need per-circuit (per-feature) and per-device (per-user) meters to understand your bill
- **Gas Gauge Model:** Token budgets are fuel gauges — when you're below empty, the feature stops. Without the gauge, you run out of gas (budget) unexpectedly
- **Receipt Itemization Model:** Each LLM call generates a receipt with items (input tokens, output tokens). Cost tracking itemizes these receipts per user and per feature for accounting

---

## Internal Mechanics

Token counts are captured from the LLM API response metadata (`response.usage.prompt_tokens`, `response.usage.completion_tokens`, `response.usage.total_tokens`). These values are recorded as OTel span attributes and also emitted as OTel Counter/Histogram metrics. Cost calculation multiplies token counts by per-model pricing configured externally. Aggregated metrics feed dashboards and alerting rules.

---

## Patterns

- **Separate Input/Output Tracking:** Track prompt and completion tokens separately because output tokens cost 2-4x more. Benefit: accurate cost modeling. Tradeoff: double the metric dimensions.
- **User Tier Aggregation:** Aggregate per-user token tracking to user tier (free/pro/enterprise) in metrics. Benefit: avoids high-cardinality Prometheus metrics. Tradeoff: per-user detail available only in traces.
- **Externalized Pricing Config:** Store cost per model as external config (env vars or API endpoint). Benefit: pricing updates don't require code changes. Tradeoff: additional config management.

---

## Architectural Decisions

**Use OTel Counter for cumulative token totals and Histogram for per-request distribution.** Counters give total consumption; histograms show request-to-request variation.

**Record `gen_ai.response.usage.total_tokens` as a span attribute on every LLM call.** This enables cost attribution at the trace level.

**Implement budget alerts at feature level, not just aggregate.** Feature-level alerts catch anomalous usage patterns that aggregate alerts miss.

---

## Tradeoffs

| Benefit | Cost | Consequence |
|---|---|---|
| Granular cost tracking enables accurate chargeback | Per-user tracking adds metric cardinality | Use user tiers for metrics; per-user detail in traces |
| Real-time cost monitoring prevents budget overruns | Dashboard refresh adds monitoring infrastructure cost | Essential for cost-sensitive AI features |
| Model-level cost data enables optimization decisions | Must maintain per-model pricing config | Pricing updates are infrequent |

---

## Performance Considerations

Token monitoring adds <1ms per LLM API call; captured from API response metadata. Cost calculation is a simple multiplication with negligible CPU cost. Token tracking storage adds ~100 bytes per LLM call record. Real-time dashboards should use minimum 5-second refresh to avoid excessive load.

---

## Production Considerations

Token usage data may reveal business-sensitive information about feature adoption. Per-user token tracking has privacy implications under GDPR; use anonymous user tiers where possible. API keys or tokens should never appear in metric labels or span attributes.

---

## Common Mistakes

**Ignoring prompt vs completion cost differential** — output tokens 2-4x more expensive per token. Track separately.

**Not tracking model version** — GPT-4o and GPT-4o-mini have 20x cost difference. Always include model dimension.

**No budget alerts** — discovering $10,000+ monthly bill after the fact. Set daily budget alerts at 80% threshold.

**Counting only API-reported tokens** — streaming responses need final stream chunk parsing.

---

## Failure Modes

**Cost explosion from prompt injection:** Attack generates millions of output tokens. Detection: cost anomaly alert for user exceeding 5x baseline. Mitigation: per-user budgets; input guardrails.

**Undersampled cost data:** Low sampling rate misses cost events. Detection: cost dashboard vs API billing discrepancy. Mitigation: sample LLM traces at 100%.

**High-cardinality metric blowup:** Per-user metric labels overwhelm Prometheus. Detection: Prometheus ingestion failures. Mitigation: aggregate to user tier in metrics.

---

## Ecosystem Usage

Token monitoring integrates with OTel Metrics API (Counter, Histogram), OTel span attributes, and Prometheus for cost dashboards. The application records token counts per request, aggregates them via OTel metrics, and exports to a metrics backend. Grafana dashboards display cost trends per model, per feature, and per user tier.

## Research Notes

OpenAI and Anthropic both return `usage` in API responses with `prompt_tokens`, `completion_tokens`, `total_tokens`. Token costs for GPT-4o: $2.50/1M input, $10/1M output; Claude Sonnet 4: $3/1M input, $15/1M output. OTel semantic conventions for LLM cost are still emerging; current approach is to multiply tokens by price in the dashboard layer. The fastest-growing cost risk for Laravel AI apps is accumulation from long conversation histories (prompt tokens).
