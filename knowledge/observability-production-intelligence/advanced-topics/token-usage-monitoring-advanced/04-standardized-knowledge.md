## Overview
Token usage is the primary cost driver for LLM-powered Laravel applications. Monitoring token consumption per model, user, feature, and time period enables cost attribution, budget control, and anomaly detection. OTel metrics (Histogram, Counter) and span attributes (`gen_ai.response.usage.*`) provide the instrumentation foundation. Combining token counts with API pricing data yields real-time cost dashboards.

## Core Concepts
- **Prompt tokens**: Input tokens sent to the model (user message + system prompt + conversation history)
- **Completion tokens**: Output tokens generated by the model
- **Total tokens**: Prompt + completion; basis for API billing
- **Cost per 1K tokens**: Provider-specific pricing (e.g., GPT-4o: $2.50/1M input tokens, $10/1M output tokens)
- **Token counting methods**: Provider API response includes usage, or use `tiktoken`-compatible tokenizers offline

## When To Use
- Any Laravel application using LLM APIs with measurable token consumption
- Cost attribution per user, team, or feature for internal chargeback
- Budget management and anomaly detection for AI feature spending
- Model comparison and optimization decisions (choosing between GPT-4o, Claude, etc.)

## When NOT To Use
- Applications without LLM integration (no token consumption to track)
- Free-tier or negligible-cost LLM usage where monitoring overhead exceeds benefit
- Simple proof-of-concept AI features where cost is not a concern

## Best Practices
- Record `gen_ai.response.usage.total_tokens` as a span attribute on every LLM call
- Separate prompt token vs completion token tracking because output tokens cost 2-4x more
- Include `model` and `feature` dimensions on token counter metrics for granular cost breakdown
- Set daily/weekly token budgets per feature; alert when consumption exceeds 80%
- Use Histogram for per-request token distribution and Counter for cumulative totals
- Multiply token counts by current model pricing in the dashboard layer (PromQL or Grafana transformations)

## Architecture Guidelines
- Use OTel Counter for cumulative token totals and Histogram for per-request distribution
- Store cost per model as external config (env vars or API endpoint) so pricing updates don't require code changes
- Aggregate per-user token tracking to user tier (free/pro/enterprise) in metrics; keep per-user detail in traces only
- Implement budget alerts at feature level, not just aggregate, to catch anomalous usage patterns
- Record cache hit/miss as a metric dimension; cached responses save tokens but need separate modeling

## Performance Considerations
- Token monitoring adds <1ms per LLM API call; captured from API response metadata
- Cost calculation is a simple multiplication with negligible CPU cost
- Token tracking storage adds ~100 bytes per LLM call record
- Real-time dashboards should use minimum 5-second refresh to avoid excessive load

## Security Considerations
- Token usage data may reveal business-sensitive information about feature adoption
- Per-user token tracking has privacy implications under GDPR; use anonymous user tiers where possible
- API keys or tokens should never appear in metric labels or span attributes

## Common Mistakes
| Mistake | Cause | Consequence | Better Approach |
|---|---|---|---|
| Ignoring prompt vs completion cost differential | Assuming uniform pricing | Output tokens 2-4x more expensive per token | Track prompt and completion separately |
| Not tracking model version | Forgetting to add model dimension | GPT-4o and GPT-4o-mini have 20x cost difference | Always include model dimension |
| No budget alerts | Assuming costs are negligible | Discovering $10,000+ monthly bill after the fact | Set daily budget alerts at 80% threshold |
| Counting only API-reported tokens | Missing streaming final stats | Under-counting tokens from streaming responses | Parse final stream chunk for usage |

## Anti-Patterns
- **Per-user ID as metric dimension**: Creates high cardinality for Prometheus; aggregate to user tier instead. Keep per-user detail in traces.
- **Token-to-cost conversion at write time**: Model pricing changes; externalize pricing to config rather than hardcoding in application code.
- **Ignoring cached responses**: Cache hits save tokens but hide usage; track them separately with `is_cached` dimension.

## Examples
```php
$tokenCounter = $meter->createCounter('gen_ai.tokens.total', [
    'unit' => 'tokens',
    'description' => 'Total LLM tokens consumed',
]);
$tokenCounter->add($totalTokens, [
    'gen_ai.model' => $model,
    'gen_ai.system' => 'openai',
    'feature' => 'chat-assistant',
    'user_tier' => $userTier,
]);
```

## AI Agent Notes
- OpenAI and Anthropic both return `usage` in API responses with `prompt_tokens`, `completion_tokens`, `total_tokens`
- Token costs for GPT-4o: $2.50/1M input, $10/1M output; Claude Sonnet 4: $3/1M input, $15/1M output
- OTel semantic conventions for LLM cost are still emerging; current approach is to multiply tokens × price in the dashboard layer
- The fastest-growing cost risk for Laravel AI apps is accumulation from long conversation histories (prompt tokens)
