# Rule 1: Install Auto-Instrumentation Alongside Manual Instrumentation

**Condition:** Setting up OTel in a Laravel application.

**Action:** Install auto-instrumentation packages for PDO, Guzzle, Redis, and Laravel framework. This covers database queries, HTTP client calls, cache operations, and framework events. Add manual spans for business logic (order processing, payment flow).

**Consequence:** Auto-instrumentation provides broad coverage with zero code changes. Manual instrumentation adds depth for business-critical paths. Both together provide complete observability.

# Rule 2: Review Auto-Instrumentation Attributes for Sensitive Data

**Condition:** Auto-instrumentation packages are capturing span attributes.

**Action:** Review the attributes generated by each auto-instrumentation package. PDO captures SQL query text. Guzzle captures full URLs. Configure attribute filtering or sanitization for sensitive data.

**Consequence:** Attribute review prevents sensitive data from appearing in tracing backends. Without review, database queries containing PII and API URLs containing tokens are stored in traces.

# Rule 3: Configure Sampling for Auto-Instrumented Spans

**Condition:** Auto-instrumentation creates spans for high-frequency operations.

**Action:** Use head-based sampling (via Sampler configuration) or tail-based sampling (via Collector) to reduce auto-instrumented span volume. Health check endpoints and polling routes add significant noise.

**Consequence:** Sampling reduces storage costs and backend load while preserving trace context for important requests. Without sampling, health check traces account for 50%+ of trace storage.

# Rule 4: Test Auto-Instrumentation After PHP Version Upgrades

**Condition:** Upgrading PHP version (e.g., 8.1 → 8.2 → 8.3).

**Action:** Run auto-instrumentation test suite after each PHP upgrade. Verify spans are created with correct attributes. Check that no instrumentation packages throw errors or warnings.

**Consequence:** PHP auto-instrumentation depends on internal PHP function interception, which can change between PHP versions. Testing catches regressions before they silently break observability.

# Rule 5: Keep Auto-Instrumentation Packages Updated

**Condition:** Maintaining OTel dependencies.

**Action:** Update auto-instrumentation packages alongside the libraries they instrument. When upgrading Laravel, Guzzle, or Redis, also upgrade the corresponding instrumentation package.

**Consequence:** Up-to-date packages maintain compatibility with instrumented libraries. Outdated instrumentation packages may silently fail to create spans, creating observability gaps.

# Rule 6: Do Not Use Auto-Instrumentation as Complete Solution

**Condition:** Teams relying on auto-instrumentation for all observability.

**Action:** Audit auto-instrumentation coverage against application features. If business logic (checkout, registration, payment) is not covered by auto-instrumentation, add manual spans for those paths.

**Consequence:** Auto-instrumentation covers libraries, not business logic. Relying on it exclusively leaves the most important parts of the application uninstrumented.

# Rule 7: Configure Resource Attributes for Auto-Instrumentation Identity

**Condition:** Auto-instrumentation SDK initialization.

**Action:** Set `service.name`, `service.version`, and `deployment.environment` in Resource before loading auto-instrumentation packages. Auto-instrumentation uses the global Resource for span creation.

**Consequence:** Proper resource configuration ensures auto-instrumented spans are correctly attributed. Without it, auto-generated spans are indistinguishable across services.

# Rule 8: Monitor Auto-Instrumentation Error Logs

**Condition:** Auto-instrumentation packages running in production.

**Action:** Review application error logs for instrumentation-related warnings. Auto-instrumentation hooks may produce errors when instrumented libraries are used in unexpected ways.

**Consequence:** Error monitoring catches instrumentation failures. A failed hook silently stops creating spans for that library — without error monitoring, the observability gap goes unnoticed.
