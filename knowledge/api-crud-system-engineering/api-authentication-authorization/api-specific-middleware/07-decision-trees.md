## ForceJson Middleware Placement — API Group vs Global Stack

---

## Decision Context

Should ForceJson middleware be placed in the API middleware group or the global middleware stack? Arises during API middleware configuration in bootstrap/app.php or Kernel.php.

---

## Decision Criteria

* route scope — which routes need JSON enforcement
* impact on web routes — HTML views and redirects must not be affected
* application architecture — API-only app vs mixed API+web app
* middleware ordering — position relative to auth and error handling

---

## Decision Tree

Does your application serve web routes (HTML, Blade, Inertia)?
↓
YES → Place ForceJson in API middleware group only
NO → Is the app API-only with zero web routes?
    YES → API group is still sufficient; global not needed
    NO → API group (always the correct choice)

---

## Rationale

ForceJson in the global stack breaks web routes by forcing Laravel to return JSON errors instead of HTML views. The API middleware group is the intended location and keeps web routes unaffected. API-only applications benefit from the same placement — the api group captures all API routes.

---

## Recommended Default

**Default:** Prepend to the `api` middleware group
**Reason:** Targets only API routes, preserves web route behavior, consistent with Laravel conventions.

---

## Risks Of Wrong Choice

Global placement: web routes return JSON errors instead of HTML, broken Inertia/Blade responses. No ForceJson at all: unauthenticated API requests return HTML login redirects (302) instead of JSON 401 responses.

---

## Related Rules

- Always Include ForceJson in the API Middleware Group (from 05-rules.md)
- Never Place ForceJson in the Global Middleware Stack (from 05-rules.md)

## Request ID Strategy — Server-Generated vs Client-Provided

---

## Decision Context

Should request IDs be generated by the server or accepted from the client? Arises when implementing request tracing middleware.

---

## Decision Criteria

* trust — client-provided IDs may collide or be malicious
* compatibility — clients may already generate their own trace IDs
* log correlation — consistent ID format across distributed systems
* validation overhead — additional processing for client-provided IDs

---

## Decision Tree

Does the client already send an X-Request-Id header?
↓
YES → Is the client's UUID format valid?
    YES → Accept client's request ID (propagate through system)
    NO → Generate server-side UUID and override
NO → Generate server-side UUID

---

## Rationale

Accepting valid client-provided request IDs enables end-to-end tracing across distributed systems where upstream services already generate trace IDs. Server generation ensures uniqueness when clients don't provide IDs. Validating the UUID format prevents collisions and injection attacks.

---

## Recommended Default

**Default:** Server-generated UUID, override invalid client-provided IDs
**Reason:** Guarantees unique, valid UUIDs while supporting distributed tracing when upstream services provide them.

---

## Risks Of Wrong Choice

Always accepting client IDs: risk of collision, malicious ID injection, format inconsistency. Always generating server IDs: breaks distributed tracing chains where upstream IDs are needed for correlation.

## Audit Storage — Synchronous DB vs Async Logging

---

## Decision Context

Where and how should audit log entries be stored? Arises when implementing audit middleware for API requests.

---

## Decision Criteria

* throughput impact — synchronous writes blocking the response
* data durability — risk of losing audit entries on crash
* queryability — ability to search and analyze audit data
* operational complexity — infrastructure required for async solutions

---

## Decision Tree

What is your request throughput?
↓
Low (<100 req/s)?
YES → Synchronous DB insert acceptable
NO → Medium (100-1000 req/s)?
    YES → Async logging (Redis/ELK channel)
    NO → High (>1000 req/s)?
        YES → Dedicated stream processor (Kafka + ELK)
        NO → Async logging (safe default)

---

## Rationale

Synchronous DB inserts become a bottleneck under load as each request waits for write completion. Async logging via Redis or ELK channels provides non-blocking writes. At very high throughput, a stream processor like Kafka handles buffering and backpressure.

---

## Recommended Default

**Default:** Async logging via Redis/ELK log channel
**Reason:** Non-blocking writes prevent audit from becoming a performance bottleneck, while still providing queryable storage.

---

## Risks Of Wrong Choice

Synchronous DB inserts at high throughput: database bottleneck, increased response latency, potential cascading failures. Async without durability guarantees: audit entry loss on system crash.

## Middleware Organization — Single Concern per Middleware vs Consolidated

---

## Decision Context

Should each cross-cutting concern have its own middleware class, or should multiple concerns be consolidated? Arises when structuring the API middleware stack.

---

## Decision Criteria

* maintainability — ease of adding/removing individual concerns
* testability — testing each concern in isolation
* performance — number of middleware classes in the chain
* reusability — applying concerns to different route groups selectively

---

## Decision Tree

Do you need to apply concerns to different route groups?
↓
YES → Single concern per middleware (selective application)
NO → Are there more than 5 separate concerns?
    YES → Consider grouping related concerns by category
    NO → Single concern per middleware (clean separation)

Consolidate only headers: grouped headers in one SecurityHeaders middleware.
Never consolidate: logging, auth, rate limiting, body parsing.

---

## Rationale

Single-concern middleware follows the Single Responsibility Principle — easier to test, maintain, and selectively apply. Security headers are a safe consolidation point because they always apply together. Never consolidate concerns with different side effects (logging changes state, auth gates access).

---

## Recommended Default

**Default:** Single concern per middleware class
**Reason:** Easier testing, selective route group application, clear separation of responsibilities, and simpler debugging.

---

## Risks Of Wrong Choice

Consolidated middleware: hard to test individual behaviors, cannot selectively apply concerns, violates SRP, debugging is harder. Too many middleware classes: negligible performance impact but cleaner organization.
