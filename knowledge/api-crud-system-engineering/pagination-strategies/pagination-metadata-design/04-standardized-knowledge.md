# Overview

Pagination Metadata Design is the structuring of pagination information returned alongside paginated API responses — including current page, per-page count, total records, last page, navigation links, and cursor information. It exists because paginated responses are incomplete without metadata telling clients where they are in the dataset and how to navigate further.

Engineers must care because pagination metadata is the contract between the API and its consumers for navigating large datasets. Inconsistent or incomplete metadata forces clients to implement brittle pagination logic. Well-designed metadata enables generic client-side pagination handling that works across all list endpoints.

---

# Core Concepts

**Metadata Object:** A structured object containing pagination state — typically `current_page`, `per_page`, `total`, and `last_page`.

**Link Object:** Navigation URLs for first, previous, next, and last pages — either as a `links` object or HTTP `Link` headers.

**Cursor Information:** For cursor-based pagination, the next and previous cursor values plus a `has_more` indicator.

**Embedded vs Separate:** Metadata can be embedded in the response body (within `meta` key) or communicated via headers.

**JSON:API Pagination:** `jsonapi` spec defines `first`, `prev`, `next`, `last` links in a `links` object at the top level.

---

# When To Use

- Every paginated list endpoint
- Search results endpoints
- Activity feeds and timelines
- Any endpoint returning a subset of a larger dataset

---

# When NOT To Use

- Non-paginated single-resource responses
- Endpoints returning complete datasets (no pagination applied)

---

# Best Practices

**Include metadata in a consistent location.** Use a `meta` key at the response root for consistency across all endpoints.

**Include navigation links.** At minimum `first`, `prev`, `next`, `last` URLs. Enables hypermedia-driven pagination.

**Use consistent key naming.** `current_page`, `per_page`, `total`, `last_page` — follow Laravel's conventions for developer familiarity.

**Include `has_more` for cursor pagination.** A boolean indicator is simpler than checking if `next` cursor is null.

**Use Link headers for standard navigation.** HTTP `Link` headers (RFC 5988) provide standard pagination navigation.

---

# Architecture Guidelines

**Pagination metadata is generated by the paginator, not controllers.** Laravel's `LengthAwarePaginator` provides all metadata. Controllers pass the paginator to responses.

**API Resources handle metadata formatting.** `ResourceCollection` with `WithPaginationMetadata` trait formats the response.

**Metadata structure is consistent across all endpoints.** Don't add endpoint-specific fields to pagination metadata.

**Link header generation is a separate concern.** Use a `PaginationLinkHeader` helper to generate RFC 5988 Link headers.

---

# Performance Considerations

**Computing last_page requires total count.** For large datasets, this adds query overhead. Use `simplePaginate()` when last_page is not needed.

**Generating full URLs for each link adds string concatenation overhead** — negligible for typical page counts.

**Cursor-based pagination metadata is lighter** — no total count, no last_page, just cursor values.

---

# Security Considerations

**Pagination metadata must not reveal sensitive information.** Total count of "admin users" could leak security-relevant information.

**Link URLs should not expose internal parameters.** Use API-facing URLs, not internal route names or parameters.

**Page navigation beyond defined limits should not be exposed.** Don't generate links for pages beyond max_allowed.

---

# Common Mistakes

**Inconsistent metadata structure.** Some endpoints return `{ "page": 2 }`, others `{ "current_page": 2 }`. Clients must handle both.

**Missing navigation links.** Only returning metadata without URLs for next/previous pages.

**Total count in cursor pagination.** Computing total count defeats cursor pagination's performance benefit.

**Not including has_more for cursor pagination.** Clients don't know when pagination is complete.

**Incorrect link generation.** Links pointing to the wrong page or missing query parameters.

---

# Anti-Patterns

**Inconsistent Metadata Location:** Some endpoints put pagination in `meta`, others in `pagination`, others in `links`.
**Better approach:** Standardize on one location. `meta` is the most common and predictable.

**Metadata Overload:** Including 15 fields in pagination metadata when clients only need 3-4.
**Better approach:** Include only what consumers actually need. Add fields as requirements emerge.

**No Navigation Links:** Returning pagination state without URLs for navigating between pages.
**Better approach:** Always include at minimum next_page_url and prev_page_url.

---

# Examples

**Offset pagination metadata:**
```json
{
    "data": [...],
    "meta": {
        "current_page": 2,
        "from": 11,
        "to": 20,
        "per_page": 10,
        "total": 57,
        "last_page": 6
    },
    "links": {
        "first": "http://api.example.com/users?page=1",
        "prev": "http://api.example.com/users?page=1",
        "next": "http://api.example.com/users?page=3",
        "last": "http://api.example.com/users?page=6"
    }
}
```
