# Skill: Implement Pagination Metadata Design

## Purpose
Design pagination metadata in API envelope responses: offset-based with `current_page`, `per_page`, `total`, `last_page`, `from`, `to`, and cursor-based with `cursor`, `next_cursor`, `prev_cursor`, `has_more`.

## When To Use
- All list endpoints returning multiple resources
- Standardized pagination metadata across the API

## When NOT To Use
- Single-resource endpoints
- Non-paginated list endpoints (<100 records expected always)

## Prerequisites
- Laravel paginator classes (LengthAwarePaginator, CursorPaginator)
- Envelope response design

## Inputs
- Pagination strategy selection
- Response envelope specification

## Workflow
1. Use `Model::paginate()` for offset-based with `append()` in query string
2. Use `Model::cursorPaginate()` for cursor-based with `cursor` parameter
3. Include pagination metadata in envelope `meta` key — not at top-level `data`
4. For offset-based: include `current_page`, `per_page`, `total`, `last_page`, `from`, `to`
5. For cursor-based: include `cursor`, `next_cursor`, `prev_cursor`, `has_more`, `per_page`
6. Add `links` section with `first`, `last`, `prev`, `next` URLs — generated by Paginator
7. Respect `per_page` parameter with max cap — never allow unlimited per_page
8. Return 422 if `per_page` > configurable max (default 100)
9. Return 422 if `page` or `cursor` format is invalid (non-integer, malformed cursor)
10. Include pagination metadata even for empty results — `total: 0` not omitted

## Validation Checklist
- [ ] Pagination metadata in envelope `meta` key
- [ ] Offset-based includes: current_page, per_page, total, last_page, from, to
- [ ] Cursor-based includes: cursor, next_cursor, prev_cursor, has_more, per_page
- [ ] Links section with first, last, prev, next URLs
- [ ] per_page capped at configurable maximum
- [ ] 422 for per_page > max
- [ ] 422 for invalid page/cursor format
- [ ] Pagination metadata included in empty results
- [ ] Pagination strategy consistent across list endpoints
- [ ] pagination metadata field names consistent across endpoints

## Common Failures
- Pagination metadata at top-level `data` key instead of `meta` — breaks envelope structure
- Missing `links` section — client cannot navigate pagination
- `per_page` unlimited — client requests 10000 rows, server timeouts
- Empty result omitting pagination metadata — client assumes no more pages available
- Mixed pagination strategies — some endpoints use offset, others cursor, without documentation
- Cursor serialization breaks between requests — cursor format not stable

## Decision Points
- Offset vs cursor — offset for positional, cursor for real-time lists
- Max per_page — 100 for most, 500 for admin endpoints
- Default sort for pagination — `-created_at` with `id` tiebreaker

## Performance Considerations
- Offset pagination with large offset (>10000) is slow on most databases
- Cursor pagination performs consistently regardless of position
- `total` count in offset pagination adds a `COUNT` query on large tables
- Cursor pagination doesn't count total — faster but doesn't show "page 3 of 20"

## Security Considerations
- Enforce max per_page to prevent resource exhaustion attacks
- Validate page parameter to prevent negative offset or string injection
- Cursor values should be opaque — don't expose raw IDs in cursor unless intended
- Pagination metadata doesn't expose total counts for sensitive filtered queries

## Related Rules
- Include Pagination Metadata In envelope `meta` Key
- Include `links` Section With Navigation URLs
- Enforce Max `per_page` Cap — Return 422 On Exceed
- Return 422 For Invalid `page` or `cursor` Format
- Include Pagination Metadata For Empty Results
- Keep Pagination Strategy Consistent Across List Endpoints

## Related Skills
- Pagination Strategy Selection — for choosing offset vs cursor
- Query Parameter Filtering — for combined filter + pagination
- Query Parameter Sorting — for combined sort + pagination
- Envelope Response Design — for overall response structure

## Success Criteria
- All list endpoints return consistent pagination metadata in envelope
- Clients navigate pages via `links` URLs
- per_page capped prevents performance abuse
- Invalid pagination parameters return 422
- Empty results include pagination metadata for correct client handling
- Pagination strategy is consistent across all list endpoints
