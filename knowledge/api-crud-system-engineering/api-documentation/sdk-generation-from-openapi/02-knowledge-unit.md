## Executive Summary

SDK generation from OpenAPI is the process of automatically producing client libraries in multiple programming languages from an OpenAPI specification. Tools like OpenAPI Generator, Fern, Speakeasy, and openapi-generator consume the spec and produce type-safe HTTP clients with request/response models, authentication handling, and documentation.

SDK generation eliminates manual client library maintenance, ensures client-server type consistency, and accelerates consumer integration. For Laravel APIs, SDK generation is the final step in the documentation pipeline: the OpenAPI spec (generated by Scramble or Scribe) feeds the codegen tool, which produces ready-to-use client libraries for JavaScript, Python, PHP, Go, Java, .NET, and other languages.

---

## Core Concepts

### Codegen Pipeline
OpenAPI spec -> Codegen tool -> Language-specific client library

```
openapi.yaml -> OpenAPI Generator -> JavaScript SDK, Python SDK, PHP SDK, etc.
```

### Generated SDK Components
A generated SDK typically includes:
- **API client** — HTTP client with base URL, auth, request/response handling
- **Models** — Typed request and response classes matching the OpenAPI schemas
- **API methods** — Typed methods for each endpoint (createUser(), listUsers(), etc.)
- **Error types** — Exception classes for error responses
- **Documentation** — README, usage examples, inline code comments

### Codegen Tools Comparison
| Tool | Languages | Key Feature |
|---|---|---|
| OpenAPI Generator | 50+ | Largest language coverage, community-maintained |
| Fern | 10+ | TypeScript-first, excellent developer experience |
| Speakeasy | 10+ | Focus on SDK publishing and versioning |
| openapi-client (OAS) | 5+ | Simple, focused client generation |

### Schema Completeness Requirement
SDK generation requires complete, typed schemas. Missing properties, anyOf/oneOf without discriminators, and untyped `object` schemas produce unusable SDKs. Schema quality directly affects SDK quality.

## Mental Models

### Spec as Compiler Input
Think of the OpenAPI spec as source code and the codegen tool as a compiler. The spec must be syntactically valid, structurally complete, and semantically precise — just like source code — to produce a working SDK.

### Generated SDK vs Hand-Written SDK
Generated SDKs are always in sync with the spec but may lack idiomatic patterns for each language. Hand-written SDKs are more idiomatic but require ongoing maintenance. The generated SDK covers 80% of needs; hand-written adjustments cover the remaining 20%.

### SDK as Documentation Extension
A generated SDK is executable documentation. The type definitions, method signatures, and inline comments from the spec become autocomplete suggestions and compile-time checks in the consumer's IDE.

## Internal Mechanics

### OpenAPI Generator
The most widely used codegen tool:

```bash
# Generate PHP client
openapi-generator generate -i openapi.yaml -g php -o ./sdk/php

# Generate JavaScript client
openapi-generator generate -i openapi.yaml -g javascript -o ./sdk/js

# Generate TypeScript client (with axios)
openapi-generator generate -i openapi.yaml -g typescript-axios -o ./sdk/ts
```

### Speakeasy
Modern codegen with version management:

```bash
speakeasy generate -s openapi.yaml -l php -o ./sdk
speakeasy publish -t npm -p ./sdk/js
```

### Fern
TypeScript-first codegen with excellent DX:

```bash
fern generate --group sdk
```

### Schema Features Used by Codegen
Codegen tools map OpenAPI features to language constructs:

| OpenAPI Feature | Language Construct |
|---|---|
| Type `object` with properties | Class with typed fields |
| `required` array | Constructor required parameters |
| `enum` | Enum or const union type |
| `oneOf` | Union type or discriminated union |
| `$ref` | Type/class reference |
| `format: date-time` | DateTime type |
| `operationId` | Method name |
| `tags` | Namespace/module grouping |

## Patterns

### operationId-Driven Method Names
The `operationId` field becomes the SDK method name. Follow a consistent naming convention:

```yaml
operationId: users.list    # -> client.users.list()
operationId: users.create  # -> client.users.create()
operationId: users.get     # -> client.users.get(id)
```

### Consistent Schema Definitions
Define schemas in components and reference them consistently. Avoid inline schemas — codegen tools may handle them poorly or generate duplicate types.

```yaml
# Good
components:
  schemas:
    User:
      type: object
      properties:
        id: { type: integer }
paths:
  /users/{id}:
    get:
      responses:
        '200':
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/User'
```

### Discriminated Unions for oneOf
When using oneOf, include a discriminator property so codegen can generate proper union types:

```yaml
PaymentMethod:
  type: object
  oneOf:
    - $ref: '#/components/schemas/CreditCard'
    - $ref: '#/components/schemas/BankTransfer'
  discriminator:
    propertyName: type
```

### SDK Documentation
Include SDK usage examples in the OpenAPI spec's `externalDocs`:
```yaml
externalDocs:
  description: JavaScript SDK Guide
  url: https://docs.example.com/sdks/javascript
```

## Architectural Decisions

### Codegen Tool Selection
Language coverage, SDK quality, community size, and publishing support. OpenAPI Generator has the broadest coverage; Fern and Speakeasy have better DX and publishing pipelines. Decision: OpenAPI Generator for maximum language support; Speakeasy/Fern for TypeScript-first projects.

### SDK Publishing Strategy
Where and how to distribute generated SDKs:
- Public package registries (npm, Packagist, PyPI)
- Private package registries (GitHub Packages, Artifactory)
- Direct download from documentation site

### Generated SDK Versioning
SDK versions should match API versions. When API v2.1.0 is released, SDK v2.1.0 should be published. Use the OpenAPI spec version to drive the SDK version.

## Tradeoffs

| Benefit | Cost | Consequence |
|---|---|---|
| Eliminates manual SDK maintenance | Spec must be complete and precise | Invest in spec quality |
| Type safety across languages | Generated code may not be idiomatic | Accept 80% coverage; hand-tune if needed |
| Instant SDK for new API versions | Codegen pipeline adds build time | Automate in CI/CD |
| Multi-language support | Each language may need different config | Maintain per-language config files |

## Performance Considerations

### Codegen Build Time
Generating SDKs for 5 languages from a spec with 100 endpoints:
- OpenAPI Generator: 2-5 minutes
- Fern: 30-60 seconds
- Speakeasy: 1-3 minutes

### SDK Size
Generated SDKs can be large (5-50 MB per language) due to generated code and dependencies. Consider tree-shaking or modular SDK generation.

## Production Considerations

### CI/CD Pipeline for SDK Generation
Automate SDK generation and publishing:

```yaml
sdk-generate:
  runs-on: ubuntu-latest
  steps:
    - uses: actions/checkout@v4
    - name: Generate SDKs
      run: |
        openapi-generator generate -i openapi.yaml -g php -o sdk/php
        openapi-generator generate -i openapi.yaml -g typescript-axios -o sdk/ts
    - name: Publish to npm
      run: cd sdk/ts && npm publish
    - name: Publish to Packagist
      run: cd sdk/php && composer publish
```

### SDK Testing
Test generated SDKs against the actual API:
1. Generate SDK from spec
2. Run integration tests using the SDK
3. Verify all endpoints work with generated clients

### Breaking Change Notification
When a spec change produces a breaking SDK change (removed method, changed parameter), notify consumers before publishing the new SDK version.

## Common Mistakes

### Missing operationId Fields
Why it happens: operationId is optional in OpenAPI. Why it's harmful: Codegen generates method names from paths, producing inconsistent or unreadable names. Better approach: Always provide a unique, descriptive operationId.

### Inline Schemas
Why it happens: Schema is defined inline in the path instead of in components. Why it's harmful: Codegen generates duplicate types for each endpoint. Better approach: Define all schemas in components with $ref references.

### Untyped Objects
Why it happens: A property is typed as `object` without properties defined. Why it's harmful: Codegen generates a generic Map or Dictionary type with no structure. Better approach: Define all object properties explicitly.

### Missing Error Models
Why it happens: Only success responses have schemas. Why it's harmful: Generated SDKs have no typed error handling. Better approach: Define error response schemas in components.

---

## Failure Modes

### Spec-Incompatible SDK Generation
The spec uses features (certain oneOf patterns, circular $ref) that codegen cannot handle. Failure mode: SDK generation fails or produces broken code. Mitigation: Validate spec against codegen capabilities before generation.

### Generated SDK Does Not Work
The generated client has bugs — wrong URL construction, incorrect serialization, missing auth. Failure mode: Consumers cannot use the SDK. Mitigation: Integration-test generated SDKs before publishing.

### SDK Overwrites Custom Code
When regenerating, the tool overwrites manually added methods or customizations. Failure mode: Lost custom code. Mitigation: Use codegen extension points (custom templates, post-generation scripts) instead of direct modification.

## Ecosystem Usage

### Stripe SDKs
Stripe maintains hand-written SDKs, but their API is so consistent that the SDKs closely mirror what codegen would produce. Stripe's SDKs are the gold standard for client library quality.

### GitHub SDKs
GitHub uses Octokit, their hand-written SDK, but also provides generated SDKs via OpenAPI Generator for community-contributed languages.

### Twilio SDKs
Twilio generates SDKs from their API specification. They maintain SDKs for 8+ languages with consistent interfaces across all languages.
