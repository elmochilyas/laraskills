# Skill: Implement Top-Level Meta and Links

## Purpose
Include top-level `meta` and `links` keys in envelope responses for cross-resource metadata and navigation: `self`, `first`, `last`, `prev`, `next`, `related` per JSON:API specification.

## When To Use
- Envelope-style API responses
- JSON:API compliant responses
- Paginated list responses

## When NOT To Use
- Non-envelope responses (204, binary)
- Single resource responses without related resources

## Prerequisites
- Envelope response design
- Pagination metadata design

## Inputs
- Link types per endpoint
- Meta field definitions

## Workflow
1. Define top-level `links` object for navigation: `self`, `first`, `last`, `prev`, `next` for paginated
2. Include `self` link on all resource responses — URL of the current endpoint
3. Include pagination links on list responses — generated by Paginator
4. Include `related` links in relationship sections for related resources
5. Define `meta` with request-scoped metadata: `request_id`, `timestamp`, `api_version`
6. Include pagination-specific meta on list responses: page info, total
7. Generate links using `route()` helper — never hardcode URLs
8. Use absolute URLs for links — relative URLs cause client confusion
9. Keep `meta` and `links` at top-level, not inside `data`
10. Exclude `links` or `meta` from 204 responses

## Validation Checklist
- [ ] Top-level `links` with `self` on all resource responses
- [ ] Pagination links on list responses (first, last, prev, next)
- [ ] `related` links in relationship sections
- [ ] `meta` with request_id, timestamp, api_version
- [ ] Pagination metadata in `meta` on list responses
- [ ] Links generated with `route()` helper
- [ ] Absolute URLs for all links
- [ ] `meta` and `links` at top-level, not in `data`
- [ ] 204 responses exclude `meta` and `links`
- [ ] Links tested for correctness in integration tests

## Common Failures
- Relative URLs — client constructs incorrect URLs from base
- `self` link missing — client can't request the same resource again
- Pagination links without proper `appends()` — query parameters lost
- `meta` inside `data` instead of top-level — breaks envelope structure
- `meta` on 204 — empty body expected, meta confuses parsers
- Links hardcoded — environment changes break all links
- `links` object with null values — omit null links instead of including with null

## Decision Points
- Required vs optional links — `self` always required, `related` optional per resource
- Relative vs absolute URLs — absolute always for API, relative for same-origin
- Null link handling — omit null links from response vs include with null value

## Performance Considerations
- URL generation adds negligible overhead
- Pagination links include query parameters — URL length may be significant with many filters
- Absolute URLs are longer than relative — ~50 bytes extra per link
- meta fields (request_id, timestamp) add ~100 bytes per response

## Security Considerations
- Links must not include sensitive query parameters (tokens, credentials)
- Self link URL must not expose internal server names
- `meta` must not include debug or internal information
- Absolute URLs must use correct scheme (HTTPS in production)
- Links pointing to related resources must be accessible by current user

## Related Rules
- Include self Link On All Resource Responses
- Include Pagination Links On List Responses
- Use route() Helper For Link Generation
- Use Absolute URLs For Links
- Keep meta And links At Top-Level
- Exclude From 204 Responses

## Success Criteria
- All resource responses include `self` link at top-level
- Paginated responses include first, last, prev, next links
- Related resources linked via `related` in relationship sections
- Links are absolute, generated via `route()` helper
- `meta` includes consistent request-level metadata
- 204 responses have no `meta` or `links`
- Links work correctly across environments
