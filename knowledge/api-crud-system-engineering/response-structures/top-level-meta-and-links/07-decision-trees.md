## Meta Field Selection

---

### Decision Context

Determining which metadata fields to include in the top-level `meta` object based on client needs, cacheability, and security considerations.

---

### Decision Criteria

* architectural
* security
* maintainability

---

### Decision Tree

Is the metadata field required for client-server log correlation?
├── YES → Include request ID in `meta` (always — every response)
└── NO → Is the metadata field a permission/authorization indicator?
    ├── YES → Include `can.update`, `can.delete` in `meta`
    │   ├── Value comes from cached role/permission check (not queried per-request)
    │   └── Low-cost — include in every authenticated response
    └── NO → Is the metadata field a timestamp (generated_at, server_time)?
        ├── YES → Does the endpoint benefit from response caching?
        │   ├── YES → Omit from meta — dynamic timestamps prevent caching
        │   └── NO → Include — useful for client-side staleness checks
        └── NO → Does the metadata reveal internal state (feature flags, config)?
            ├── YES → Review if safe to expose — only include non-sensitive flags
            └── NO → Include if clients find it useful

---

### Rationale

Meta fields provide operational context but can prevent response caching if they change every request. Request ID is always included for correlation. Permission fields are cheap enough to include on authenticated responses. Dynamic timestamps should be omitted from cacheable endpoints.

---

### Recommended Default

**Default:** Always include `request_id`; include `can.update`/`can.delete` on authenticated responses; include `generated_at` only on non-cacheable endpoints
**Reason:** Request ID is essential for debugging; permission fields are cheap; timestamps trade off against cacheability.

---

### Risks Of Wrong Choice

Missing request ID prevents log correlation. Including timestamps on cacheable endpoints defeats caching. Permission meta that doesn't match server-side policy creates authorization confusion.

---

### Related Rules

* Always Include a Request ID in `meta`
* Keep `meta` Fields Stable and Additive

## Link Generation and Management

---

### Decision Context

Deciding which links to include in the `links` object and how to generate them without hardcoding URLs.

---

### Decision Criteria

* maintainability
* architectural

---

### Decision Tree

Is the response a single resource or a collection?
├── Single resource → Include `self` link with canonical URL
│   ├── Are there related resources (author, comments)?
│   │   ├── YES → Include `related` links in the `relationships` object
│   │   └── NO → Only `self` link needed
│   └── Is HATEOAS fully supported?
│       ├── YES → Include action links (update, delete) conditionally based on permissions
│       └── NO → Only `self` link
└── Collection → Let `PaginatedResourceResponse` handle pagination links
    ├── Is the collection paginated?
    │   ├── YES → Auto-generated `first`, `prev`, `next`, `last` links
    │   └── NO → Add a `self` link pointing to the collection URL
    └── Are custom links needed alongside pagination links?
        ├── YES → Add via `with()` method, not `additional()`
        └── NO → Default paginator links are sufficient

Are link URLs generated via `route()` helper?
├── YES → Good — named routes prevent hardcoded URLs
└── NO → Refactor to use `route()` — hardcoded URLs break on environment change

---

### Rationale

Self links provide canonical resource URLs for cache invalidation and HATEOAS navigation. Pagination links are auto-generated by `PaginatedResourceResponse` and should not be manually overridden. Named route helpers prevent hardcoded URL strings that break between environments.

---

### Recommended Default

**Default:** `self` link on every resource via `route()` helper; pagination links auto-generated; no `additional()` overrides of `links` object
**Reason:** Self links enable canonical reference; auto-generated pagination links are correct and maintainable.

---

### Risks Of Wrong Choice

Missing `self` link forces clients to construct URLs manually. Overriding pagination links via `additional()` breaks pagination navigation. Hardcoded URLs break when domain or path structure changes.

---

### Related Rules

* Always Include a `self` Link in Every Resource
* Never Override Pagination `links` via `additional()`

## Meta Injection Method: `with()` vs `additional()`

---

### Decision Context

Choosing between `with()` and `additional()` for injecting metadata into envelope responses.

---

### Decision Criteria

* reliability
* framework usage

---

### Decision Tree

Is the data resource-level metadata that belongs under the `meta` key?
├── YES → Use `with()` method on the resource class
│   ├── Is the data dynamic and request-aware?
│   │   ├── YES → Access request data inside `with()`
│   │   └── NO → Return static meta values
│   └── Can `with()` throw an exception?
│       ├── YES → Wrap in try-catch — never let meta crash the response
│       └── NO → Safe to return directly
└── NO → Is the data a response-wide override at the top level (outside meta)?
    ├── YES → Use `additional()` — but never with `data` key
    └── NO → Use `with()` — it's the correct method for meta

Is the data an override for the `links` key on a paginated response?
├── YES → Do NOT use `additional()` — it overrides auto-generated pagination links
└── NO → Safe to use `additional()` for non-link top-level keys

---

### Rationale

`with()` merges under the `meta` key — it's the correct injection point for resource metadata. `additional()` merges at the top level and can overwrite keys like `data`, `errors`, or `links`. Using `additional()` with `data` key overwrites the resource payload.

---

### Recommended Default

**Default:** Use `with()` for all resource-level metadata; use `additional()` only for intentional top-level overrides (never `data` key)
**Reason:** `with()` is specifically designed for meta content; `additional()` is for exceptional cases.

---

### Risks Of Wrong Choice

Using `additional()` for meta pollutes the top-level response structure. `additional(['data' => ...])` overwrites the resource payload silently.

---

### Related Rules

* Use `with()` for Meta, Never `additional()`
* Never Throw Exceptions Inside `with()`
