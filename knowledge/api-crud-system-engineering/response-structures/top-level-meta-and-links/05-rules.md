# top-level-meta-and-links Rules

## Rule 1: Always Include a Request ID in `meta`
---
## Category
Reliability
---
## Rule
Always include a unique request identifier in the `meta` object of every response for client-to-server log correlation.
---
## Reason
Without a request ID in the response, clients cannot correlate errors or issues to specific server-side requests. Support requests require manual timestamp matching between client reports and server logs.
---
## Bad Example
```php
public function with($request)
{
    return [
        'generated_at' => now()->toIso8601String(),
        // Missing request_id — cannot correlate client reports to logs
    ];
}
```
---
## Good Example
```php
public function with($request)
{
    return [
        'request_id' => (string) Str::uuid(),
        'generated_at' => now()->toIso8601String(),
    ];
}
```
---
## Exceptions
No common exceptions.
---
## Consequences Of Violation
Client reports "error at 14:32" require manually scanning server logs by timestamp. Correlation failure leads to unresolved issues and wasted debugging time.

## Rule 2: Use `with()` for Meta, Never `additional()`
---
## Category
Framework Usage
---
## Rule
Always return resource metadata from the `with()` method on the resource class, never from `additional()` for meta fields.
---
## Reason
`additional()` merges at the top level of the entire response and can overwrite keys like `data`, `errors`, or `links`. `with()` is specifically designed for `meta` content and merges under the `meta` key safely.
---
## Bad Example
```php
return new UserResource($user)->additional([
    'request_id' => Str::uuid(), // merges at top level — pollutes response
    'meta' => ['key' => 'value'], // works but not idiomatic
]);
```
---
## Good Example
```php
class UserResource extends JsonResource
{
    public function with($request)
    {
        return [
            'request_id' => (string) Str::uuid(),
            'generated_at' => now()->toIso8601String(),
        ];
    }
}
```
---
## Exceptions
Response-wide overrides where the developer intentionally wants to add/modify a top-level key outside `meta`.
---
## Consequences Of Violation
`additional()` keys pollute the top-level response structure. Key collision: `additional(['data' => ...])` overwrites the resource payload. Response structure becomes unpredictable.

## Rule 3: Always Include a `self` Link in Every Resource
---
## Category
Design
---
## Rule
Always include a `self` link with the canonical URL in every resource response, generated via the `route()` helper.
---
## Reason
Self links provide the canonical resource URL for cache invalidation, HATEOAS navigation, and consistent client-side URL construction. Without `self`, clients must construct URLs from base URLs and IDs.
---
## Bad Example
```php
public function toArray($request)
{
    return [
        'id' => $this->id,
        'name' => $this->name,
        // No self link — client must construct URL
    ];
}
```
---
## Good Example
```php
public function toArray($request)
{
    return [
        'id' => $this->id,
        'name' => $this->name,
        '_links' => [
            'self' => route('api.users.show', $this),
        ],
    ];
}
```
---
## Exceptions
Bare-body APIs where hypermedia is communicated through headers.
---
## Consequences Of Violation
Clients hardcode URL construction logic. Cache invalidation cannot target specific resources. API path changes require updating every client URL builder.

## Rule 4: Never Override Pagination `links` via `additional()`
---
## Category
Reliability
---
## Rule
Never set pagination `links` through `additional()` on paginated resource collections — let `PaginatedResourceResponse` generate them automatically.
---
## Reason
`PaginatedResourceResponse` auto-injects `first`, `prev`, `next`, `last` links into the response. Adding `links` via `additional()` overwrites these auto-generated URLs entirely, breaking pagination navigation.
---
## Bad Example
```php
return UserResource::collection(User::paginate())->additional([
    'links' => ['self' => '/api/users'], // overrides pagination links
]);
// Pagination navigation (first, next, last) is lost
```
---
## Good Example
```php
return UserResource::collection(User::paginate());
// Links auto-generated by PaginatedResourceResponse:
// "links": { "first": "/api/users?page=1", "next": "/api/users?page=2", ... }
```
---
## Exceptions
When custom links are needed in addition to pagination links — use the `links` key inside `with()` instead.
---
## Consequences Of Violation
Pagination navigation completely missing from response. Clients cannot navigate beyond the first page. "Load more" and "next page" functionality broken.

## Rule 5: Keep `meta` Fields Stable and Additive
---
## Category
Maintainability
---
## Rule
Always ensure meta field changes are additive within a version — never rename or remove existing `meta` keys without a version bump.
---
## Reason
Clients parse `meta` fields for operational context. Removing or renaming a meta field breaks client code that depends on it, just like removing a resource field.
---
## Bad Example
```php
// V1 meta
'meta' => ['request_id' => 'abc', 'server_time' => '2026-06-02T12:00:00Z']

// V1.1 — renamed request_id to correlation_id
'meta' => ['correlation_id' => 'abc', 'server_time' => '2026-06-02T12:00:00Z']
// Breaking — clients reading request_id now get undefined
```
---
## Good Example
```php
// V1 meta — stable
'meta' => ['request_id' => 'abc', 'server_time' => '2026-06-02T12:00:00Z']

// V1.1 — additive only, old fields still present
'meta' => [
    'request_id' => 'abc',
    'server_time' => '2026-06-02T12:00:00Z',
    'feature_flags' => ['dark_mode' => true], // new — backward-compatible
]
```
---
## Exceptions
No common exceptions.
---
## Consequences Of Violation
Client code that reads `request_id` crashes or returns undefined. Mobile apps show "undefined" in debug UIs. Log correlation breaks because the field name changed.

## Rule 6: Never Throw Exceptions Inside `with()`
---
## Category
Reliability
---
## Rule
Always wrap `with()` content in try-catch or ensure it never throws — an exception in meta generation should not crash the entire resource response.
---
## Reason
The `with()` method runs during `toResponse()`. An uncaught exception in `with()` — from a failed authorization check, missing relation, or database query — crashes the entire endpoint, even though the resource data itself serialized successfully.
---
## Bad Example
```php
public function with($request)
{
    return [
        'can_update' => $request->user()->can('update', $this->resource),
        // If user is null, calling ->can() on null throws — crashes response
    ];
}
```
---
## Good Example
```php
public function with($request)
{
    try {
        return [
            'can_update' => $request->user()?->can('update', $this->resource) ?? false,
        ];
    } catch (\Throwable $e) {
        Log::warning('Meta generation failed', ['error' => $e->getMessage()]);
        return []; // Meta omitted — response still delivered
    }
}
```
---
## Exceptions
No common exceptions.
---
## Consequences Of Violation
500 error on every response where meta generation fails. Resource data is correct and serializable but never reaches the client because a meta field crashed.

## Rule 7: Separate Dynamic Meta from Cacheable Meta
---
## Category
Scalability
---
## Rule
Always separate dynamic meta fields (request ID, server time) from stable meta fields (pagination, permissions) to avoid cache fragmentation.
---
## Reason
Dynamic meta fields that change on every request (request ID, timestamp) prevent response caching entirely. Separating them allows the core response (resource data + stable meta) to be cached while dynamic fields are added after cache fetch.
---
## Bad Example
```php
public function with($request)
{
    return [
        'request_id' => (string) Str::uuid(), // dynamic — changes every request
        'current_page' => 1,                  // stable — same for all requests to this page
        'total' => 50,                        // stable
    ];
}
// Mixing dynamic and stable prevents caching
```
---
## Good Example
```php
// Cached response contains stable meta only
// Cache key: users:page:1
$cached = [
    'data' => [...],
    'meta' => ['current_page' => 1, 'total' => 50],
];

// Middleware adds dynamic meta after cache fetch
$cached['meta']['request_id'] = (string) Str::uuid();
$cached['meta']['generated_at'] = now()->toIso8601String();
```
---
## Exceptions
Low-traffic endpoints where caching is not needed.
---
## Consequences Of Violation
Every response has a unique cache key due to request ID/timestamp. Cache hit rate is 0%. All responses are generated from scratch, defeating the purpose of caching.
