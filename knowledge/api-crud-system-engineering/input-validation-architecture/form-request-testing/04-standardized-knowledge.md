## Overview

Testing FormRequest validation rules at three levels: unit tests for rules in isolation via `Validator::make()`, integration tests for the full lifecycle via HTTP, and authorization tests for the security boundary. Proper testing ensures validation contracts are enforced and regression-free, catching contract violations before they reach production.

## Core Concepts

- **Three Testing Levels**: Unit (`Validator::make()`), Integration (HTTP), Authorization (authorize() logic).
- **Unit Testing Rules**: Fast, narrow — test rules with `Validator::make()` directly, no HTTP overhead.
- **Integration Testing**: Slower, broad — tests full lifecycle including middleware and authorization.
- **Authorization Testing**: Security boundary — tests `authorize()` method logic.
- **Custom Assertion Helpers**: `assertValidationPasses()`, `assertValidationFails()`, `assertValidationHasError()`.
- **Parameterized Tests**: `@testWith` or Pest `->with()` for edge case coverage.

## When To Use

- For every FormRequest with validation rules
- When adding new validation rules (test before implementation)
- When refactoring validation logic
- For ensuring backward compatibility when changing rules
- For documenting contract behavior through tests

## When NOT To Use

- For trivial validation (one field, one rule) in prototypes
- When validation rules auto-generated by packages (Spatie Data)
- For endpoints that will be immediately removed or replaced

## Best Practices (WHY)

- **Test both passes and fails for every rule**: Ensures validation works correctly on both sides.
- **Use `Validator::make()` for unit tests**: Fast (<1ms), no DB, no HTTP, precise failure location.
- **Use HTTP tests for key validations**: Full stack coverage including authorization and error formatting.
- **Use custom assertion helpers**: Reduces boilerplate and ensures consistent assertions.
- **Test boundary values**: Min, max, empty, null, and special characters.
- **Test `authorize()` separately**: Security boundary deserves its own test class.
- **Map every rule to at least one test**: Comprehensive coverage prevents regressions.
- **Test `validationData()` override**: Ensure input scope is correctly restricted.

## Architecture Guidelines

- Create a `ValidatesFormRequest` trait with reusable assertion helpers.
- Test rules via `Validator::make()` using the rules from the FormRequest class.
- Use HTTP tests to verify error response shape, status code, and headers.
- Use `@testWith` for exhaustive edge case coverage of each field.
- Test `prepareForValidation()` transformations separately.
- Test `failedValidation()` custom error shapes via HTTP tests.
- Run validation tests in CI before the full test suite — they catch most regressions fast.

## Performance Considerations

- Unit validation tests run in milliseconds — prefer them for most coverage.
- HTTP tests should cover only key validations — not every edge case.
- Use `RefreshDatabase` sparingly — it adds seconds per test class.
- Mock external services in HTTP tests to avoid network calls.
- Group validation tests into a fast CI stage separate from slower integration tests.

## Security Considerations

- Test authorization failure scenarios (403 responses).
- Test that validation errors don't leak sensitive data.
- Test boundary value handling for security edge cases.
- Test that `exclude_if` and `prohibited_if` work correctly.
- Test that `prepareForValidation()` doesn't introduce security vulnerabilities.
- Test that validation rules are not bypassed by manipulating input structure.

## Common Mistakes

| Mistake | Description | Cause | Consequence | Better Approach |
|---------|-------------|-------|-------------|-----------------|
| Testing only passing case | Missing validation logic errors | Assuming validation works | Production bugs from invalid data | Test both pass and fail for every rule |
| Testing only via HTTP | Too slow for comprehensive coverage | No unit tests | Slow test suite; skipped edge cases | Unit test rules; HTTP test key scenarios |
| Not testing authorize() | Security boundary untested | Forgetting | Authorization regressions undetected | Separate authorize() test class |
| Using mock data not matching production | Tests pass but validation wrong | Not using realistic data | False confidence | Use realistic data shapes and values |
| Not resetting state between tests | Shared state → flaky tests | No isolation | Intermittent CI failures | Use `#[TestWith]` or Pest `->with()` |

## Anti-Patterns

- **Testing all rules only via HTTP**: Too slow — unit test rules via `Validator::make()`.
- **No negative testing**: Only testing that valid data passes, not that invalid data fails.
- **Testing error message text exactly**: Fragile — messages change for localization.
- **Skipping authorization tests**: Security boundary is critical and often overlooked.
- **One test per FormRequest**: Single test that doesn't cover all rules.

## Examples

```php
trait ValidatesFormRequest
{
    protected function assertValidationPasses(array $data, array $rules): void
    {
        $validator = Validator::make($data, $rules);
        $this->assertTrue($validator->passes());
    }

    protected function assertValidationFails(array $data, array $rules): void
    {
        $validator = Validator::make($data, $rules);
        $this->assertTrue($validator->fails());
    }
}

class StorePostRequestValidationTest extends TestCase
{
    use ValidatesFormRequest;

    /** @testWith ["title", ""] */
    /** @testWith ["title", null] */
    /** @testWith ["title", str_repeat("a", 256)] */
    public function test_title_validation(string $field, mixed $value): void
    {
        $this->assertValidationFails(
            array_merge($this->validData, [$field => $value]),
            (new StorePostRequest())->rules(),
        );
    }
}
```
