## Rules for OWASP LLM Compliance

### R1: Conduct OWASP LLM-specific security reviews at every major dependency update
- **Category:** Security
- **Rule:** Schedule an OWASP LLM Top 10-focused security review whenever the LLM provider SDK, gateway version, or model version is updated; never update AI dependencies without a security review.
- **Reason:** New SDK/model versions can change security-relevant behavior: different tool-calling format, new prompt handling, or altered safety defaults. An SDK update can silently remove your existing protections.
- **Bad Example:** Updating the OpenAI SDK from v1 to v2 — the tool-calling format changed, and the existing argument validation middleware no longer intercepts tool calls correctly, remaining undetected for weeks.
- **Good Example:** Before upgrading the SDK, a checklist review: "Tool call format unchanged? Response structure unchanged? Injection patterns still captured?"
- **Exceptions:** Security patch updates that explicitly fix vulnerabilities.
- **Consequences of Violation:** Existing security mitigations silently broken by dependency updates; vulnerable window between update and detection of broken protections.

### R2: Implement supply chain security for AI dependencies via pinned checksum-verified downloads
- **Category:** Security
- **Rule:** Pin all AI-related dependencies (provider SDKs, gateway packages, guard libraries) to exact versions with checksum verification in composer.lock; verify checksums before deployment; never use version ranges or unverified packages.
- **Reason:** Supply chain attacks on AI packages are increasing (LLM05). A compromised SDK package can exfiltrate prompts and responses. Checksum verification ensures the deployed package is the exact reviewed version.
- **Bad Example:** A composer.json requiring `"openai-php/client": "^1.0"` — a malicious v1.5.0 release is automatically installed by CI with no checksum verification.
- **Good Example:** Exact version pinned: `"openai-php/client": "1.4.0"`, composer.lock checksum committed and verified via `composer audit` in CI.
- **Exceptions:** First-party or air-gapped packages with verified provenance.
- **Consequences of Violation:** Malicious code injected via compromised dependency; exfiltration of all prompts, responses, and API keys to an attacker-controlled server.

### R3: Implement user-facing disclaimers for AI-generated content in regulated domains (finance, medical, legal)
- **Category:** Safety
- **Rule:** Display a clear disclaimer ("This content was generated by AI and may contain errors. Verify with a qualified professional.") alongside every AI response in regulated domains; never present AI output as authoritative.
- **Reason:** LLMs hallucinate with high confidence. In regulated domains, over-reliance on AI outputs (LLM08) can lead to real-world harm, regulatory fines, or legal liability. Disclaimers reduce liability and manage expectations.
- **Bad Example:** A financial advisory AI presenting stock recommendations without any disclaimer — a user loses money following a hallucinated recommendation and sues.
- **Good Example:** Every response in the financial advice agent is preceded by: "⚠ AI-generated financial information. Consult a licensed financial advisor before making investment decisions."
- **Exceptions:** Entertainment or educational content where no actionable decisions are based on the output.
- **Consequences of Violation:** Legal liability for damages caused by reliance on erroneous AI outputs; regulatory penalties for presenting AI-generated professional advice as authoritative.
