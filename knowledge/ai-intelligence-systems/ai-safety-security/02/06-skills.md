# Skill: Implement Content Moderation and Safety Filtering

## Purpose
Deploy multi-layer content moderation for AI inputs and outputs — combining fast rule-based filters with ML-based classifiers to detect and block harmful content (hate, harassment, violence, self-harm, sexual) with configurable policies per feature and user segment.

## When To Use
- Any user-facing AI application (chatbots, content generation, image generation)
- Applications in regulated industries (healthcare, finance, education, social media)
- Applications serving minors or vulnerable populations
- Public-facing systems where brand safety is a concern

## When NOT To Use
- Internal tools used by trusted employees with clear usage policies
- Applications that already have a human review layer for all content
- When the moderation system introduces unacceptable latency (consider async moderation)

## Prerequisites
- KU-02 (Content Moderation & Safety Filtering) — understanding of harm categories and moderation approaches
- Moderation service or classifier model (OpenAI Moderation API, Azure Content Safety, custom classifier)
- Rule-based filter patterns for known unacceptable content
- Policy configuration defining harm categories → actions

## Inputs
- User input text to check before LLM processing
- LLM output text to check before user delivery
- Moderation policy (harm categories, actions per category, confidence thresholds)
- User segment metadata (age group, feature context, region)
- Rule-based filter patterns (keywords, regex)

## Workflow
1. **Define harm categories**: Identify relevant categories for your application: hate, harassment, violence, self-harm, sexual, and domain-specific (medical advice, financial recommendations). Map each to an action (block, flag, warn, allow).
2. **Configure rule-based filter**: Create a pattern list for obvious harmful content. Use keyword matching and regex for high-precision patterns. This is the first line of defense — fast (<0.5ms), catches common cases.
3. **Configure ML classifier**: Set up a moderation API or classifier model. Define confidence thresholds per category. For example: hate >0.9 → block, hate 0.7-0.9 → flag, hate <0.7 → allow with logging.
4. **Apply input moderation**: Before sending user content to the LLM, run it through the rule filter first (fast rejection), then the ML classifier (if rule filter passes). Block or flag based on policy.
5. **Apply output moderation**: After receiving the LLM response, run it through the same moderation pipeline. Output moderation catches harmful content generated by the model even with benign inputs.
6. **Handle streaming output moderation**: For streaming responses, apply per-chunk moderation. Buffer minimally (2-3 chunks) for context-aware detection. Block the stream if harmful content is detected.
7. **Implement graduated responses**: For blocked content, return a clear user-facing message explaining why. For flagged content, log and possibly escalate for review. For low-confidence matches, allow with logging.
8. **Log moderation decisions**: Log every moderation action with: timestamp, content (redacted), category, confidence score, action taken, and whether it was a false positive (if user appeals).
9. **Monitor and tune**: Track false positive rate (content blocked but legitimate) and false negative rate (harmful content that passed). Tune thresholds based on observed performance. Review weekly.
10. **Provide appeal mechanism**: Allow users to appeal moderation decisions. Review appeals to identify false positives and improve detection.

## Validation Checklist
- [ ] Both input and output moderation are implemented
- [ ] Multiple moderation layers exist (rule-based + ML classifier)
- [ ] Harm categories are clearly defined and mapped to actions
- [ ] Moderation decisions are logged with content, category, confidence, and action taken
- [ ] Users receive clear feedback when content is blocked
- [ ] False positive rate is monitored and below the acceptable threshold
- [ ] Moderation policies are configurable per feature and user segment

## Common Failures
- **Input-only moderation**: Users can still see harmful LLM outputs. Fix: implement output moderation too.
- **Single keyword filter**: High false positive rate blocks legitimate content. Fix: use layered approach (rules + ML + policy).
- **No appeal mechanism**: Users frustrated by false blocks with no recourse. Fix: provide feedback channel and review process.
- **Same policy for all segments**: Children and adults need different moderation thresholds. Fix: make policies configurable per segment.
- **Silent blocking**: Content blocked with no explanation. Fix: return clear, helpful error messages.

## Decision Points
- **Rule-based vs. ML classifier**: Rules for high-precision, known patterns (no ambiguity). ML for nuanced content (sarcasm, context-dependent). Use both in sequence.
- **Real-time vs. async moderation**: Real-time for user-facing chat (block before display). Async for batch processing (review after generation). Real-time is mandatory for public-facing apps.
- **Per-category thresholds**: Lower threshold (more aggressive) for hate speech and self-harm. Higher threshold (more permissive) for violence in educational or news contexts.

## Performance Considerations
- Rule-based filters: <0.5ms per check
- ML classifier: 10-100ms per check (model-dependent)
- Moderation API (OpenAI, Azure): 200-500ms per call
- Cache moderation results: 30-50% hit rate for repeated user inputs
- Streaming per-chunk moderation: sub-millisecond per chunk with simple rules
- For high throughput, use a dedicated moderation service that scales independently

## Security Considerations
- Attackers may encode harmful content (base64, leetspeak, emoji substitution) to evade filters
- Ensure moderation provider's data handling meets your compliance requirements
- If using a custom classifier, ensure training data is clean and model is regularly evaluated
- Aggressive moderation that blocks legitimate content damages user trust — tune carefully
- Moderation decisions should be auditable for compliance

## Related Rules
- Never pass raw LLM output to the user — always validate through an output guard
- Always implement a "refusal detection" guard that catches model refusals disguised as responses
- Implement factual consistency checking against the knowledge base for domain-specific outputs

## Success Criteria
- Input moderation blocks >90% of harmful user content before reaching LLM
- Output moderation catches >90% of harmful LLM-generated content
- False positive rate <5% (content blocked that should have been allowed)
- False positive appeals reviewed within 24 hours
- Moderation policies are configurable per feature and user segment
- Moderation decisions are logged with full audit trail
- Streaming output moderation prevents harmful content from reaching users in real-time
