## Rules for Output Guarding & Validation

### R1: Never pass raw LLM output to the user — always validate through an output guard
- **Category:** Safety
- **Rule:** Every LLM response must pass through an output guard that checks for harmful content (toxicity, PII, hallucinated facts, off-topic replies) before being presented to the user; never trust raw LLM output.
- **Reason:** LLMs can produce harmful, unsafe, or incorrect content even when the input is benign. Output guards are the last line of defense before the user sees potentially damaging content.
- **Bad Example:** A chat application that renders LLM output directly in the UI via `{!! $response->text !!}` without any filtering.
- **Good Example:** An `OutputGuard` pipeline that checks for toxicity, PII leakage, and factual consistency against the knowledge base before rendering.
- **Exceptions:** Read-only display of LLM output with a visible disclaimer that content is unverified.
- **Consequences of Violation:** Users may see offensive, dangerous, or factually incorrect content generated by the model; brand damage, legal liability, and user safety risks.

### R2: Always implement a "refusal detection" guard that catches model refusals disguised as responses
- **Category:** Reliability
- **Rule:** Check LLM responses for refusal patterns ("I cannot", "I'm sorry, but", "As an AI") even when the response format is structured; when a refusal is detected, return a consistent application-level error.
- **Reason:** Models can refuse requests even for legitimate use cases (content policy false positives). Without detection, refusal text in a response field is treated as valid data, confusing users or downstream systems.
- **Bad Example:** A summarization tool receives "I cannot summarize this content as it may contain harmful material" as the output, which is then displayed as the summary.
- **Good Example:** An `OutputGuard` detects refusal patterns and returns a consistent message: "This request could not be processed. Please rephrase or contact support."
- **Exceptions:** When showing refusal messages provides useful feedback to the user.
- **Consequences of Violation:** Users see confusing error messages or refusal text embedded in otherwise normal-looking responses; downstream automated systems may process refusal text as valid data.

### R3: Implement factual consistency checking against the knowledge base for domain-specific outputs
- **Category:** Reliability
- **Rule:** For domain-specific answers (product info, company policies, medical/legal), validate LLM output against a trusted knowledge base using semantic similarity; never serve hallucinated facts.
- **Reason:** LLMs hallucinate — they confidently assert false claims. For domain-specific applications, uncorroborated hallucinations can lead to wrong decisions, financial loss, or safety issues.
- **Bad Example:** A customer support agent that generates a refund policy answer without verifying it against the actual policy document — it invents a policy that doesn't exist.
- **Good Example:** An `OutputFactChecker` that extracts claims from the LLM output and checks each against the RAG knowledge base using semantic similarity; outputs below 0.85 similarity score are flagged.
- **Exceptions:** Creative or brainstorming applications where factual accuracy is not required.
- **Consequences of Violation:** Users receive incorrect information presented with high confidence; in medical, legal, or financial contexts, hallucinated facts can cause real-world harm.
