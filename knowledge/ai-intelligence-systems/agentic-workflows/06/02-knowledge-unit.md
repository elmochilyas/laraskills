## Executive Summary

Agent memory refers to how an agent persists and retrieves information beyond the current context window. While the LLM's context window serves as working memory (limited to ~4K-200K tokens depending on model), long-running agents need persistent storage for facts, conversation history, learned preferences, and task progress. State management covers the lifecycle of agent state â€” creation, mutation, persistence, and cleanup. In the Laravel ecosystem, memory backends are implemented as Eloquent models, Redis stores, or vector databases.

## Core Concepts

- **Ephemeral Memory (Context Window):** The current conversation history. Lost when the session ends. Limited by model context length.
- **Conversation Memory:** Persisted message history that spans multiple sessions. Stored in a database or cache, loaded into context on each turn.
- **Semantic Memory:** Vector-embedded facts and knowledge about the user, domain, or past interactions. Retrieved via similarity search.
- **Working Memory:** Temporary state during a single task execution (tool results, partial plans, intermediate calculations).
- **Memory Consolidation:** The process of extracting salient information from working memory and storing it in long-term memory.
- **Memory Retrieval:** Querying long-term memory to inject relevant context into the current window (e.g., "user prefers concise answers").
- **State Machine:** The agent's status lifecycle (idle â†’ processing â†’ waiting_for_tool â†’ completed â†’ failed). Transitions triggered by events.

## Internal Mechanics

The internal mechanics of Agent Memory & State follow established patterns within the Agentic Workflows domain. The implementation leverages the Laravel AI SDK conventions and ecosystem best practices.

## Architectural Decisions

- Implement memory backends using the **repository pattern**: `MemoryRepository` interface with implementations for `DatabaseMemory`, `RedisMemory`, `VectorMemory`.
- Use **Redis** for ephemeral workspaces and **database** for persistent conversation history.
- For semantic memory, use a **vector database** (pgvector, Qdrant) with embeddings generated by the LLM provider or a dedicated embedding model.
- Design memory as **key-value stores** where the key is `agent_id:session_id:memory_type` and the value is the serialized memory object.
- Implement a **memory manager** that orchestrates read/write operations across backends and handles TTL/eviction.

## Tradeoffs

Standard approach vs Custom implementation is the primary tradeoff in this KU. Standard implementations offer faster development and community support but may have overhead. Custom implementations provide tailored solutions at the cost of maintenance burden.

## Performance Considerations

- Memory retrieval latency varies by backend: Redis <1ms, Database 1-10ms, Vector DB 5-50ms.
- Loading too much memory into context degrades LLM attention. Limit retrieved memories to the top-N most relevant (typically 5-10 items).
- Write-behind caching: write to memory asynchronously (queue) after the agent responds, not during the agent loop.
- Consolidation overhead: running summarization on every turn is expensive. Batch consolidate every N turns or when context is >80% full.
- Memory fragmentation: vector search quality degrades as the memory store grows. Implement periodic re-indexing.

## Production Considerations

- **Memory isolation:** Agent A should never read Agent B's memory. Enforce strict namespace isolation (agent_id/session_id).
- **Data encryption:** Persisted memory may contain PII or sensitive business data. Encrypt at rest.
- **Memory poisoning:** If user input can influence what gets stored, an attacker could inject false memories. Validate and sanitize all memory writes.
- **Right to deletion:** GDPR/CCPA requires the ability to delete user memories. Implement a `deleteByUser()` method.
- **Audit trail:** All memory writes should be logged (what was written, by whom, when).

## Common Mistakes

- Persisting the entire raw message history without summarization â€” leads to unbounded storage growth and context bloat.
- Not implementing memory eviction â€” the store grows forever, increasing cost and search latency.
- Storing sensitive data in memory without encryption â€” every persisted conversation becomes a liability.
- Using memory retrieval without relevance filtering â€” injecting irrelevant memories confuses the agent.
- Hardcoding memory backend â€” makes it impossible to swap from database to Redis or vector store.

## Failure Modes

- **Firehose Memory:** Injecting all available memory into every agent turn. Kills attention and increases cost. Use selective retrieval.
- **Mutable Memory Without Versioning:** Writing over old memories without retaining history. Prefer append-only logs with snapshot reads.
- **Orphaned Sessions:** Agent sessions that never expire. Implement TTL and garbage collection on the memory store.
- **Cross-Context Leakage:** Sharing a single memory namespace across all users. Every user must have an isolated memory partition.

## Ecosystem Usage

### Memory Repository Interface
```php
interface MemoryRepository {
    public function read(string $agentId, string $sessionId, MemoryType $type): ?Memory;
    public function write(string $agentId, string $sessionId, MemoryType $type, array $data, int $ttl = null): void;
    public function search(string $agentId, string $query, int $limit = 5): array;
    public function delete(string $agentId, string $sessionId): void;
}
```
