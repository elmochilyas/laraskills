# Agent Memory & State

## Overview

Agent memory refers to how an agent persists and retrieves information beyond the current context window. While the LLM's context window serves as working memory (limited to ~4K-200K tokens depending on model), long-running agents need persistent storage for facts, conversation history, learned preferences, and task progress. State management covers the lifecycle of agent state — creation, mutation, persistence, and cleanup. In the Laravel ecosystem, memory backends are implemented as Eloquent models, Redis stores, or vector databases.

## Core Concepts

- **Ephemeral Memory (Context Window):** The current conversation history. Lost when the session ends. Limited by model context length.
- **Conversation Memory:** Persisted message history that spans multiple sessions. Stored in a database or cache, loaded into context on each turn.
- **Semantic Memory:** Vector-embedded facts and knowledge about the user, domain, or past interactions. Retrieved via similarity search.
- **Working Memory:** Temporary state during a single task execution (tool results, partial plans, intermediate calculations).
- **Memory Consolidation:** The process of extracting salient information from working memory and storing it in long-term memory.
- **Memory Retrieval:** Querying long-term memory to inject relevant context into the current window (e.g., "user prefers concise answers").
- **State Machine:** The agent's status lifecycle (idle → processing → waiting_for_tool → completed → failed). Transitions triggered by events.

## When To Use

- Agents that interact with users across multiple sessions (support bots, personal assistants).
- Long-running agent workflows that exceed the model's context window (batch processing, research agents).
- Systems needing to remember user preferences, past decisions, or learned patterns.
- Multi-agent systems where shared state must be accessible across agent boundaries.

## When NOT To Use

- Single-turn, stateless interactions (no memory overhead needed).
- Short workflows that fit comfortably within the context window.
- Prototypes where memory requirements are unknown — start stateless, add memory when evidence supports it.
- Systems with strict data privacy requirements that forbid persisting conversation history.

## Best Practices

- **Explicitly separate ephemeral from persistent memory.** Don't persist everything — define what matters (decisions, preferences, results).
- **Use summarization for memory consolidation.** When the context window is full, summarize older messages into a condensed "memory chunk".
- **Implement a memory TTL.** Not all information is relevant forever. Set expiry on conversation memories (e.g., 30-day retention).
- **Store memory as structured data, not raw text.** Extract facts (JSON), not verbatim transcripts.
- **Version memory schemas** — agents evolve, and stored state must evolve with them. Include a schema version field.

## Architecture Guidelines

- Implement memory backends using the **repository pattern**: `MemoryRepository` interface with implementations for `DatabaseMemory`, `RedisMemory`, `VectorMemory`.
- Use **Redis** for ephemeral workspaces and **database** for persistent conversation history.
- For semantic memory, use a **vector database** (pgvector, Qdrant) with embeddings generated by the LLM provider or a dedicated embedding model.
- Design memory as **key-value stores** where the key is `agent_id:session_id:memory_type` and the value is the serialized memory object.
- Implement a **memory manager** that orchestrates read/write operations across backends and handles TTL/eviction.

## Performance Considerations

- Memory retrieval latency varies by backend: Redis <1ms, Database 1-10ms, Vector DB 5-50ms.
- Loading too much memory into context degrades LLM attention. Limit retrieved memories to the top-N most relevant (typically 5-10 items).
- Write-behind caching: write to memory asynchronously (queue) after the agent responds, not during the agent loop.
- Consolidation overhead: running summarization on every turn is expensive. Batch consolidate every N turns or when context is >80% full.
- Memory fragmentation: vector search quality degrades as the memory store grows. Implement periodic re-indexing.

## Security Considerations

- **Memory isolation:** Agent A should never read Agent B's memory. Enforce strict namespace isolation (agent_id/session_id).
- **Data encryption:** Persisted memory may contain PII or sensitive business data. Encrypt at rest.
- **Memory poisoning:** If user input can influence what gets stored, an attacker could inject false memories. Validate and sanitize all memory writes.
- **Right to deletion:** GDPR/CCPA requires the ability to delete user memories. Implement a `deleteByUser()` method.
- **Audit trail:** All memory writes should be logged (what was written, by whom, when).

## Common Mistakes

- Persisting the entire raw message history without summarization — leads to unbounded storage growth and context bloat.
- Not implementing memory eviction — the store grows forever, increasing cost and search latency.
- Storing sensitive data in memory without encryption — every persisted conversation becomes a liability.
- Using memory retrieval without relevance filtering — injecting irrelevant memories confuses the agent.
- Hardcoding memory backend — makes it impossible to swap from database to Redis or vector store.

## Anti-Patterns

- **Firehose Memory:** Injecting all available memory into every agent turn. Kills attention and increases cost. Use selective retrieval.
- **Mutable Memory Without Versioning:** Writing over old memories without retaining history. Prefer append-only logs with snapshot reads.
- **Orphaned Sessions:** Agent sessions that never expire. Implement TTL and garbage collection on the memory store.
- **Cross-Context Leakage:** Sharing a single memory namespace across all users. Every user must have an isolated memory partition.

## Examples

### Memory Repository Interface
```php
interface MemoryRepository {
    public function read(string $agentId, string $sessionId, MemoryType $type): ?Memory;
    public function write(string $agentId, string $sessionId, MemoryType $type, array $data, int $ttl = null): void;
    public function search(string $agentId, string $query, int $limit = 5): array;
    public function delete(string $agentId, string $sessionId): void;
}
```

### Memory Types
```php
enum MemoryType: string {
    case Conversation = 'conversation';
    case Semantic = 'semantic';
    case Preference = 'preference';
    case Working = 'working';
    case Summary = 'summary';
}
```

## AI Agent Notes

- When asked to implement agent memory, first clarify: what needs to be remembered, for how long, and who needs access?
- For memory-related bugs, check: TTL expiration, memory namespace isolation, and consolidation frequency.
- Prefer reading the memory repository interface before the specific backend implementation.
- When investigating context window overflow, look at the memory retrieval logic — it may be injecting too much data.

## Verification

- [ ] Memory backends are interchangeable via a repository interface.
- [ ] Agent memory has a defined TTL and eviction policy.
- [ ] Memory retrieval returns top-N relevant items (not everything).
- [ ] Memory consolidation (summarization) is triggered when context is near capacity.
- [ ] Memory writes are validated and sanitized for injection attacks.
- [ ] User data can be deleted on request with memory isolation guarantees.
- [ ] Memory schemas include a version field for migration support.
