## Executive Summary

Laravel executes migrations in lexicographic filename order. The `YYYY_MM_DD_HHmmss` prefix enforces deterministic sequencing. Naming conventions must encode the migration's intent — creation, alteration, index addition, or data change — and respect foreign key dependency ordering. A migration that creates a referenced table must always precede the migration that adds the foreign key. Failure to manage ordering produces broken deployments and CI failures.

---

## Core Concepts

- **Timestamp prefix**: `2026_06_02_123456_create_posts_table.php` — generated by `date('Y_m_d_His')` when running `php artisan make:migration`.
- **Lexicographic sort**: Laravel sorts migration files by full filename string using PHP's `sort()`. `2026_01_01_000000` comes before `2026_01_01_000001`.
- **Batch tracking**: The `migrations` table stores the batch number per migration file. A single `php artisan migrate` command assigns the same batch number to all executed migrations.
- **FK ordering dependency**: If `create_books_table` references `authors` via foreign key, `create_authors_table` must have an earlier timestamp.
- **Name readability**: The portion after the timestamp communicates the migration's purpose. Following team conventions improves maintainability at scale.

```
Filename Component Breakdown:
  2026_06_02_123456         create_posts_table          .php
  └── timestamp ───────────┘ └── descriptive name ──┘ └── extension

Timestamp provides ordering.
Descriptive name provides intent.
```

## Mental Models

Migration ordering is a directed acyclic graph (DAG) of schema dependencies. Each migration is a node. A foreign key reference is a directed edge from the migration that creates the FK to the migration that creates the referenced table. The sort order must respect these edges. The migrations table is the execution log — once a filename is recorded, replaying that filename produces no action (idempotent playback).

---

## Internal Mechanics

`Illuminate\Database\Migrations\Migrator::resolve()` reads all files from configured migration paths, sorts by full filename, and compares against the `migrations` table to determine which are pending. The sort is a simple string comparison — no parsing of timestamps as dates. This means `2026_01_01_000000_create_a.php` sorts before `2026_01_01_000000_create_b.php` because '...a' < '...b'.

The `migrations` table schema:
```
id  | migration                              | batch
1   | 2026_01_01_000000_create_authors_table  | 1
2   | 2026_01_01_000001_create_books_table    | 1
3   | 2026_06_02_123456_add_published_at      | 2
```

When `php artisan migrate` runs, it:
1. Gets all migration files from `database/migrations/`
2. Sorts by filename
3. Queries `migrations` table for executed filenames
4. Filters to pending migrations
5. Executes `up()` on each pending migration in sorted order
6. Records each filename in the `migrations` table with the current batch number

## Patterns

**Dependency-first naming**: When new tables reference existing tables via FK, ensure the referenced table's migration has an earlier timestamp. Use manual timestamp adjustment when `make:migration` produces an incorrect order.

```bash
# Correct: referenced table first
2026_06_02_000000_create_authors_table.php   # authors table
2026_06_02_000001_create_books_table.php     # FK to authors

# Wrong ordering (will fail):
2026_06_02_000001_create_books_table.php
2026_06_02_000000_create_authors_table.php
```

**Semantic verb prefix convention**: Standardize prefixes so that migrations are self-documenting and group logically when sorted.

```php
// Naming conventions by operation type:
create_products_table              // NEW table
add_category_id_to_products_table  // NEW column
add_foreign_key_to_products_table  // NEW constraint
add_index_to_products_table        // NEW index
change_price_column_on_products    // ALTER column
rename_status_to_state_on_products // RENAME column
drop_obsolete_column_from_products // REMOVE column
```

**Timestamp gap strategy**: Create migrations with intentional time gaps (e.g., 10 seconds apart) to insert future migrations in between without reordering existing files.

```bash
# Create with gaps:
php artisan make:migration create_authors_table   # 2026_06_02_000010
php artisan make:migration create_books_table     # 2026_06_02_000020

# Later, you can insert between them:
# 2026_06_02_000015 create_publishers_table
```

## Architectural Decisions

| Practice | When | When Not |
|----------|------|----------|
| Verb prefix naming | All teams > 1 developer | Solo project with few migrations |
| Gap strategy | Anticipating intermediate migrations | Fixed schema, rapid iteration |
| Manual timestamp | FK ordering, hotfix migration ordering | Default `make:migration` ordering works |
| One batch per deploy | Granular rollback capability | Large deploys with 50+ migrations |

---

## Tradeoffs

| Benefit | Cost |
|---------|------|
| Deterministic ordering prevents FK failures | Manual timestamp management for complex dependencies |
| Self-documenting names improve team communication | Longer filenames |
| Batch grouping simplifies rollback | All migrations in a batch roll back together |
| Renaming never needed if conventions are followed | Cannot reorder deployed migrations |

## Performance Considerations

Not applicable to migration ordering directly. However, migration execution time matters for deployment duration. Ordering migrations so that slow operations (data backfills, large table alterations) run in separate batches allows other migrations to proceed independently.

---

## Production Considerations

- **Never rename a deployed migration**: The `migrations` table records the exact filename. Renaming creates a mismatch — the old name persists in the database and the new name is treated as a new migration, potentially executing the same changes again.
- **Rollback order**: `migrate:rollback` executes `down()` in reverse order of the last batch. If migration A creates authors and migration B creates books with FK, `down()` of B must run before `down()` of A.
- **Multi-server deploys**: Without the `isolated` option, two servers running `php artisan migrate` simultaneously can race. Both may try to execute the same migration. Use `isolated` (cache lock) to coordinate.
- **Hotfix ordering**: A hotfix migration must sort after all deployed migrations. Use the current timestamp: `php artisan make:migration` produces the correct ordering automatically.

## Common Mistakes

**Same-second timestamp collision**: Two developers run `make:migration` at the same second, producing identical timestamps. The PHP sort by full string is deterministic but arbitrary between the two files. Run `php artisan migrate:status` to verify the intended order.

```bash
# Bad: two files with same second
2026_06_02_123456_create_authors_table.php
2026_06_02_123456_create_books_table.php  # collision!

# Good: stagger by a second
2026_06_02_123456_create_authors_table.php
2026_06_02_123457_create_books_table.php
```

**Ambiguous migration names**: `2026_06_02_123456_update_users.php` — what is being updated? A column? A table? Data? Without a clear prefix convention, developers must open the file to understand its purpose.

**Circular FK dependency**: Table A references B, B references A. Neither migration can run first. Use a nullable FK on the first table, create the second table, then add the FK to the first table in a third migration.

```php
// Migration 1: create_authors_table with nullable editor_id
Schema::create('authors', function (Blueprint $table) {
    $table->id();
    $table->unsignedBigInteger('editor_id')->nullable();
    // No FK here yet
});

// Migration 2: create_books_table with FK to authors
Schema::create('books', function (Blueprint $table) {
    $table->id();
    $table->foreignId('author_id')->constrained();
});

// Migration 3: add_editor_foreign_key_to_authors
Schema::table('authors', function (Blueprint $table) {
    $table->foreign('editor_id')->references('id')->on('authors');
});
```

## Failure Modes

- **FK constraint failure**: `General error: 1215 Cannot add foreign key constraint` — the referenced table doesn't exist at migration time. Fix: reorder migrations so the referenced table comes first.
- **Migration skipped in batch collision**: Two servers run the same migration simultaneously. Without `isolated`, one succeeds, the other fails with a duplicate key error on the `migrations` table. Fix: use the `isolated` option: `php artisan migrate --isolated`.
- **Rollback failure from naming collision**: Two migrations named `2026_06_02_123456_create_something.php` from different developers. The first execution creates class `CreateSomething` in the first file. The second execution fails because the class is already defined (PHP fatal error). Fix: anonymous classes (Laravel 9+) or unique class names.

## Ecosystem Usage

Laravel Forge and Envoyer run `php artisan migrate` as part of deployment scripts, relying on correct ordering. Packages like `stancl/tenancy` fan out migrations to tenant databases preserving the original filename order. Laravel Shift analyzes migration naming during upgrades.

## Research Notes

Laravel 9+ introduced anonymous class migrations, which eliminated the class name collision problem but did not change the ordering mechanism — ordering is still purely filename-based. The `--isolated` option (Laravel 10+) added cache-lock-based coordination to prevent concurrent migration execution. Future versions may introduce migration dependency declarations similar to Django's `dependencies = [...]` pattern in migration classes, which would decouple ordering from filenames and allow more explicit dependency management.
