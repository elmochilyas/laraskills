## Overview

Laravel executes migrations in lexicographic filename order. The `YYYY_MM_DD_HHmmss` prefix enforces deterministic sequencing. Naming conventions must encode the migration's intent — creation, alteration, index addition, or data change — and respect foreign key dependency ordering. A migration that creates a referenced table must always precede the migration that adds the foreign key. Failure to manage ordering produces broken deployments and CI failures.

---

## Core Concepts

- **Timestamp prefix**: `2026_06_02_123456_create_posts_table.php` — generated by `date('Y_m_d_His')` when running `php artisan make:migration`.
- **Lexicographic sort**: Laravel sorts migration files by full filename string using PHP's `sort()`. `2026_01_01_000000` comes before `2026_01_01_000001`.
- **Batch tracking**: The `migrations` table stores the batch number per migration file. A single `php artisan migrate` command assigns the same batch number to all executed migrations.
- **FK ordering dependency**: If `create_books_table` references `authors` via foreign key, `create_authors_table` must have an earlier timestamp.
- **Name readability**: The portion after the timestamp communicates the migration's purpose. Following team conventions improves maintainability at scale.
- ```
- Filename Component Breakdown:
- 2026_06_02_123456         create_posts_table          .php
- └── timestamp ───────────┘ └── descriptive name ──┘ └── extension
- Timestamp provides ordering.
- Descriptive name provides intent.

## Performance Considerations

- Not applicable to migration ordering directly. However, migration execution time matters for deployment duration. Ordering migrations so that slow operations (data backfills, large table alterations) run in separate batches allows other migrations to proceed independently.

## Anti-Patterns

- - **FK constraint failure**: `General error: 1215 Cannot add foreign key constraint` — the referenced table doesn't exist at migration time. Fix: reorder migrations so the referenced table comes first.
- - **Migration skipped in batch collision**: Two servers run the same migration simultaneously. Without `isolated`, one succeeds, the other fails with a duplicate key error on the `migrations` table. Fix: use the `isolated` option: `php artisan migrate --isolated`.
- - **Rollback failure from naming collision**: Two migrations named `2026_06_02_123456_create_something.php` from different developers. The first execution creates class `CreateSomething` in the first file. The second execution fails because the class is already defined (PHP fatal error). Fix: anonymous classes (Laravel 9+) or unique class names.
