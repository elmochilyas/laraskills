## Rule Name

Only Blacklist Files That Break with Caching

## Category

Maintainability

## Rule

Never add files to the OpCache blacklist unless caching causes demonstrably incorrect behavior.

## Reason

Blacklisted files are recompiled on every request — 50–75% slower performance for those files. The blacklist is not a performance optimization tool; it is a last resort for files that genuinely cannot be cached.

## Bad Example

```bash
# Blacklisted 50 files for "performance debugging" — all are now 75% slower
```

## Good Example

```bash
# Only blacklisted the one file that uses __FILE__ in a way that breaks with caching
```

## Exceptions

No common exceptions. Fix the underlying code rather than blacklisting.

## Consequences Of Violation

Unnecessary 50–75% performance penalty on blacklisted files, wasted CPU on recompilation, masked code bugs.

## Rule Name

Fix the Code Instead of Blacklisting

## Category

Maintainability

## Rule

Never blacklist a file without first investigating whether the underlying code can be fixed to work correctly with OpCache.

## Reason

Most "OpCache incompatibilities" are actually code bugs — using `__FILE__` where `__DIR__` was intended, reading files with relative paths, or relying on `filemtime()` of the executing script. Fixing the code resolves the issue without sacrificing caching performance.

## Bad Example

```bash
# Blacklisted because __FILE__ returns cached path
# Instead, fix the code to use __DIR__
```

## Good Example

```php
// Before (breaks with OpCache): uses __FILE__ for resource path
$configPath = dirname(__FILE__) . '/../config/app.php';
// After (works with OpCache): uses __DIR__ for script directory
$configPath = __DIR__ . '/../config/app.php';
```

## Exceptions

Files from third-party packages that cannot be modified (submit a patch upstream instead).

## Consequences Of Violation

Per-request recompilation waste, deferred code quality improvements, technical debt accumulation.

## Rule Name

Use Absolute Paths in the Blacklist File

## Category

Maintainability

## Rule

Always use absolute paths when adding entries to the OpCache blacklist file. Never use relative paths.

## Reason

Relative paths may resolve differently depending on the current working directory when PHP executes. Absolute paths are unambiguous and guarantee the correct files are matched regardless of execution context.

## Bad Example

```
; Blacklist with relative paths — may not match
./temp/
tools/self-aware.php
```

## Good Example

```
; Blacklist with absolute paths
/var/www/html/temp/
/var/www/html/tools/self-aware.php
```

## Exceptions

No common exceptions. Always use absolute paths.

## Consequences Of Violation

Blacklist entries that silently do not match, files that should be excluded from caching being cached anyway, incorrect application behavior in production.

## Rule Name

Document Why Each File Is Blacklisted

## Category

Maintainability

## Rule

Always add a comment explaining the reason for every entry in the OpCache blacklist file.

## Reason

Without documentation, no one knows whether the issue that caused the blacklisting has been fixed. Blacklist entries accumulate over time, silently degrading performance as more files are excluded from caching.

## Bad Example

```
/var/www/html/generated-config.php
/var/www/html/tools/debug.php
```

## Good Example

```
; Dynamically generated by deployment script — changes every deploy
/var/www/html/generated-config.php
; Uses __FILE__ in a way that breaks with OpCache — fix tracked in JIRA-1234
/var/www/html/tools/debug.php
```

## Exceptions

No common exceptions. Always document.

## Consequences Of Violation

Orphaned blacklist entries that persist after the underlying issue is fixed, permanent performance penalty, team unaware of technical debt.

## Rule Name

Review Blacklist Quarterly and Remove Stale Entries

## Category

Maintainability

## Rule

Schedule a quarterly review of the OpCache blacklist file and remove entries whose underlying issues have been resolved.

## Reason

Blacklist entries are technical debt that permanently degrades performance. Code changes, library updates, or PHP version upgrades may fix the incompatibility. Regular reviews ensure the blacklist does not accumulate entries that are no longer needed.

## Bad Example

```bash
# Blacklist file has not been reviewed in 2 years
# 12 entries, 8 of which are no longer needed
# Unnecessary per-request recompilation continues
```

## Good Example

```bash
# Quarterly review removes 3 entries that were fixed upstream
# Remaining entries have active bug tracker references
```

## Exceptions

No common exceptions. Blacklist entries should always have a review process.

## Consequences Of Violation

Permanent performance degradation from unnecessary blacklisting, team unaware of removable entries.

## Rule Name

Restart PHP-FPM After Changing the Blacklist File

## Category

Maintainability

## Rule

Always restart PHP-FPM after modifying the OpCache blacklist file.

## Reason

OpCache reads the blacklist file only once at startup. Changes to the blacklist file are not picked up until PHP-FPM is restarted, regardless of `opcache_reset()`.

## Bad Example

```bash
# Modified blacklist and called opcache_reset()
# Blacklist changes have no effect — old blacklist still active
```

## Good Example

```bash
# Modified blacklist — restart required
sudo systemctl restart php8.5-fpm
```

## Exceptions

No common exceptions. Blacklist is loaded at PHP-FPM startup only.

## Consequences Of Violation

Blacklist modifications silently do not take effect, files continue to be cached or excluded incorrectly.

## Rule Name

Do Not Use Blacklist as a Development or Debugging Tool

## Category

Performance

## Rule

Never use the OpCache blacklist for debugging or development purposes. Disable OpCache entirely or use `validate_timestamps=1` instead.

## Reason

The blacklist is designed for specific production edge cases where caching causes incorrect behavior. Using it for debugging masks the actual issue and introduces a permanent performance penalty if the entry is forgotten.

## Bad Example

```bash
# Added file to blacklist to "see if OpCache is the problem"
# Forgot to remove it — file is now permanently uncached
```

## Good Example

```bash
# For debugging, temporarily disable OpCache
opcache.enable=0
# Or use validate_timestamps=1 in development
```

## Exceptions

No common exceptions.

## Consequences Of Violation

Accumulated blacklist entries from past debugging sessions, permanent performance degradation, unclear debugging trail.
