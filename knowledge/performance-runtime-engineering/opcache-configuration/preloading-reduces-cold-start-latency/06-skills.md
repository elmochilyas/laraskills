# Skill: Reduce Cold-Start Latency with OpCache Preloading

## Purpose

Configure `opcache.preload` to load framework classes at PHP-FPM startup, eliminating autoloading overhead for preloaded classes and reducing cold-start latency.

## When To Use

- Framework applications (Laravel, Symfony, Magento) with many classes loaded per request
- Cold-start latency (first request after deployment or restart) exceeds acceptable threshold
- Running Octane or memory-resident runtimes where preloading provides repeated benefit

## When NOT To Use

- For small applications (<500 classes) where autoloading overhead is minimal
- Without first configuring OpCache memory and max files
- When the preload script has side effects (database connections, file writes)
- For development environments where class changes are frequent

## Prerequisites

- PHP 7.4+ runtime
- OpCache enabled and configured
- List of frequently-loaded classes (from profiling or autoloader statistics)
- Understanding that preloading requires PHP-FPM restart to take effect

## Inputs

- Current OpCache configuration
- Autoloader class load statistics
- Framework type (Laravel, Symfony, etc.)
- PHP-FPM restart capability

## Workflow (numbered steps)

1. Profile the application to identify the most frequently autoloaded classes (top 100-200)
2. Create a preload script: `config/preload.php` that explicitly requires the identified classes
3. For Laravel: the optimized preload script can be generated by `php artisan optimize`
4. For Symfony: preload configuration is handled in `config/preload.php` generated by `composer dump-autoload --classmap-authoritative`
5. Set `opcache.preload=/path/to/config/preload.php` in php.ini
6. Set `opcache.preload_user=www-data` (or appropriate PHP-FPM user) for security
7. Restart PHP-FPM to execute the preload script and load classes into shared memory
8. Verify preloaded classes: `opcache_get_status(false)['preload_statistics']` should show the preloaded class list
9. Benchmark cold-start latency before and after preloading
10. Document the preloading configuration and expected latency reduction

## Validation Checklist

- [ ] Frequently-loaded classes identified (top 100-200)

- [ ] opcache.preload configured in php.ini

## Common Failures

- **Preloading with side effects**: Preload scripts that connect to databases or write files execute at every PHP-FPM restart
- **Not setting preload_user**: Without preload_user, preloaded classes are owned by the PHP-FPM user — security best practice to set it explicitly
- **Forgetting to invalidate preloaded classes**: opcache_reset() does NOT invalidate preloaded classes — requires full PHP-FPM restart
- **Preloading too many classes**: Preloading 1000+ classes increases startup time and memory without proportional benefit

## Decision Points

- Framework app (Laravel, Symfony): preload the top 100-200 framework classes
- Custom app with few dependencies: preloading may not be worth the configuration complexity
- Octane/long-running workers: preloading provides persistent benefit across thousands of requests
- Short-lived containers: preloading reduces cold-start but increases container start time — evaluate trade-off

## Performance Considerations

- Preloading reduces autoloading time by 1-3ms per request for preloaded classes
- Preloaded classes are loaded into shared memory at startup — all workers share them
- Total preload time at startup: 50-500ms depending on number of classes
- Preloaded classes cannot be garbage collected — they remain in memory for the lifetime of the PHP process

## Security Considerations

- Preload script runs with full privileges — only trusted code should be preloaded
- Set opcache.preload_user to the PHP-FPM user to restrict class ownership
- Preloaded classes are in shared memory — accessible to all workers on the same system
- Never preload classes that handle sensitive data initialization with side effects
