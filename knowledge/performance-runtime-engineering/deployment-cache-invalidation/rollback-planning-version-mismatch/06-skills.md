# Skill: Plan and Execute Rollback with OpCache Version Mismatch Handling

## Purpose
Design a one-click automated rollback in the CI/CD pipeline that reverts code, reloads workers, warms cache, and verifies health, handle OpCache file cache version mismatches when rolling back PHP minor versions (delete incompatible file cache before restart), kill and restart Octane workers during rollback (old code cannot handle state from new code), keep schema changes additive for backward compatibility, and test rollback after every production deployment — ensuring rollback completes in under 60 seconds.

## When To Use
- Every production deployment (have a rollback plan before deploying)
- PHP version upgrades (OpCache format changes between versions)
- Octane/Swoole/RoadRunner deployments (stateful rollback)
- Canary deployments (quick rollback to main pool)

## When NOT To Use
- Development/staging deployments (just fix forward)
- Blue-green deployments (rollback is switching load balancer back to blue)
- Stateless PHP-FPM deployments with pure code changes

## Prerequisites
- Automated rollback in CI/CD pipeline (one-click or on-failure trigger)
- Load balancer drain capability
- Understanding of Octane stateful worker requirements
- Additive (backward-compatible) schema changes

## Inputs
- Previous release directory path
- OpCache file cache path
- Load balancer target group ARN
- Octane server configuration (if applicable)

## Workflow

### 1. Build One-Click Automated Rollback
- Rollback pipeline stage: revert code → reload workers → warm cache → health check
- Execute in under 60 seconds for simple code rollbacks
- Trigger automatically on health check failure after deployment
- Manual rollback button for operator-initiated rollbacks

### 2. Handle OpCache Version Mismatch
- When rolling back PHP minor version (e.g., 8.5 → 8.4):
  - OpCache file cache format is NOT compatible across PHP minor versions
  - Delete file cache directory: `rm -rf /var/opcache-cache/*` before restart
  - Shared memory OpCache is automatically cleared by PHP-FPM restart
  - Never load file cache generated by a newer PHP version on an older PHP version

### 3. Kill and Restart Octane Workers
- Octane workers hold persistent state (DB connections, in-memory cache, objects)
- Old code cannot correctly handle state initialized by new code
- Sequence: drain → stop all workers → revert code → restart workers → warm → rejoin
- Never rely on code-only rollback without worker restart

### 4. Keep Schema Changes Additive
- Schema changes should be additive (new columns, new tables) for the current release cycle
- Old and rollback code must work with the current schema
- Destructive schema changes (DROP, RENAME) require separate, tested schema rollback
- Additive schema = fast code rollback without database revert

### 5. Drain Before Rollback Operations
- Signal load balancer to stop sending connections
- Wait for in-flight requests to complete
- Only then: revert code, restart workers, clear cache
- Without draining, active users experience dropped connections and errors

### 6. Test Rollback After Every Deployment
- After production deployment, test rollback in staging
- Verify: code reverts, workers restart, cache warms, health passes
- Rollback procedures and infrastructure drift over time — regular testing catches drift
- Untested rollback is not a rollback

- [ ] Schema changes additive for current release cycle

## Success Criteria
- Automated rollback completes in under 60 seconds for code-only rollbacks

- Load balancer drain prevents connection drops during rollback
