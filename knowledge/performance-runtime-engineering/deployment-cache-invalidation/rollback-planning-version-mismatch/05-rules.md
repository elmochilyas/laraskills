# Rules: Rollback Planning and Version Mismatch

---

### Rule 1: Have a One-Click Automated Rollback in the CI/CD Pipeline

**Category:** Reliability

**Rule:** Every deployment pipeline must include a one-click or automated rollback mechanism that reverts code, reloads workers, warms cache, and verifies health. Never rely on manual rollback procedures.

**Reason:** During a production incident, every second counts. Manual rollback requires engineers to diagnose the issue, decide to roll back, locate instructions, execute multiple commands, and verify the result — a 10+ minute process. Automated rollback executes in under 60 seconds.

**Bad Example:**
```yaml
# No rollback step in the pipeline
stages:
  - build
  - deploy
  - warm
  - health
```

**Good Example:**
```yaml
stages:
  - build
  - deploy
  - warm
  - health
  - enable
rollback:
  stage: rollback
  script: |
    rsync -a --delete /app/releases/previous/ /app/current/
    systemctl reload php8.3-fpm
    for url in / /api/health; do curl -s -o /dev/null http://localhost$url; done
    curl -s http://localhost/health | grep '"status":"ok"'
  when: on_failure
```

**Exceptions:** Blue-green deployments where rollback is instantaneous — just switch load balancer back to blue.

**Consequences Of Violation:** Extended production incidents (10+ minutes) while engineers manually execute rollback procedures under pressure, increasing user impact.

### Rule 2: Always Delete OpCache File Cache When Rolling Back PHP Version

**Category:** Performance

**Rule:** When rolling back to a previous PHP minor version, delete the OpCache file cache directory before restarting PHP-FPM. Never attempt to use OpCache file cache generated by a different PHP version.

**Reason:** OpCache file cache format is not guaranteed compatible across PHP minor versions. Loading a file cache generated by PHP 8.5 with PHP 8.4 can cause segmentation faults, fatal errors, or silent corruption of compiled classes.

**Bad Example:**
```bash
# Rolled back PHP but didn't clear file cache
systemctl stop php8.5-fpm
systemctl start php8.4-fpm
# Segfault from incompatible file cache
```

**Good Example:**
```bash
# Clear incompatible file cache before starting old PHP
systemctl stop php8.5-fpm
rm -rf /var/opcache-cache/*
systemctl start php8.4-fpm
```

**Exceptions:** Rollbacks that stay within the same PHP minor version with no OpCache format changes.

**Consequences Of Violation:** PHP-FPM crashes with segmentation faults after rollback, causing complete service unavailability until the file cache directory is manually cleared.

### Rule 3: Always Kill and Restart Octane Workers During Rollback

**Category:** Reliability

**Rule:** When rolling back an Octane-based application, fully kill and restart all workers. Never rely on partial worker recycling or code-only rollback.

**Reason:** Octane workers hold persistent state — database connections, in-memory caches, and application objects initialized by the new code. Old code cannot correctly handle state created by new code, leading to corruption, crashes, or inconsistent behavior.

**Bad Example:**
```bash
# Code rollback without worker restart
rsync -a --delete /app/releases/previous/ /app/current/
# Workers still have state from the new code
```

**Good Example:**
```bash
# Full worker restart after code rollback
rsync -a --delete /app/releases/previous/ /app/current/
php artisan octane:stop
php artisan octane:start --server=roadrunner --host=0.0.0.0 --port=8000
```

**Exceptions:** Stateless Octane configurations where workers hold no persistent application state.

**Consequences Of Violation:** Application crashes or undefined behavior after rollback as old code attempts to manipulate state structures created by new code.

### Rule 4: Keep Database Schema Changes Additive and Backward-Compatible for At Least One Release Cycle

**Category:** Maintainability

**Rule:** All database schema changes deployed alongside a code release must be additive (new columns, new tables) and compatible with the previous code version. Schema changes should never be destructive or require a coordinated code+schema rollback.

**Reason:** If a deployment is rolled back and the schema change is not backward-compatible, the old code cannot function with the new schema. This forces a schema rollback, which is slow, risky, and may cause data loss.

**Bad Example:**
```sql
-- Destructive change — old code breaks with new schema
ALTER TABLE users DROP COLUMN old_email;
ALTER TABLE users RENAME COLUMN email TO old_email;
```

**Good Example:**
```sql
-- Additive change — old code ignores new column
ALTER TABLE users ADD COLUMN new_email VARCHAR(255) NULL;
-- Old code continues using 'email', new code prefers 'new_email'
```

**Exceptions:** Zero-downtime deployments where schema rollback is tested and automated as part of the pipeline.

**Consequences Of Violation:** Rollback is blocked by incompatible schema, forcing teams to choose between reverting the schema (slow, risky) or leaving the problematic code in production.

### Rule 5: Test Rollback After Every Production Deployment

**Category:** Testing

**Rule:** After each production deployment, execute the rollback procedure in a staging environment. Verify that rollback completes successfully with correct cache state, worker state, and health status.

**Reason:** Rollback procedures and the infrastructure they depend on drift over time. Script paths change, credentials expire, and environment configurations evolve. Without regular testing, rollback will fail when needed — always during an incident.

**Bad Example:**
```bash
# Deploying regularly but never testing rollback
./deploy-production.sh
# Rollback script exists but hasn't been run in 6 months
```

**Good Example:**
```bash
./deploy-production.sh
# Test rollback in staging immediately after production deployment
./rollback-staging.sh
curl -s http://staging/health | grep '"status":"ok"'
curl -s http://staging/health | grep '"opcache_hit_rate":95'
```

**Exceptions:** None — rollback testing is mandatory after every deployment to ensure the safety net works.

**Consequences Of Violation:** During a real incident, the rollback script fails due to configuration drift, and engineers must manually execute emergency procedures, extending downtime.

---

### Rule 6: Drain Traffic Before Executing Any Rollback Procedure

**Category:** Reliability

**Rule:** Before executing any rollback action that involves worker restart or code replacement, signal the load balancer to drain the instance. Wait for in-flight requests to complete before proceeding.

**Reason:** Rollback operations (worker restart, code revert, cache clear) disrupt active requests. Draining first ensures current users complete their operations without errors, even while the rollback is happening.

**Bad Example:**
```bash
# Rollback without draining — drops active requests
systemctl restart php8.3-fpm
rsync -a --delete /app/releases/previous/ /app/current/
```

**Good Example:**
```bash
# Drain before rollback operations
aws elbv2 deregister-targets --target-group-arn $TG --targets Id=$INSTANCE
sleep 30
systemctl reload php8.3-fpm
rsync -a --delete /app/releases/previous/ /app/current/
for url in / /api/health; do curl -s -o /dev/null http://localhost$url; done
aws elbv2 register-targets --target-group-arn $TG --targets Id=$INSTANCE
```

**Exceptions:** Emergency security rollbacks where the risk of the current code poses a greater threat than dropped connections.

**Consequences Of Violation:** Users experience interrupted requests and errors during the rollback, turning a deployment incident into a customer-facing outage.
