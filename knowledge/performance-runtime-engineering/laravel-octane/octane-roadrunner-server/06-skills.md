# Skill: Operate and Tune Laravel Octane with RoadRunner Driver

## Purpose
Configure, deploy, and tune RoadRunner as the Laravel Octane driver — setting worker pool parameters (num_workers, max_jobs, supervisor limits), managing process lifecycle, securing the RPC interface, and tuning Goridge-based communication for production workloads.

## When To Use
- Running Laravel Octane with RoadRunner as the chosen driver
- Deploying RoadRunner to production for the first time
- Tuning RoadRunner worker pool settings based on monitoring data
- Debugging RoadRunner-specific issues (worker crashes, bootstrap timeouts, RPC connectivity)
- Setting up RoadRunner process supervision and monitoring

## When NOT To Use
- When using Swoole or FrankenPHP as the Octane driver (different configuration semantics)
- For local development with the RoadRunner binary (use `--watch` flag instead)
- Without first selecting RoadRunner as the driver (see driver selection skill)

## Prerequisites
- Laravel application with `laravel/octane` installed
- RoadRunner selected as the driver (`php artisan octane:install` with RoadRunner option)
- `rr` binary downloaded and executable in the project directory
- `config/octane.php` with RoadRunner-specific configuration
- Server with PHP 8.1+, required extensions, and access to install the `rr` binary
- Understanding of Laravel Octane's persistent execution model

## Inputs
- `.rr.yaml` configuration file (generated by octane:install or manually)
- Server specifications: CPU cores, RAM, database max_connections
- Workload profile: CPU-bound or I/O-bound
- Current worker RSS (idle and under load)
- Current listen queue depth
- Database connection utilization

## Workflow

### 1. Verify RoadRunner Installation
- Confirm `rr` binary exists and is executable: `./rr --version`
- Verify `config/octane.php` has `'server' => 'roadrunner'`
- Check `.rr.yaml` was created in the project root
- Verify Goridge extension is not required (RoadRunner uses standard PHP I/O — pipes)
- Run `php artisan octane:start --server=roadrunner` and confirm the server starts

### 2. Configure Worker Pool Parameters in .rr.yaml
- Set `http.pool.num_workers`: CPU cores for CPU-bound, 1.5-2× for I/O-bound
- Set `http.pool.max_jobs` (equivalent to max_requests): 1000 as safe default
- Set `http.pool.allocate_timeout`: 60s (Laravel bootstrap takes 2-10s)
- Set `http.pool.supervisor.max_workers`: num_workers × 1.5 (cap for worker expansion)
- Set `http.pool.supervisor.watch_ticks`: 10 (health check interval)
- Example configuration:
```yaml
http:
  address: "0.0.0.0:8080"
  pool:
    num_workers: 8
    max_jobs: 1000
    allocate_timeout: 60s
    supervisor:
      max_workers: 12
      watch_ticks: 10
```

### 3. Secure the RPC Interface
- Set `rpc.listen` to `tcp://127.0.0.1:6001` — never bind to `0.0.0.0`
- Verify RPC is not accessible externally: `Test-NetConnection -ComputerName <server> -Port 6001` should fail
- Configure firewall rules to block external access to port 6001
- Document the RPC port for monitoring tools that need local access

### 4. Configure Process Supervision
- Set up Supervisor (or systemd) to manage the `rr` binary:
```ini
[program:octane]
command=/var/www/app/rr serve -c /var/www/app/.rr.yaml
user=www-data
autostart=true
autorestart=true
startretries=3
stderr_logfile=/var/log/octane.err.log
stdout_logfile=/var/log/octane.out.log
```
- RoadRunner manages its own internal worker pool — Supervisor's role is to restart the Go process if it crashes
- Set `stopwaitsecs=30` to allow in-flight requests to drain during restart

### 5. Tune Worker Count Based on Monitoring
- After deployment, monitor for 24-48 hours:
  - Listen queue depth (primary tuning signal)
  - Per-worker RSS growth
  - Database connection utilization
- If listen queue is consistently 0: workers are sufficient
- If listen queue is growing: increase num_workers by 25-50%
- If RSS > 80% of memory budget or connections > 80% of max_connections: decrease num_workers or add servers
- Adjust max_jobs based on RSS growth rate:
  - RSS growth < 5% per 1000 requests: increase max_jobs to 5000
  - RSS growth > 20% per 1000 requests: decrease max_jobs to 300-500

### 6. Implement Graceful Reload Procedure
- Use `php artisan octane:reload` for zero-downtime code deployments
- This sends SIGTERM to workers one at a time — each finishes its current request then exits
- New workers boot with the latest code
- Monitor the reload: `rr workers -i` shows workers cycling
- Verify no in-flight requests are dropped during reload (use concurrent load test)

### 7. Set Up RoadRunner-Specific Monitoring
- Monitor `rr` process memory (Go binary is ~30MB baseline, should be stable)
- Monitor per-worker RSS via `rr workers -i` or Octane health endpoint
- Set up logging: `stderr_logfile` captures PHP worker errors (fatal errors, exceptions outside request lifecycle)
- Configure metric collection: RoadRunner supports Prometheus via a plugin
- Alert on: worker crashes, RSS > threshold, listen queue > 0 for more than 30 seconds

### 8. Test RoadRunner-Specific Failure Modes
- Simulate worker crash: kill a worker process, verify RoadRunner spawns a replacement
- Simulate `rr` process crash: kill the Go binary, verify Supervisor restarts it
- Test connection exhaustion: fill database connections, verify workers fail gracefully
- Test bootstrap timeout: temporarily add sleep() to provider boot, verify allocate_timeout catches it

## Common Failures

| Failure | Symptom | Root Cause | Mitigation |
|---------|---------|------------|------------|
| Workers killed during bootstrap | Workers in "killed" state | allocate_timeout too short for Laravel bootstrap (2-10s) | Set allocate_timeout to 60s |
| Database connection exhaustion | Connection refused errors | Workers × connections exceed max_connections | Reduce num_workers or increase max_connections |
| Memory exhaustion | OOM kills, workers crash | num_workers too high for available RAM | Reduce num_workers, increase max_jobs recycling |
| RPC port exposed | External access to worker management | rpc.listen set to 0.0.0.0 | Bind to 127.0.0.1, add firewall rule |
| Worker count drops under load | Less workers than num_workers | Supervisor.max_workers reached | Increase max_workers or investigate why workers are dying |
| Go binary memory grows | rr RSS increases over time | Plugin memory leak or goroutine leak | Restart rr periodically, investigate Go version compatibility |

## Decision Points

| Decision | How To Decide |
|----------|---------------|
| num_workers value | CPU cores for CPU-bound, 1.5-2× for I/O-bound. Constrain by memory and DB connections |
| max_jobs value | 1000 default. Lower if RSS grows >20% per 1000 requests. Raise to 5000 if RSS <5% growth |
| allocate_timeout | 60s for production. 30s for apps with fast bootstrap. 10s minimum |
| Supervisor vs Docker/K8s | Supervisor for traditional servers. Container orchestration for Kubernetes. Docker for single-server deployments |
| With or without RPC | Always configure RPC on localhost for management commands. Secure with firewall |

## Performance Considerations
- RoadRunner adds ~30MB baseline for the Go binary (stable, doesn't grow with traffic)
- Each PHP worker adds 30-80MB RSS — total memory = Go binary + (workers × per-worker RSS)
- Goridge protocol adds ~0.1-0.5ms per request vs embedded SAPI (FrankenPHP)
- Process isolation ensures a worker crash doesn't affect other workers — worth the overhead
- Workers maintain persistent database connections — account in connection budget
- Go goroutine scheduler handles thousands of concurrent connections efficiently
- RoadRunner's performance advantage over PHP-FPM is 41-111% depending on workload

## Security Considerations
- RPC interface provides administrative access — must be bound to 127.0.0.1
- Workers inherit environment variables from the `rr` process — do not store secrets in `.rr.yaml`
- Each worker runs as a separate OS process — strongest isolation model among Octane drivers
- PHP worker crashes are contained within the individual process
- The `rr` binary itself should be readable only by the deploying user
- File permissions on `.rr.yaml` should restrict access (it may contain configuration details)
