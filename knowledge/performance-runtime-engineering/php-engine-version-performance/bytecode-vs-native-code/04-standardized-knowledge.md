## Overview

PHP operates at two compiled-code levels: **bytecode** (Zend opcodes, stored by OpCache) and **native machine code** (x86-64/ARM64 instructions, generated by JIT). Bytecode is interpreted by the Zend VM; native code runs directly on the CPU. OpCache eliminates re-compilation to bytecode. JIT eliminates VM dispatch overhead for hot code paths.

## Core Concepts

- **Bytecode/opcodes**: Platform-independent intermediate representation (~200 instruction types). Each opcode is ~12 bytes. Stored in OpCache shared memory.
- **Native code**: CPU-specific machine instructions. Generated by JIT compiler via DynASM. Stored in JIT buffer (separate from OpCache).
- **VM dispatch overhead**: Each opcode requires handler function call, operand decode, and result write-back — ~5-15 CPU cycles per opcode.
- **JIT eliminates dispatch**: Native code has no opcode fetch/decode overhead; operations flow directly via registers.

## When To Use

- Understanding PHP's execution model for performance tuning
- Deciding between OpCache vs JIT optimization investments
- Debugging performance issues at the execution level
- Capacity planning for OpCache memory and JIT buffer size

## When NOT To Use

- When tuning application-level performance (focus on profiling hotspots instead)
- When JIT is disabled or unavailable (pre-JIT PHP versions)
- For understanding framework-level performance characteristics

## Best Practices (WHY)

- **Enable OpCache first**: Provides 2-4x throughput vs no opcode caching. Single highest-ROI change.
- **Add JIT for CPU-bound workloads**: JIT provides 61-95% additional gain for CPU-bound tasks but only 0-5% for I/O-bound web apps.
- **Size JIT buffer appropriately**: Default 128MB is sufficient for most workloads. CPU-heavy apps may benefit from 256MB.
- **Understand the two-level cache**: OpCache caches bytecode across requests (shared memory). JIT caches native code within a process (per-request buffer).

## Architecture Guidelines

- **Pipeline**: Source -> Lexer -> Parser -> Compiler -> Opcodes -> (JIT) -> Native code -> Execution
- **OpCache interception**: On warm requests, OpCache serves pre-compiled opcodes from shared memory, skipping lex/parse/compile phases entirely.
- **JIT interception**: At opcode dispatch level, JIT redirects hot traces to compiled native code paths, skipping the VM dispatch loop.

## Performance

- OpCache provides 2-4x throughput vs no opcode caching
- JIT provides 0-5% additional gain for I/O-bound web apps but 61-95% for CPU-bound tasks
- Memory overhead: OpCache uses configurable shared memory (default 128MB); JIT buffer adds 64-256MB
- Opcode dispatch via computed goto (PHP 8.4+): ~5-8% execution speedup

## Security

- JIT buffer is writable and executable memory — a JIT bug could theoretically be exploited
- JIT is disabled when running PHP in debug mode or with certain opcode caches
- OpCache shared memory is isolated per-user in multi-tenant environments

## Common Mistakes

| Mistake | Cause | Consequence | Better Approach |
|---------|-------|-------------|-----------------|
| Assuming JIT replaces OpCache | Misunderstanding two-level architecture | Missing OpCache's bytecode caching benefit | Both are complementary; enable both |
| Not sizing JIT buffer | Using default for CPU-bound workloads | JIT thrashing, wasted compilation | Set jit_buffer_size=256M for CPU-heavy apps |
| Expecting JIT to fix I/O-bound apps | Assuming "native code = faster always" | Disappointment, wasted RAM | Profile first; JIT helps CPU-bound only |
| Disabling OpCache in production | Dev environment habit | 2-4x throughput loss | Always enable OpCache in production |

## Anti-Patterns

- **Treating bytecode and native code as alternatives**: They are complementary layers. OpCache caches bytecode; JIT compiles hot bytecode to native code. Enable both.
- **Over-provisioning JIT buffer**: 256MB+ for I/O-bound lightweight apps wastes RAM. Size based on workload profile.
- **Disabling OpCache for "freshness"**: Use opcache_reset() for deployments instead of disabling OpCache entirely.
