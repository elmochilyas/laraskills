# Rules: COS-08 — Naming Conventions

## R01: Use Verb-Noun Pattern for Action Classes
---
## Category
Code Organization
---
## Rule
Name action classes with Verb-Noun syntax: `CreateInvoice`, `ProcessRefund`, `SendWelcomeEmail`.
---
## Reason
Verb-Noun communicates exactly what the class does — both the operation and the subject. The name is the contract. A class called `CreateUser` does one thing: create a user. This eliminates ambiguity about class responsibility.
---
## Bad Example
```php
class UserCreation { }     // What kind of creation?
class CreateUserAction { } // Redundant suffix — "Action" adds nothing
class UserCreator { }      // Is this a service, a factory, or an action?
```
---
## Good Example
```php
class CreateUser { }      // Exactly one thing: create a user
class ProcessRefund { }   // Exactly one thing: process a refund
class SendEmail { }       // Exactly one thing: send an email
```
---
## Exceptions
When retrofitting actions into an existing codebase with established naming, follow the existing pattern.
---
## Consequences Of Violation
Ambiguous class names. Developers must open the file to understand what the class does. Inconsistent naming across the codebase.
---

## R02: Use Past-Tense for Event Classes
---
## Category
Code Organization
---
## Rule
Name event classes using past-tense verbs: `UserRegistered`, `OrderShipped`, `PaymentFailed`.
---
## Reason
Events describe something that has already happened. Past-tense naming communicates this temporal semantics clearly — unlike a command (future/imperative), an event is a record of the past.
---
## Bad Example
```php
class RegisterUser { }      // Sounds like a command/action, not an event
class UserRegistration { }  // Ambiguous — is this the process or the event?
```
---
## Good Example
```php
class UserRegistered { }    // Clearly an event — it happened already
class OrderShipped { }      // Clearly an event
class PaymentFailed { }     // Clearly an event
```
---
## Exceptions
Events generated by third-party packages may follow different conventions.
---
## Consequences Of Violation
Confusion between commands/actions and events. Developers may accidentally dispatch actions as events and vice versa.
---

## R03: Use {Domain}Service for Service Names
---
## Category
Code Organization
---
## Rule
Name service classes with a domain or entity qualifier: `PaymentService`, `UserService`, `BillingService`. Never name a service just `Service`.
---
## Reason
A class named `Service` communicates nothing about what it does. As the project grows, unqualified names become ambiguous. `PaymentService` immediately communicates the service's domain of responsibility.
---
## Bad Example
```php
class Service { }           // Service of what? Payment? Email? User?
class DataService { }       // Which data? All data?
class ServiceProvider { }   // Generic, conflicts with Laravel concept
```
---
## Good Example
```php
class PaymentService { }    // Clear — handles payment operations
class UserService { }       // Clear — handles user operations
class BillingService { }    // Clear — handles billing operations
```
---
## Exceptions
No common exceptions — always qualify service names.
---
## Consequences Of Violation
Ambiguous service names force developers to open files to understand responsibility. Name collisions as the project grows.
---

## R04: Never Use `Manager`, `Helper`, `Handler`, or `Processor` as Suffixes
---
## Category
Code Organization
---
## Rule
Avoid generic suffixes like `Manager`, `Helper`, `Handler`, `Processor`. Use role-specific suffixes like `Service`, `Action`, `UseCase` instead.
---
## Reason
Generic suffixes do not communicate architectural role — they are placeholders for "I don't know what this class does." `Helper` is the worst offender: it can contain anything. Specific suffixes make responsibility clear from the filename alone.
---
## Bad Example
```php
class DataManager { }      // What data? Managing what?
class StringHelper { }     // Static utility? OOP wrapper?
class PaymentHandler { }   // Handling what? Processing? Storing?
class ObjectProcessor { }  // Processing objects? What kind?
```
---
## Good Example
```php
class InvoiceService { }   // Orchestrates invoice operations
class CreateInvoice { }    // Single action — create an invoice
class InvoiceNormalizer { }     // Transforms invoice data
class PaymentGateway { }   // Abstractions over external payment provider
```
---
## Exceptions
Framework base classes that use these suffixes (e.g., Laravel's `ExceptionHandler`).
---
## Consequences Of Violation
Catch-all classes that grow unbounded. Classes named `Helper` become dumping grounds for unrelated static methods.
---

## R05: Use Singular for Service and Model Names
---
## Category
Code Organization
---
## Rule
Name models and services in singular: `User`, `Invoice`, `UserService`, `InvoiceService`. Use plural only for controllers: `UsersController`.
---
## Reason
Singular class names represent a single instance (one User, one Service). Controllers handle collections (HTTP resources are typically plural). This is the Laravel convention and the PHP community standard.
---
## Bad Example
```php
class UsersService { }     // Singular service with plural name
class InvoicesController { } // Plural — correct (controller)
class UserController { }   // Singular — uncommon in Laravel conventions
```
---
## Good Example
```php
class UserService { }      // Singular — correct (handles one User domain)
class InvoiceService { }   // Singular — correct
class UsersController { }  // Plural — correct (CRUD for User resource)
```
---
## Exceptions
Controllers following custom routes where the resource name is inherently singular (e.g., `ProfileController`).
---
## Consequences Of Violation
Inconsistent naming across the codebase. Confusion about whether `UsersService` handles multiple users or one user.
---

## R06: Use Noun+Data or Noun+Dto for Data Transfer Objects
---
## Category
Code Organization
---
## Rule
Name DTOs with `Data` or `Dto` suffix: `UserData`, `CreateInvoiceDto`, `RegistrationData`.
---
## Reason
DTOs serve a distinct role from models or value objects — they carry data between boundaries without behavior. The `Data` or `Dto` suffix immediately communicates "this is a data container, not a domain entity."
---
## Bad Example
```php
class CreateUserRequest { }    // Sounds like a Form Request
class UserPayload { }           // Vague — what kind of payload?
class RegistrationObject { }    // Not clear what role this serves
```
---
## Good Example
```php
class UserData { }              // Clear — immutable data container
class CreateInvoiceDto { }      // Clear — DTO for invoice creation
class RegistrationData { }      // Clear — registration data transfer
```
---
## Exceptions
Projects using a different established convention (e.g., `ValueObject` suffix for value objects).
---
## Consequences Of Violation
DTOs confused with models, Form Requests, or value objects. Inconsistent naming prevents clear role identification.
---

## R07: Ensure File Name Matches Class Name Exactly
---
## Category
Framework Usage
---
## Rule
Name every PHP file exactly the same as the class it contains, including case: `CreateInvoice.php` contains `class CreateInvoice`.
---
## Reason
PSR-4 autoloading resolves class names to file names. A mismatch (`invoiceService.php` containing `class InvoiceService`) causes class-not-found errors. Case-sensitive production filesystems make even casing mismatches fatal.
---
## Bad Example
```php
// File: app/Services/paymentService.php
class PaymentService { }
// Autoloader looks for PaymentService.php, not paymentService.php
```
---
## Good Example
```php
// File: app/Services/PaymentService.php
class PaymentService { }
// Perfect match — autoloader resolves correctly
```
---
## Exceptions
Legacy files being migrated — temporary, with documented plan to rename.
---
## Consequences Of Violation
ClassNotFoundException in production. IDE navigation broken. `composer dump-autoload` validation fails.
---

## R08: Document Naming Conventions in Team Documentation
---
## Category
Maintainability
---
## Rule
Write down naming conventions in CONTRIBUTING.md or project ADRs — class name patterns, suffix rules, and examples.
---
## Reason
Without documentation, naming conventions are tribal knowledge. New developers learn by making mistakes in code review. Documentation ensures consistent application and enables automated linting.
---
## Bad Example
```php
// No naming convention documentation
// Team of 8 produces 5 different naming styles:
// UserRegister, RegisterUser, RegistrationService, UserCreator, NewUser
```
---
## Good Example
```markdown
// CONTRIBUTING.md:
// ## Naming Conventions
// - Actions: VerbNoun (CreateUser, ProcessPayment)
// - Events: NounPastTense (UserRegistered, PaymentFailed)
// - Services: {Domain}Service (UserService, BillingService)
// - DTOs: NounData or NounDto (UserData, CreateInvoiceDto)
```
---
## Exceptions
Single-developer projects with no external contributors.
---
## Consequences Of Violation
Inconsistent naming across the codebase. Proliferation of `Manager`, `Helper`, and `Handler` classes. Code review fatigue correcting naming violations.
