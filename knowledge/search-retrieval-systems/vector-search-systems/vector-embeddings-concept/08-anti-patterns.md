## 1. Unnormalized Embeddings

**Category:** Performance

**Description:** Using embeddings without verifying or enforcing unit vector normalization, causing inconsistent distance computations and degraded search accuracy.

**Why It Happens:** Some embedding APIs return normalized vectors by default while others do not. Developers assume consistency across providers without explicit checks. Normalization is often overlooked during initial implementation when results "look correct" with small datasets, hiding the issue until scale.

**Warning Signs:**
- Cosine similarity produces unexpected rankings
- Distance values vary widely between queries
- Results differ between development and production environments
- Embeddings generated by different models produce incomparable vectors

**Why Harmful:** Unnormalized vectors break the mathematical assumptions underlying cosine similarity and dot product operations. Cosine similarity requires unit vectors to produce meaningful [0,1] range results. Without normalization, query results are unreliable and search quality degrades non-deterministically.

**Consequences:**
- Incorrect similarity rankings
- Inconsistent search results across embedding batches
- Debugging difficulty when comparing vectors across different models
- Wasted time on tuning other parameters while root cause is normalization

**Alternative:** Always normalize embeddings immediately after generation, before storage. Use a consistent normalization pipeline that verifies unit vector length (within tolerance) for every embedding.

**Refactoring Strategy:**
1. Add a normalization step in the embedding pipeline after API response
2. Verify existing stored embeddings — compute norm, flag deviations
3. Re-normalize or regenerate non-normalized stored embeddings
4. Add CI check that verifies embedding norms before write operations
5. Document which embedding models require normalization vs. output normalized by default

**Detection Checklist:**
- [ ] Is there a normalization step after embedding generation?
- [ ] Are stored embeddings verified for unit vector length?
- [ ] Does the pipeline handle both normalized and unnormalized model outputs?
- [ ] Are integration tests checking similarity ranking consistency?

## 2. Mismatched Distance Metric

**Category:** Design

**Description:** Using a distance metric (cosine, dot product, L2) that does not match the embedding model's training objective, reducing retrieval accuracy.

**Why It Happens:** Developers often default to cosine similarity without verifying which metric the embedding model was trained with. Some vector databases or libraries default to L2 distance. The difference between metrics is subtle and easy to miss during initial setup when small datasets yield acceptable results regardless of metric choice.

**Warning Signs:**
- Search results seem semantically irrelevant despite correct embedding generation
- Accuracy metrics are lower than published model benchmarks
- Changing the distance metric significantly changes result rankings
- Different teams use different metrics for the same embedding model

**Why Harmful:** A mismatched metric fundamentally undermines the geometry the embedding model learned during training. The model was optimized during training to place similar items close together in a specific distance space. Using a different metric for search measures distances in a way the model was never trained for, producing suboptimal rankings.

**Consequences:**
- 10-40% reduction in retrieval recall depending on the model
- Inconsistent search quality that cannot be tuned away
- Wasted time on re-embedding, re-indexing, and parameter tuning
- False conclusion that a particular embedding model is "bad"

**Alternative:** Always verify the recommended distance metric for your specific embedding model (documented by the provider). Use cosine distance for OpenAI, Cohere, and most sentence-transformer models; use dot product for models explicitly trained with inner-product loss.

**Refactoring Strategy:**
1. Check embedding model documentation for recommended distance metric
2. Change vector store index to use correct metric (may require re-indexing)
3. Re-index all stored embeddings with the correct metric
4. Verify improved search accuracy with A/B comparison
5. Add metric choice to runbook for future model changes

**Detection Checklist:**
- [ ] Is the distance metric documented for the chosen embedding model?
- [ ] Does the configured metric match the model's training objective?
- [ ] Is re-indexing required to change the metric in production?
- [ ] Are accuracy benchmarks run for the chosen metric before production?

## 3. No Embedding Caching

**Category:** Performance

**Description:** Generating embeddings from scratch every time content is indexed, without caching previously computed embeddings for identical text.

**Why It Happens:** Initial implementations often embed content once during import, making caching seem unnecessary. As the system grows to handle real-time indexing, webhook-triggered updates, or re-processing, the same text gets embedded repeatedly. Developers may not realize how frequently the same content passes through the embedding pipeline.

**Warning Signs:**
- API costs for embedding generation grow linearly with total processing volume, not unique content
- Repeated content in the system generates duplicate embeddings
- Re-indexing operations regenerate embeddings for unchanged content
- Embedding generation is a bottleneck in the indexing pipeline

**Why Harmful:** Embedding generation is expensive both in API cost ($0.02-0.13 per million tokens) and latency (100-500ms per API call). Without caching, the same text embedded N times costs N× the time and money. For systems with recurring content (daily crawls, webhook re-processing), this waste compounds significantly.

**Consequences:**
- Unnecessary API costs that grow linearly with processing frequency
- Increased indexing latency due to redundant API calls
- Higher rate-limit contention for embedding API providers
- Environmental impact of unnecessary computation

**Alternative:** Cache embeddings by content hash (MD5 or SHA-256 of normalized text) using a persistent cache store (Redis, database, or filesystem). Use the cached embedding when available; generate and store only for unseen text.

**Refactoring Strategy:**
1. Add content hashing before embedding generation
2. Implement cache check (lookup by hash) before API call
3. Store new embeddings keyed by content hash after API response
4. Set appropriate cache TTL (embeddings are immutable per text)
5. Add cache hit/miss metrics to monitor effectiveness
6. Consider warm caching for known frequently-embedded content

**Detection Checklist:**
- [ ] Is there a cache layer before embedding API calls?
- [ ] Are embeddings keyed by content hash (not model ID)?
- [ ] Are cache hit rates monitored?
- [ ] Does re-indexing reuse cached embeddings for unchanged content?

## 4. Over-Embedding Everything

**Category:** Architecture

**Description:** Generating vector embeddings for all content indiscriminately — including data that never needs semantic search — wasting storage and compute resources.

**Why It Happens:** The excitement around semantic search and RAG systems creates pressure to "vectorize everything." Teams often embed entire database tables without analyzing which fields benefit from semantic search. The low initial cost of embedding a few thousand records masks the long-term storage and maintenance burden at scale.

**Warning Signs:**
- Vector storage size exceeds the original data size by 5-10×
- Embedding costs dominate the search system budget
- Embeddings are stored for fields used only for exact-match filtering
- Most embedding storage is never queried
- Index maintenance (ANN rebuilds) takes hours for mostly irrelevant data

**Why Harmful:** Each embedding vector consumes storage equal to dimensionality × bytes per dimension (e.g., 1536 × 4 bytes = 6KB per vector for float32). For 10 million records, that's 60GB of vectors alone. Embedding generation costs, index build times, and query latency all scale with total vector count. Embedding data that is never semantically searched wastes all these resources.

**Consequences:**
- Significant unnecessary storage costs ($0.10-0.50/GB/month in cloud)
- Slower ANN index builds due to larger vector count
- Higher query latency from scanning through irrelevant vectors
- Increased embedding API costs for non-searchable content
- More complex maintenance operations (backup, re-index)

**Alternative:** Only embed fields and entities that will actually be searched semantically. Use traditional database indexes (B-tree, full-text) for exact-match fields like IDs, categories, dates, and enums. Apply vector embedding selectively to text-heavy, meaning-rich fields (descriptions, articles, reviews, titles).

**Refactoring Strategy:**
1. Audit all existing vector fields — classify as "semantically searched" vs. "stored but never queried"
2. Remove non-searchable vector fields; migrate filtering to traditional indexes
3. Redesign the embedding schema to include only searchable fields
4. Reduce vector database size by re-indexing only semantically useful embeddings
5. Establish a data governance rule: "embed only what you search"

**Detection Checklist:**
- [ ] Is there a documented policy for what gets embedded?
- [ ] Are all embedded fields actually used in similarity search queries?
- [ ] Is vector storage cost tracked and attributed per entity?
- [ ] Do exact-match filters use vector fields instead of traditional indexes?

## 5. Wrong Dimensionality Choice

**Category:** Scalability

**Description:** Using unnecessarily high-dimensional embeddings (e.g., 3072-dim text-embedding-3-large) when lower dimensions (e.g., 1536-dim small) would meet accuracy requirements, increasing storage and compute costs.

**Why It Happens:** Developers gravitate toward the largest available embedding model ("more dimensions = more accuracy" intuition). Embedding provider defaults often point to larger models. The incremental accuracy gain from higher dimensions is rarely measured against the cost increase, and the storage impact is hidden until scale exceeds initial expectations.

**Warning Signs:**
- Search accuracy is acceptable but dimensions are at the maximum available
- Embedding storage costs are unexpectedly high
- ANN index build times are slow (build time correlates with dimensions)
- Recall-at-k metrics do not improve significantly with higher dimensions
- Vector database memory usage exceeds budget for HNSW indexes

**Why Harmful:** Storage and query costs scale linearly with embedding dimensionality. HNSW index memory usage is approximately dimensions × M × 4 bytes per vector. Doubling dimensions from 768 to 1536 doubles memory. Query latency also increases with dimensions because distance computation requires iterating over more dimensions. The marginal accuracy gain from 1536→3072 dimensions is typically <1% for most search tasks.

**Consequences:**
- 2-4× higher vector storage costs than necessary
- Slower ANN index builds and queries
- Higher memory requirements for HNSW graph structures
- Longer distance computation time per query
- Increased backup and transfer costs

**Alternative:** Start with the smallest embedding model that meets your accuracy requirements. Benchmark accuracy at each dimensionality level (e.g., 256, 512, 768, 1536, 3072) using your specific dataset and query types. Choose the lowest dimension that achieves acceptable recall.

**Refactoring Strategy:**
1. Benchmark current search accuracy with production queries
2. Test lower-dimensional embeddings with the same benchmark
3. If accuracy is acceptable at lower dimensions, plan migration
4. Re-embed content with lower-dimensional model
5. Update vector store schema and re-index with new dimensions
6. Verify accuracy post-migration is within acceptable thresholds

**Detection Checklist:**
- [ ] Is dimensionality justified by accuracy benchmarks (not guesses)?
- [ ] Have lower-dimensional alternatives been evaluated?
- [ ] Are storage costs tracked per dimension level?
- [ ] Is the dimension-to-accuracy tradeoff documented in the system design?
