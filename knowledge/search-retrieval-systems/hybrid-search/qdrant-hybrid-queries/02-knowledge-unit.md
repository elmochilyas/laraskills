# Knowledge Unit: Qdrant Hybrid Queries

## Metadata

- **ID:** K049
- **Subdomain:** Hybrid Search
- **Source:** Qdrant Docs
- **Maturity:** Stable
- **Laravel Relevance:** Built-in hybrid search (dense + sparse)

## Executive Summary

Qdrant provides native hybrid search by combining dense vector search (from embedding models) with sparse vector search (representing keyword importance, like BM25). The fusion happens server-side using a configurable scoring strategy (RRF or DBSF). This eliminates the need for application-level fusion logic.

## Core Concepts

- **Dense Vectors**: Traditional embeddings from models like OpenAI, Cohere.
- **Sparse Vectors**: High-dimensional sparse representations (e.g., SPLADE, BM25-derived) where each dimension represents a vocabulary token.
- **Server-Side Fusion**: Qdrant handles the combination internally using RRF or DBSF (Distribution-Based Score Fusion).
- **Single Query**: One API call with both dense and sparse vectors — Qdrant returns fused results.

## Internal Mechanics

Qdrant maintains separate indexes for dense and sparse vectors. A hybrid query provides both vectors. Qdrant searches each index independently, then applies the configured fusion strategy within the server. RRF is the default. The `fusion` parameter in the query controls the strategy. Results are returned as a single scored list.

## Patterns

- **Best-of-both**: Dense captures semantics, sparse captures exact keyword matches and rare terms.
- **One API call**: Simpler than managing two separate queries and fusion logic.
- **Tunable fusion**: Experiment with RRF vs DBSF for different content types.

## Architectural Decisions

Qdrant's native hybrid search reflects a design philosophy that dense-only vector search misses important keyword signals. Providing both natively (rather than requiring external fusion) simplifies the application stack.

## Tradeoffs

- Native hybrid search simplifies application code but ties you to Qdrant as the vector store.
- Dense + sparse indexing doubles index storage and memory requirements.
- RRF is simple and effective; DBSF is more nuanced but harder to tune.

## Performance Considerations

- Hybrid queries search two indexes — latency is approximately the sum of both searches plus fusion overhead.
- Sparse vector search can be slower than dense on large vocabularies.
- Memory usage approximately doubles due to maintaining both indexes.
- Candidate pooling (prefetch) limits can be tuned to balance recall vs latency.

## Production Considerations

- **Generate both dense and sparse vectors** at indexing time. Sparse vectors can be generated by models like SPLADE or BM25-derived.
- **Tune the fusion strategy** — test RRF vs DBSF on your specific dataset.
- **Monitor index growth** — sparse vector indexes can be large for high-vocabulary content.
- **Configure prefetch limits** — how many candidates are retrieved by each path before fusion.

## Common Mistakes

- Providing only dense vectors — hybrid search requires both vector types.
- Not tuning fusion parameters — defaults may not fit your use case.
- Expecting sparse vectors from models that only produce dense embeddings.

## Failure Modes

- **One path fails**: If dense search works but sparse search errors, hybrid may return only dense results or fail.
- **Sparse vector quality**: Poorly generated sparse vectors degrade hybrid results below pure dense search.
- **Storage exhaustion**: Maintaining both indexes on large datasets may exceed storage capacity.

## Ecosystem Usage

Used by teams that want built-in hybrid search without building custom fusion. Especially valuable for applications where exact keyword matching (proper nouns, codes) is important alongside semantic understanding.

## Research Notes

Source: Qdrant docs. Qdrant's hybrid search is the most mature built-in fusion among vector databases. Sparse vector support enables true hybrid (not just dense + metadata filtering). The DBSF strategy is unique to Qdrant among major vector databases.


## Mental Models

- **Payload as Passport**: Qdrant treats vector search as identity verification and payload filtering as passport checks. A vector finds candidates, then payload filters validate their credentials.
- **Storage Engine**: Qdrant's HNSW index is like a skip list in high-dimensional space — you navigate through layers of increasing precision to find nearest neighbors.
