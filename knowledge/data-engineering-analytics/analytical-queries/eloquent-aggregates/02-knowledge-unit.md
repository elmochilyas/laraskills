## Executive Summary

Eloquent's aggregate methods (`withSum`, `withAvg`, `withCount`, `withMin`, `withMax`) and subquery patterns (`addSelect` with closures, `joinSub`, `whereExists`) bridge the gap between ORM convenience and analytical query power by attaching aggregated values from related tables as computed columns on the parent result. These methods are SQL optimizations that replace N+1 lazy-loaded relationship counts with a single query using correlated subqueries or lateral joins.

---

## Core Concepts

- **withCount():** Adds `{relation}_count` column to parent query result — generates correlated subquery `SELECT *, (SELECT COUNT(*) FROM orders WHERE user_id = users.id) as orders_count FROM users` — single query, not N+1
- **withSum() / withAvg() / withMin() / withMax():** Similar to `withCount` but for summation and statistical aggregation — works on any numeric column in related table
- **addSelect() with Subquery:** Most flexible pattern — allows adding arbitrary subquery results as computed columns: `User::addSelect(['last_order_date' => Order::select('created_at')->whereColumn('user_id', 'users.id')->latest()->take(1)])`
- **joinSub():** Joins subquery result to main query — useful for complex aggregated values that are beyond simple subqueries — replaces raw SQL JOINs to derived tables
- **whereExists():** Adds WHERE EXISTS clause with subquery — more efficient than `WHERE IN (SELECT ...)` for existence checks — `whereHas()` internally uses `whereExists()`

---

## Mental Models

- **withCount as Post-It Note:** Adding `withCount('orders')` is like attaching a post-it note to each user record that says "this user has X orders." The post-it note is computed in the same trip to the filing cabinet — no second trip needed.
- **Correlated Subquery as Personal Assistant:** A correlated subquery is like a personal assistant who, for each user in a list, runs to another room to count that user's orders and reports back. If the assistant is fast (good index), the process works well. If the other room has no filing system (no index), each trip takes forever.

---

## Internal Mechanics

When `withCount('orders')` is called, Laravel adds a correlated subquery to the SELECT clause. The subquery references the parent table's primary key and counts matching rows in the related table. For `withSum`, `withAvg`, etc., the subquery uses SUM, AVG, MIN, or MAX instead of COUNT. The database executes the outer query with embedded subqueries — the optimizer may choose different strategies (correlated execution, join, or materialization) depending on the query planner. `joinSub` creates a derived table from a subquery and joins it to the main query, which the optimizer can handle differently from correlated subqueries.

---

## Patterns

- **Prefer withCount Over loadCount:** `loadCount` runs a separate query after the main query — `withCount` includes count in the main query — always more efficient unless conditional loading needed
- **Use Relationship Aliasing:** When using multiple aggregates on the same relationship, alias them: `withCount('orders as paid_orders', 'orders as pending_orders')` with closure conditions
- **Check Generated SQL:** Review SQL generated by `withSum` and similar methods — ensure indexes support the correlated subquery — without index on foreign key, correlated subqueries are slow

---

## Architectural Decisions

Use `withCount` and related methods for replacing N+1 lazy-loaded relationship counts and adding aggregate columns to API responses and dashboard widgets. Do not use for complex multi-table aggregations (use raw SQL or a view) or for aggregations over large datasets in real-time API responses (use cached pre-computed values). Combine with `select()` to minimize data transfer when only aggregated data is needed.

---

## Tradeoffs

| Benefit | Cost | Consequence |
|---------|------|-------------|
| Single query replaces N+1 | Correlated subqueries execute per parent row | Index on join column is essential |
| Clean, readable Eloquent syntax | Generated SQL may not use optimal index | Always review generated SQL for performance |
| Multiple aggregates in one query | Complex queries become unreadable with many clauses | Consider raw SQL or view for >3 aggregates |
| No schema changes needed | Query-time cost increases with data volume | Consider pre-computed read models at scale |

---

## Performance Considerations

Correlated subqueries execute once per parent row — for 10K users, the subquery runs 10K times. Ensure indexes exist on the join column. `withCount` adds minimal overhead for tables under 1M rows — beyond 1M, consider pre-computed counters. `joinSub` may be more efficient than `withCount` for complex aggregations because the database optimizer can use different join strategies. `whereExists` is faster than `whereHas` with closures because it avoids subquery evaluation overhead.

---

## Production Considerations

Index foreign key columns used in aggregate subqueries — without indexes, correlated subqueries become full table scans. Combine `withCount` with `select()` to avoid transferring unnecessary columns (passwords, tokens). Monitor slow query logs for correlated subquery performance issues — they can be hidden N+1 problems in disguise.

---

## Common Mistakes

- **withCount on Large Dataset Without Index:** `User::withCount('orders')->get()` on 100K users without index on `orders.user_id` — each subquery scans orders table — 100K full table scans = minutes. Better: ensure `orders.user_id` is indexed, consider pre-computed counters.
- **withSum Over Non-Indexed Column:** `withSum('orders', 'total')` without index on `orders.total` — SUM forces full table scan per parent row. Better: index `orders.total` or use covering index on `(user_id, total)`.
- **Chained Aggregate Calls Without Select:** `User::withCount('orders')->withSum('orders', 'total')->get()` returns all columns including password — unnecessary data transfer. Better: `User::select('id', 'name')->withCount('orders')->withSum('orders', 'total')->get()`.

---

## Failure Modes

- **Hidden N+1 with withCount:** Developer thinks `withCount` solves performance but subquery still causes N+1 without index — each subquery takes 50ms, 1000 users = 50 seconds. Mitigation: review EXPLAIN plans, verify index usage.
- **Over-Nesting Aggregate Closures:** Complex conditions in `withCount` closures generate deeply nested SQL — query planner chooses poor execution plan. Mitigation: simplify conditions, consider raw SQL for complex logic.
- **Memory Exhaustion with Large Result Sets:** `->get()` on 100K parent records with aggregates — all rows and aggregates loaded into memory. Mitigation: use cursor or chunk for large result sets.

---

## Ecosystem Usage

Eloquent aggregate methods are built into Laravel core — no external package needed. They are used extensively in dashboard widget providers, API resources, and admin panels. The methods work across all supported databases (MySQL, PostgreSQL, SQLite, SQL Server). Understanding the generated SQL is essential for production usage.

## Research Notes

Eloquent's aggregate methods are among the most underutilized features in Laravel — many developers know about `withCount` but not `withSum`, `withAvg`, or the more flexible `addSelect` with subquery pattern. The key insight is that these methods are SQL optimizations, not conveniences — they replace N+1 queries with correlated subqueries that, with proper indexing, are dramatically more efficient. The generated SQL should always be reviewed for index usage.
