# Overview

Eloquent's aggregate methods (`withSum`, `withAvg`, `withCount`, `withMin`, `withMax`) and subquery patterns (`addSelect` with closures, `joinSub`, `whereExists`) bridge the gap between ORM convenience and analytical query power. These methods allow attaching aggregated values from related tables as computed columns on the parent result — essentially creating denormalized read models on-the-fly without schema changes. The key engineering insight: these are not ORM conveniences — they are SQL optimizations that replace N+1 lazy-loaded relationship counts with a single SQL query using correlated subqueries or lateral joins.

Engineers must care because N+1 relationship loading is the most common performance antipattern in Laravel applications. `withCount` and its siblings solve this elegantly, but many developers are unaware of their full capabilities or the SQL they generate. Understanding the generated SQL enables informed decisions about when to use these methods vs. raw subqueries.

# Core Concepts

## withCount()

Adds a `{relation}_count` column to the parent query result. Generates a correlated subquery: `SELECT *, (SELECT COUNT(*) FROM orders WHERE user_id = users.id) as orders_count FROM users`. This is a single query, not N+1.

## withSum() / withAvg() / withMin() / withMax()

Similar to `withCount` but for summation and statistical aggregation. `withSum('orders', 'total')` adds a `orders_sum_total` column. These work on any numeric column in the related table.

## addSelect() with Subquery

The most flexible pattern. Allows adding arbitrary subquery results as computed columns: `User::addSelect(['last_order_date' => Order::select('created_at')->whereColumn('user_id', 'users.id')->latest()->take(1)])`.

## joinSub()

Joins a subquery result to the main query. Useful for attaching aggregated values that are more complex than simple subqueries. `joinSub` with aggregates replaces raw SQL JOINs to derived tables.

## whereExists()

Adds a WHERE EXISTS clause with a subquery. More efficient than `WHERE IN (SELECT ...)` for existence checks. Laravel's `whereHas()` internally uses `whereExists()`.

## Generated Columns

PostgreSQL generated columns (stored or virtual) can be used with Eloquent. A generated column like `total_revenue` is computed from `quantity * price` at the database level, exposed as an Eloquent attribute. This shifts computation from PHP to the database.

# When To Use

- Replacing N+1 lazy-loaded relationship counts
- Adding aggregate columns to API responses and dashboard widgets
- Building reporting queries with grouped aggregate data
- Denormalizing computed values at query time
- Replacing raw SQL subqueries with readable Eloquent methods

---

# When NOT To Use

- Complex multi-table aggregations (use raw SQL or a view)
- Aggregations over large datasets in real-time API responses (use cached pre-computed values)
- Cases where the aggregation is used in multiple queries (consider a materialized view or read model)

---

# Best Practices

## Prefer withCount Over loadCount

`loadCount` runs a separate query after the main query. `withCount` includes the count in the main query. `withCount` is always more efficient — use it unless you need conditional loading.

## Use Relationship Aliasing

When using multiple aggregates on the same relationship, alias them: `withCount('orders as paid_orders', 'orders as pending_orders')` with closure conditions.

## Understand Generated SQL

Review the SQL generated by `withSum` and similar methods. Ensure indexes support the correlated subquery. Without an index on the foreign key, correlated subqueries are slow.

## Combine With Select

When you only need aggregated data, select minimal columns: `User::select('id', 'name')->withCount('orders')`. This reduces data transfer.

---

# Performance Considerations

- Correlated subqueries execute once per parent row. For 10K users, the subquery runs 10K times. Ensure indexes exist on the join column.
- `withCount` adds minimal overhead for tables under 1M rows. Beyond 1M, consider pre-computed counters.
- `joinSub` may be more efficient than `withCount` for complex aggregations because the database optimizer can use different join strategies.
- `whereExists` is faster than `whereHas` with closures because it avoids the subquery evaluation overhead.

# Common Mistakes

## Mistake: withCount on Large Dataset Without Index

`User::withCount('orders')->get()` on 100K users without an index on `orders.user_id`. Each subquery scans the orders table. 100K × full table scans = minutes of query time.

**Better approach:** Ensure `orders.user_id` is indexed. For very large datasets, pre-compute counts in a `user_stats` table.

## Mistake: withSum Over Non-Indexed Column

`withSum('orders', 'total')` without an index on `orders.total`. The SUM aggregation forces a full table scan per parent row.

**Better approach:** Index `orders.total` or use a covering index on `(user_id, total)`.

## Mistake: Chained Aggregate Calls Without Select

`User::withCount('orders')->withSum('orders', 'total')->get()` returns all user columns (password, remember_token) in addition to aggregates. Unnecessary data transfer.

**Better approach:** `User::select('id', 'name')->withCount('orders')->withSum('orders', 'total')->get()`

## Mistake: Using loadCount When withCount Is Possible

`User::get()->loadCount('orders')` loads all users, then runs a second query for counts. This is two round trips. `User::withCount('orders')->get()` is one round trip.

**Better approach:** Always use `withCount` when the count is needed for all results. Use `loadCount` only for conditional loading.
