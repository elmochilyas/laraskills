## 1. Hand-Written Middleware Without Command

### Category
Code Organization

### Description
Manually creating job middleware files without using `php artisan make:job-middleware`, risking incorrect namespace, interface signature, or directory structure.

### Why It Happens
The developer finds it faster to create the file manually than to run the Artisan command. The middleware class is simple — a `handle()` method and some logic. The command seems unnecessary for a small class. Without the command scaffolding, the developer may miss the correct interface implementation, use the wrong namespace, or forget imports.

### Warning Signs
- Job middleware created by copying-and-pasting from another file
- No `make:job-middleware` command output in project history
- Manual tests passing but integration tests failing
- Runtime errors about missing interface or wrong signature

### Why Harmful
The manually created middleware may not implement `MiddlewareInterface` correctly. The `handle()` signature might be wrong (e.g., missing type hints). The namespace might be incorrect. The middleware is applied to a job, but when the worker processes it, PHP throws a type error because the class doesn't implement the expected contract. The job fails, and debugging reveals the trivial scaffolding mistake.

### Consequences
- Runtime errors from wrong interface or signature
- Debugging time wasted on scaffolding mistakes
- Inconsistent namespace conventions across the codebase
- Code review overhead for structural correctness

### Alternative
Always use `php artisan make:job-middleware` for new middleware to ensure correct structure.

### Refactoring Strategy
1. Delete manually created middleware files
2. Run `php artisan make:job-middleware MiddlewareName`
3. Re-implement the logic in the correctly structured file
4. Delete the old manual file

### Detection Checklist
- [ ] All middleware created via `make:job-middleware` command
- [ ] Correct namespace `App\Queue\Middleware`
- [ ] Correct `handle()` signature
- [ ] No manual middleware creation observed

## 2. Middleware Misplaced in HTTP Directory

### Category
Code Organization

### Description
Placing job middleware in `app/Http/Middleware/` instead of `app/Queue/Middleware/`, mixing HTTP and queue middleware in a way that reduces discoverability and violates conventions.

### Why It Happens
The `app/Http/Middleware/` directory already exists and is the familiar location for all middleware. The developer places job middleware there out of habit, not realizing a separate directory is standard. Both HTTP and job middleware use similar terminology ("middleware"), making the placement seem natural.

### Warning Signs
- Job middleware files exist in `app/Http/Middleware/`
- Mixed job and HTTP middleware in the same directory
- New team members unable to find job middleware
- Namespace `App\Http\Middleware` for job middleware

### Why Harmful
HTTP middleware uses a different interface (handles `$request` and `$next`) while job middleware handles `$job` and `$next`. Placing them together creates confusion — developers may try to use HTTP middleware as job middleware or vice versa. The non-standard location means code reviews may miss job middleware files, and auto-discovery by tooling may be inconsistent.

### Consequences
- Reduced discoverability for job middleware
- Confusion between HTTP and queue middleware conventions
- Wrong interface implementations from misleading location
- Code review misses on job middleware files

### Alternative
Always place job middleware in `app/Queue/Middleware/` as generated by `make:job-middleware`.

### Refactoring Strategy
1. Move middleware files from `app/Http/Middleware/` to `app/Queue/Middleware/`
2. Update namespace from `App\Http\Middleware` to `App\Queue\Middleware`
3. Update all imports referencing the old namespace
4. Verify middleware still applies correctly to jobs

### Detection Checklist
- [ ] All job middleware in `app/Queue/Middleware/`
- [ ] Correct namespace `App\Queue\Middleware`
- [ ] No job middleware in `app/Http/Middleware/`
- [ ] Imports updated to reflect new location
