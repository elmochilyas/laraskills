# Knowledge Unit: PHPStan Baseline Patterns

## Metadata
- **Subdomain:** Code Quality & Static Analysis
- **Domain:** Platform Engineering & Developer Experience
- **KU ID:** code-quality-static-analysis/phpstan-baseline-patterns
- **Maturity:** Mature
- **Related Technologies:** PHPStan, Larastan, PHP, Static Analysis

## Executive Summary

PHPStan's baseline feature allows teams to adopt strict static analysis on existing codebases by recording current errors as a "known issues" baseline file, then only reporting new errors going forward. The baseline is generated via `phpstan analyse --generate-baseline`, which creates a `phpstan-baseline.neon` file listing all current errors with specific file lines and error messages. Subsequent PHPStan runs ignore baseline-listed errors, allowing teams to work at strict levels (6-9) while gradually resolving pre-existing issues. The baseline includes error counts per file, enabling trend tracking. Effective baseline management involves: periodic baseline regeneration (to remove fixed errors), baseline review in CI (to catch regressions), and incremental reduction targets (e.g., "reduce baseline by 10% per quarter"). The baseline is a PHPStan-core feature, not Laravel-specific, but is critical for Laravel teams adopting Larastan on mature projects.

## Core Concepts

- **Baseline File:** A `.neon` file listing all known PHPStan errors with file paths, line numbers, and error descriptions; generated by `--generate-baseline` flag
- **Error Grouping:** The baseline groups errors by file and error type, allowing PHPStan to match current analysis errors against the baseline
- **Baseline Scope:** Only errors at the current analysis level are captured; if you increase the level later, new errors at the higher level are not covered
- **Baseline Freshness:** PHPStan warns when the baseline is stale (errors listed in baseline are no longer present), encouraging regular regeneration
- **Baseline Counts:** Each baseline entry includes the error count; PHPStan tracks whether current errors match or exceed baseline counts
- **CI Integration:** In CI, regenerate baseline regularly and diff against the committed baseline to detect new errors

## Mental Models

- **Baseline as Technical Debt Ledger:** The baseline tracks every known static analysis debt item, with location, type, and count—like a financial balance sheet for code quality
- **Baseline as Gradualism Enabler:** Instead of the all-or-nothing choice between low analysis level (miss issues) vs high level (too many errors to fix), the baseline enables high-level analysis with known exceptions
- **Baseline as Bug Insurance:** Existing errors are documented (baseline) but new code is held to strict standards—the baseline prevents the debt from growing while the team pays it down

## Internal Mechanics

1. **Error Collection:** PHPStan runs all rules against the codebase and collects every error with file path, line number, error message, and error type (rule identifier)
2. **Baseline Serialization:** Errors are serialized to the NEON format grouped by file: `parameters: > ignoreErrors: - '#Undocumented property \$name#' path: app/Models/User.php count: 1`
3. **Baseline Loading:** On subsequent runs, PHPStan loads the baseline and, before reporting each error, checks if it matches a baseline entry (file, line, message pattern)
4. **Baseline Matching:** Errors are matched by file path, error message pattern (regex from baseline), and line number; if matched, the error is suppressed
5. **Count Tracking:** PHPStan tracks how many errors of each type were suppressed; if the actual count exceeds the baseline count, the excess is reported as new errors
6. **Staleness Detection:** If a baseline entry references errors that no longer exist, PHPStan warns during analysis, indicating the baseline can be regenerated to remove stale entries

## Patterns

- **Level Graduation Pattern:** Start at level 2 with baseline, fix the most common error types, then graduate to level 4 with baseline, fix, and continue—each level increases precision
- **Baseline Reduction Target Pattern:** Set quarterly targets for baseline reduction (e.g., "reduce baseline by 15% per quarter"). Track baseline file line count or `wc -l` in CI.
- **New Code Gate Pattern:** In CI, fail the build if the baseline has new errors (i.e., the proposed changes introduce new static analysis debt). Compare regenerated baseline against committed baseline.
- **Blocker Pattern for Baseline Items:** For critical error types (undefined methods on core models, type errors in service classes), add explicit `--ignoreErrors` in phpstan.neon instead of baseline, forcing fixes before merge
- **Scheduled Baseline Cleanup Pattern:** Run a weekly/monthly scheduled CI job that regenerates the baseline; if the regenerated baseline is smaller, create a PR with the reduction
- **Team Baseline Ownership Pattern:** Assign baseline ownership to specific team members or guilds; each team tracks and reduces their segment of the baseline

## Architectural Decisions

| Decision | Options | Recommendation |
|---|---|---|
| Baseline inclusion | In version control vs .gitignored vs CI-generated | In version control (committed baseline visible to team); ignore for ephemeral environments |
| Baseline regeneration | Manual only vs CI automated vs scheduled | Manual for controlled reductions; scheduled for automatic cleanup |
| Baseline vs ignoreErrors | Baseline (debt) vs ignoreErrors (permanent exceptions) | Baseline for methodical debt; ignoreErrors for false positives that can't be fixed |
| Baseline granularity | Per-file vs per-project with global count | Per-file baseline (default, gives visibility into which files have debt) |
| Baseline enforcement | New errors fail CI vs warn vs info | New errors fail CI in strict projects; warn in transitioning projects |

## Tradeoffs

- **Comprehensive Baseline vs Clean Baseline:** A comprehensive baseline captures all errors (easier initial adoption) but may be very large (thousands of entries) making tracking difficult. Start comprehensive, then reduce aggressively in the first quarter.
- **Baseline Renewal Frequency:** Frequent baseline regeneration keeps the baseline lean but may hide new errors that are inadvertently fixed by regeneration. Infrequent regeneration allows the baseline to collect stale entries. Monthly regeneration is a good balance.
- **Baseline vs Raising Level:** Raising the analysis level creates new errors that require updating the baseline. Fix errors before raising the level rather than expanding the baseline indefinitely.

## Performance Considerations

- **Baseline Loading:** PHPStan loads and processes the baseline on every run. A baseline with 10,000 entries adds ~100ms to analysis time. This is negligible compared to total analysis time.
- **Baseline Size:** A baseline file with 10,000 entries is ~500KB-1MB. The file is loaded from disk and parsed on each run; this is fast (<50ms).
- **Baseline in CI:** Baseline comparison in CI adds minimal overhead (<1 second). The baseline is committed and diffed against the generated baseline on each run.
- **Freshness Check:** PHPStan's stale baseline check adds negligible overhead but provides valuable feedback about cleanup progress.

## Production Considerations

- **CI Baseline Validation:** In CI, compare the regenerated baseline against the committed baseline. If the regenerated baseline has fewer entries (some errors were fixed), approve the PR. If it has more entries (new errors introduced), flag for review.
- **Baseline in Code Review:** When reviewing PRs that modify baseline files, inspect what errors were added (debt increase) or removed (debt reduction). Require justification for new baseline entries.
- **Team Visibility:** Make the baseline visible to the team (committed, not .gitignored). Track baseline size over time in team dashboards or CI artifacts.
- **Gradual Reduction:** Don't expect the baseline to reach zero. Set realistic targets: level 6 baseline in 6 months, level 9 baseline in 12-18 months.

## Common Mistakes

- **Not regenerating the baseline after fixing errors:** PHPStan continues to suppress errors that were already fixed; the baseline becomes stale and its size doesn't reflect actual debt
- **Using baseline at the lowest analysis level:** Generating baseline at level 1 and never raising the level; the value of static analysis is dramatically reduced
- **Ignoring baseline staleness warnings:** PHPStan's "baseline is stale" warning is ignored; fixed errors remain in the baseline and the team loses the sense of progress
- **Not tracking baseline size over time:** The baseline grows without anyone noticing; what started as 500 entries grows to 5000 without intervention
- **Using baseline as permanent cleanup exemption:** Keeping errors in the baseline indefinitely instead of fixing them; treat the baseline as a temporary debt tracker, not a permanent exception
- **Baseline conflicts in PRs:** Two PRs both regenerate the baseline, causing merge conflicts. Mitigate: regenerate baseline only in dedicated cleanup PRs.

## Failure Modes

- **Baseline Merge Conflict:** Two developers regenerate the same baseline file with different line orders. Mitigate: standardize baseline regeneration in a single cleanup PR per sprint.
- **Baseline Not Matching Actual Errors:** After a PHPStan version upgrade, the baseline format changes and errors that should be suppressed are no longer matched. Mitigate: regenerate baseline after every PHPStan upgrade.
- **Baseline Covers Up Regressions:** A baseline entry matches a new error by coincidence (same file, similar error message). Mitigate: review CI baseline comparison output to catch suppressed regressions.
- **Baseline File Corruption:** Manual editing of the NEON baseline file introduces syntax errors. Mitigate: always regenerate baseline via `--generate-baseline` rather than editing manually.

## Ecosystem Usage

- **PHPStan + Larastan:** The most common use case—Larastan generates many errors on existing codebases; baseline enables teams to adopt Larastan without fixing everything first
- **Open Source Laravel Packages:** Packages use baselines to maintain high analysis levels while accepting contributions from varied skill levels
- **CI/CD Workflows:** Baseline comparison is a standard step in Laravel CI workflows; teams use `deployer` or GitHub Actions to enforce baseline stability
- **Large-Scale Laravel Applications:** Enterprise Laravel applications (>1000 models, >500k LOC) rely on baselines for feasible static analysis adoption

## Research Notes

- PHPStan's baseline feature was introduced in PHPStan 0.11.x and has been refined in each major version
- The baseline format changed in PHPStan 1.x from simple path patterns to more precise count-based matching, reducing false positive suppression
- PHPStan 2.x improved baseline matching with regular expression patterns for error messages, allowing more selective suppression
- Baseline file format: NEON (PHPStan's configuration format, similar to YAML); entries are grouped under `parameters > ignoreErrors > path/count/message`
