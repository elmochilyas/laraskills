# Skill: Configure Laravel Sail

## Purpose
Install and configure Laravel Sail as the Docker-based development environment for Laravel projects, providing consistent PHP, database, cache, and service configurations across the team.

## When To Use
- All new Laravel projects for team environment consistency
- Cross-platform teams (Windows, macOS, Linux) needing identical setups
- Onboarding new developers quickly

## When NOT To Use
- Production environments (Sail is a development tool)
- Simple projects where Docker overhead outweighs consistency benefits
- Teams already using native PHP setup with no environment issues

## Prerequisites
- Docker Desktop or Docker Engine installed
- Composer installed

## Inputs
- `docker-compose.yml` — generated by Sail installation
- `.env` — Sail-specific variables
- `docker/` directory (published for customization)

## Workflow

1. **Install Sail:** If creating a new project, use `composer create-project laravel/laravel example-app --prefer-dist`. Install Sail with `php artisan sail:install` and select the needed services.

2. **Configure Services:** Select services during install: MySQL (default), PostgreSQL, Redis, Meilisearch, Mailpit, Selenium, MinIO. Use `--with=mysql,redis` for selective services.

3. **Set Up sail Alias:** Create shell alias: `alias sail='[ -f sail ] && bash sail || bash vendor/bin/sail'` for convenient command execution.

4. **Start Environment:** Run `sail up -d` (detached mode) to start all containers. First run builds images and may take 2-5 minutes.

5. **Configure PHP Version:** Set `PHP_VERSION=8.3` in `.env` to match production. Rebuild with `sail build --no-cache` after changing.

6. **Run Setup Commands:** After starting, run `sail artisan migrate`, `sail npm install && sail npm run build` to set up the application.

7. **Match PHP Version to Production:** Set `PHP_VERSION` to match the exact production PHP version. Prevents version-dependent bugs.

8. **Use Selective Services:** Install only needed services with `sail:install --with=mysql,redis,mailpit`. Unnecessary services consume resources.

## Validation Checklist

- [ ] Sail environment starts with `sail up -d`
- [ ] Application accessible at `http://localhost` (or configured port)
- [ ] Database accessible via `sail artisan migrate`
- [ ] Redis accessible for cache and queues
- [ ] Mailpit UI accessible for email previews
- [ ] `sail` alias configured for convenience
- [ ] PHP version matches production

## Common Failures

| Failure | Early Detection |
|---------|----------------|
| Docker not running | `sail up` fails; start Docker Desktop |
| Port conflicts | Host ports (80, 3306) already in use; change via .env |
| Build fails on first run | Check Docker disk space and internet connection |
| Sail command not found | Not in project directory or `vendor/bin/sail` doesn't exist |

## Decision Points

- **Use for all new Laravel projects** for team environment consistency
- **Use for cross-platform teams** (Windows, macOS, Linux) needing identical setups
- **Don't use in production** — Sail is a development tool
- **Don't use for simple projects** where Docker overhead outweighs consistency benefits

## Performance/Security Considerations

- **Docker resource usage:** Sail containers use 2-3GB RAM; allocate sufficient Docker resources
- **Match PHP version:** Using different PHP versions in dev and prod causes version-dependent bugs
- **Never Sail in production:** Sail is designed exclusively for local development
