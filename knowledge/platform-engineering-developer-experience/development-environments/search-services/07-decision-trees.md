## Architecture-Level Decision Trees

### D01: Search Engine Selection

```
START: Which search engine should we use?
│
├── Meilisearch (recommended for most apps)
│   ├── Sail: --with=meilisearch (easy setup)
│   ├── Pro: typo-tolerant by default
│   ├── Pro: instant faceted filtering
│   ├── Pro: easy setup, minimal config
│   ├── Pro: open-source, self-hosted or cloud
│   ├── Laravel support: first-class via Scout
│   └── Best for: most Laravel apps, product search, docs search
│
├── Typesense (alternative)
│   ├── Similar to Meilisearch, different query syntax
│   ├── Pro: extremely fast, typo-tolerant
│   ├── Con: not natively in Sail (manual Docker Compose config)
│   ├── Con: less Laravel ecosystem integration
│   └── Best for: teams preferring Typesense's API
│
├── Algolia (cloud-only, no local dev)
│   ├── Pro: fully managed, no infrastructure
│   ├── Pro: global CDN, fast everywhere
│   ├── Con: costs scale with usage
│   ├── Con: requires internet for dev (no local emulation)
│   └── Best for: production search without self-hosting
│
└── Decision factors
    ├── Self-hosted dev+prod? → Meilisearch
    ├── Cloud-only? → Algolia (use Meilisearch locally for dev parity)
    ├── Open-source requirement? → Meilisearch
    └── Existing Typesense experience? → Typesense
```

### D02: Scout Configuration

```
START: How should we configure Laravel Scout?
│
├── Driver selection (.env)
│   ├── SCOUT_DRIVER=meilisearch (if using Meilisearch)
│   ├── SCOUT_DRIVER=algolia (if using Algolia)
│   ├── SCOUT_DRIVER= (empty/null) → database driver (fallback)
│   └── Set in .env per environment
│
├── Meilisearch-specific config
│   ├── MEILISEARCH_HOST=http://meilisearch:7700 (Docker hostname)
│   ├── MEILISEARCH_KEY=masterKey (auto-generated by Sail)
│   ├── Need: Docker service hostname, not localhost
│   └── Filterable attributes: configure in Meilisearch settings
│
├── Scout model configuration
│   ├── Add Searchable trait to models
│   ├── Override toSearchableArray() for indexed fields
│   ├── Define filterable attributes in Meilisearch
    │   │   └── Product::search('shoes')->where('category_id', 5)->get()
│   └── Configure searchable data: what fields to index
│
└── Index management
    ├── php artisan scout:import "App\Models\Product" (initial import)
    ├── php artisan scout:flush "App\Models\Product" (clear index)
    ├── Auto-sync: model changes automatically update index (if Scout enabled)
    └── Re-index: after changing toSearchableArray()
```

### D03: Indexing Strategy

```
START: When and how should we index data?
│
├── Synchronous indexing (simple, small datasets)
│   ├── Model changes → immediate index update
│   ├── Config: SCOUT_QUEUE=false (default)
│   ├── Pro: index always up-to-date
│   ├── Con: slow for bulk operations, blocks response
│   └── Best for: small datasets, low write volume
│
├── Queue-based indexing (recommended for production)
│   ├── Config: SCOUT_QUEUE=true
│   ├── Model changes → job dispatched → indexed asynchronously
│   ├── Pro: non-blocking, better UX
│   ├── Pro: batches updates, handles load spikes
│   ├── Need: queue worker running
│   └── Best for: write-heavy apps, production
│
├── Manual indexing (artisan commands)
│   ├── Full re-index: php artisan scout:import
│   ├── Scheduled: cron for periodic re-indexing
│   ├── Pro: full control over timing
│   ├── Con: index lag between model save and search availability
│   └── Best for: batch updates, data migrations
│
└── Development indexing
    ├── After adding Searchable: run scout:import once
    ├── For small dev datasets: sync is fine
    ├── Test: search returns expected results after import
    └── Re-import after changing indexed fields
```

### D04: CI Integration

```
START: Should we run the search engine in CI?
│
├── Use Scout's database driver in CI (no search engine needed)
│   ├── Config: SCOUT_DRIVER= (empty/null) in phpunit.xml
│   ├── Falls back to database LIKE queries
│   ├── Pro: no additional service, fast tests
│   ├── Con: doesn't test search engine-specific features
│   └── Best for: tests that don't depend on search engine behavior
│
├── Run Meilisearch as CI service (recommended for search tests)
│   ├── GitHub Actions:
│   │   services:
│   │     meilisearch:
│   │       image: getmeili/meilisearch:latest
│   │       env: { MEILI_MASTER_KEY: testKey }
│   │       ports: [7700:7700]
│   ├── Config: MEILISEARCH_HOST=http://127.0.0.1:7700 in CI
│   ├── Pro: tests real search behavior (typo tolerance, faceted filters)
│   ├── Pro: catches search-specific issues
│   └── Best for: apps heavily using search features
│
└── Test patterns
    ├── Import data before search-dependent tests
    ├── Clear index between test suites
    ├── Test: search results, filters, typo tolerance
    └── Test: empty results, edge cases
```
