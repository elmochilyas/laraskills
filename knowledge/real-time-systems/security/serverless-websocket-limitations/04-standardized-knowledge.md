## Overview
Serverless platforms (AWS Lambda, Vercel Functions, Cloudflare Workers) have fundamental limitations for WebSocket applications due to their stateless, short-lived execution model. WebSocket connections require persistent state (connection tracking, subscription state, presence data) that serverless functions do not natively support. The solution splits the architecture: use serverless functions for broadcast event dispatch and a managed service (Pusher, Ably, API Gateway) for connection management.

## Core Concepts
- Serverless functions are invoked per-request and have no memory between invocations
- WebSocket servers maintain persistent in-memory state across the entire connection lifecycle
- These two models are fundamentally incompatible
- Cloudflare Workers are an exception—they support WebSocket connections via `acceptWebSocket()` API
- Managed WebSocket services (Pusher, Ably) handle the stateful part; serverless functions handle auth and dispatch

## When To Use
- Applications already invested in serverless infrastructure (Vapor, Lambda, etc.)
- Prototypes and low-traffic applications not warranting self-hosted WebSocket infrastructure
- Applications using managed WebSocket services for global edge delivery

## When NOT To Use
- High-traffic applications with predictable WebSocket loads (self-hosted Reverb is more cost-effective)
- Applications requiring custom WebSocket server configuration or protocols
- Bidirectional real-time features requiring client events

## Best Practices (Why)
- **Use managed WebSocket service + serverless app**: Pusher/Ably for WebSocket; serverless functions for auth and broadcast dispatch—the only reliable pattern
- **Implement connection heartbeat**: API Gateway drops connections after 10-min idle timeout; send ping every 5 minutes to prevent premature disconnection
- **Design for reconnection**: Serverless auth endpoints may have cold start latency; clients should use jitter and backoff
- **Have a migration path**: If WebSocket requirements outgrow serverless, plan to move to self-hosted Reverb

## Architecture Guidelines
- Do not attempt to run Reverb on Lambda or Vapor—Reverb requires a long-running PHP process
- Use Laravel Vapor + Pusher/Ably for serverless Laravel real-time applications
- API Gateway is the AWS-native option with Lambda backends; Cloudflare Workers for V8-based WebSocket relay
- Monitor API Gateway connection limits and usage costs

## Performance Considerations
- API Gateway: 10-min idle timeout, 32KB max message, 5-min Lambda timeout
- Cloudflare Workers: connections count against CPU time limits (30ms free, paid plans more)
- Lambda cold starts add ~500ms+ to WebSocket connection setup (auth endpoint)
- Message publishing via API Gateway `@connections` API: HTTP request per broadcast event

## Security Considerations
- Managed services handle WSS transport security automatically
- Auth tokens generated by serverless functions must be short-lived and properly signed
- API Gateway WebSocket API integrates with AWS IAM for authorization
- Cloudflare Workers can validate origin and token before accepting WebSocket connections

## Common Mistakes

| Mistake | Description | Cause | Consequence | Better Approach |
|---------|-------------|-------|-------------|----------------|
| Running Reverb on Lambda | Reverb requires long-running PHP process | Not understanding serverless limitations | Fails to deploy or run | Use managed WebSocket service with serverless functions |
| Not accounting for API Gateway idle timeout | Connections drop after 10-min inactivity | Ignoring API Gateway limits | Silent disconnections | Implement connection heartbeat |
| Exceeding 32KB message limit | Messages silently dropped by API Gateway | Not checking message size | Missing broadcasts | Chunk large payloads or use Reverb |
| Assuming Cloudflare Workers run Laravel | Workers are JavaScript/V8, not PHP | Platform misunderstanding | Cannot run Laravel broadcasting | Use Workers as WebSocket relay, not Laravel runtime |

## Anti-Patterns
- **Self-hosting state in serverless database tables for WebSocket connections**: The polling overhead and latency make this impractical
- **Using Vapor without a managed WebSocket service**: Laravel Vapor does not support self-hosted Reverb
- **Ignoring cold start impact on WebSocket auth**: A 500ms+ auth delay during reconnection storm compounds the problem

## Examples

### Laravel Vapor with Pusher
```env
# .env.vapor
BROADCAST_CONNECTION=pusher
PUSHER_APP_ID=your-app-id
PUSHER_APP_KEY=your-app-key
PUSHER_APP_SECRET=your-app-secret
PUSHER_APP_CLUSTER=mt1
VAPOR_BROADCAST=true
```
